# Branch Hygiene — WIP-Branch Naming, Stash Lifetimes, Stale-Branch Policy

**Classification:** Hybrid. The naming convention, stash-lifetime discipline, and "when to close a stale branch" guidance are Pattern (self-applied by the operator and the orchestrator). The mechanical layer is the extended `stale-active-plan-surfacer.sh` SessionStart hook (item 8 of the git-best-practices 9-item initiative), which surfaces stale local branches at session start so the operator sees them without having to remember to check.

**Ships with:** PR for item 8 of the `docs/plans/git-bestpractices-9-item-initiative-2026-05-29.md` initiative. Pairs with `session-start-git-freshness.sh` (item 1, which surfaces uncommitted state on non-WIP-pattern branches) and `pre-push-divergence-check.sh` (item 2, which catches stale-fetch pushes).

## Why this rule exists

Across multiple concurrent code sessions on multiple machines, local branches accumulate. Some hold genuinely in-progress work; some are leftover from sessions that ended without cleanup; some duplicate content that has since landed via a different SHA path; some are abandoned hypotheses. Without a naming convention + a periodic surfacing mechanism, the operator's `git branch` list becomes an unreadable swamp and useful work hides in it (the 2026-05-29 fork-sync investigation found 15+ local branches on a single machine, of which only 2 carried unique content).

This rule codifies three pieces of discipline that, together, keep the branch list survivable:

1. **WIP-branch naming** so a stranger reading `git branch` knows what each branch is.
2. **Stash-lifetime discipline** so untracked uncommitted state doesn't accumulate beyond the session that created it.
3. **Stale-branch policy** so branches without recent commits or pushes get reviewed and either closed or revived.

## WIP-branch naming convention

When creating any branch for in-progress work, use one of the following named prefixes:

| Prefix | When | Example |
|---|---|---|
| `wip/<machine>-<topic>-<date>` | Genuinely work-in-progress, not yet a feature with a clear shape. Multi-machine context implied. | `wip/laptop-investigate-checkout-divergence-2026-05-29` |
| `feat/<descriptive-slug>` | A new feature with a clear shape; expected to ship via PR. | `feat/git-bestpractice-pre-push-divergence` |
| `fix/<descriptive-slug>` | A bug fix. | `fix/sync-script-unbound-trap` |
| `salvage/<short-id>` | Branch holding work that needs to be extracted from a session that died. | `salvage/nervous-lehmann-review` |
| `backup/<context>` | Snapshot of state before a risky operation. | `backup/pre-rebase-2026-05-27` |
| `rebase/<context>` | Active rebase work-area; expected to be short-lived. | `rebase/cleanup-merge-conflicts` |
| `reconverge/<context>` | Cross-fork reconciliation branches. | `reconverge/personal-prs-40-44-onto-pt-2026-05-28` |
| `sync/<context>` | Cross-fork sync working branches (consumed by `sync-pt-to-personal.sh`). | (auto-generated by the script) |

**Why include machine in `wip/` branches:** the user has multiple machines, and in the multi-machine workflow `wip/<machine>-...` makes it instantly clear from which machine the work originated and whether it might still be live. The other prefixes don't need machine names because they imply a sharper exit condition (a PR; a finished rebase; a specific salvage operation) that closes within hours, not days.

**Why include date in `wip/` branches:** WIP branches accumulate; the date makes the stale-branch policy (below) mechanically applicable without having to remember when each branch was created.

**Branches whose names match the prefix list are treated as "expected WIP" by `session-start-git-freshness.sh`** (item 1) — uncommitted changes on them don't trigger the warning. Branches with names that DON'T match the prefix list will trigger a warning if they carry uncommitted changes.

## Stash-lifetime discipline

A `git stash` is a temporary holding pen for uncommitted state, useful for "I need to switch branches briefly." It is NOT a place to store work for tomorrow. **Stashes older than 24 hours are a leak signal** — either the work in them is needed (in which case it belongs on a real branch) or it's not (in which case it belongs deleted).

**Mechanical guidance:**

- `git stash` is fine for "switch branches, run something, come back" within a session.
- At session end, if a stash exists, **either pop it or convert it to a branch** with `git stash branch <wip-name> stash@{N}`.
- Stashes that survive multiple sessions are themselves the failure mode this rule names. Audit at session start (`git stash list`) and decide.

**Stash naming when long-lived is unavoidable:** include the date and reason in the stash message:

```bash
git stash push -u -m "auto-pre-branch-switch-$(date -u +%Y%m%dT%H%M%SZ): <one-line-reason>"
```

The `sync-pt-to-personal.sh` script and the auto-pre-pull mechanic in `git-discipline.md` Rule 2 both follow this convention.

## Stale-branch policy

A local branch is **stale** when ALL of these are true:

- It is ahead of `master` by ≥ 1 commit.
- It has not been pushed to any remote in the last 7 days.
- It has not received a new commit in the last 7 days.

Stale branches are not automatically deleted — they may hold real work. The mechanical layer (the extended `stale-active-plan-surfacer.sh` hook, item 8 of the initiative) surfaces them at SessionStart so the operator can decide.

**For each stale branch, the operator's decision tree:**

1. **Is the content shipped elsewhere?** Frequently the branch's commits are content-equivalent to commits that already landed on master via a different SHA path (cherry-pick, squash-merge, etc.). Confirm with `git diff <branch> master`. If empty diff → safe to delete: `git branch -D <branch>`.
2. **Is the work still relevant?** If the branch holds a hypothesis that turned out wrong or a feature that was descoped — delete it (after a final check that nothing on the branch is needed elsewhere).
3. **Is the work still in-flight?** Then push it, OR rename to `wip/<machine>-...` per the convention so future sessions see it as work-in-progress, OR convert to a tracked PR via `gh pr create`.
4. **Is it a backup/salvage from a specific moment?** Audit whether the underlying need is still active. If not, delete.

**Surfacing doesn't compel deletion.** The hook lists stale branches with their last-commit date and an ahead-count; the operator chooses what to do. Surfacing is the load-bearing mechanism — without it, stale branches silently accumulate.

The rule is documentation (Pattern-level). The two SessionStart surfacers are Mechanism. Together they make a session that starts with stale branches or unexpected uncommitted state see it at-a-glance, without the operator having to remember to check.

## Scope

Applies in every project whose Claude Code installation has the WIP-pattern list referenced by `session-start-git-freshness.sh` and the local-branch extension wired in `stale-active-plan-surfacer.sh`. The rule is loaded contextually by the harness; no opt-in or config flag is required. The naming convention binds every code session creating a branch; the stale-branch policy applies to the user's main checkout (where stale branches accumulate); the stash-lifetime guidance is universal.
