# Workstreams State — Write the Semantically True Tree, Not Just Any Well-Shaped Tree

> **Renamed 2026-06-01** (Workstreams reframe, Task 2b of `docs/plans/workstreams-phase-3.md`; conceptual rename in ADR 045). Formerly *Conversation-Tree State* (`conversation-tree-state.md`). The hook scripts referenced below were renamed `conversation-tree-*.sh` → `workstreams-*.sh` with 30-day backward-compat shims at the old names (deletable 2026-06-30). The subsystem's runtime state paths (`.claude/state/conversation-tree/`, the emit log/ledger dirs) are intentionally unchanged — only the script/rule FILENAMES were renamed.

**Classification:** Hybrid. The mechanical layer is two hooks: `workstreams-state-gate.sh` (PreToolUse, matcher `mcp__ccd_session__spawn_task|mcp__ccd_session_mgmt__start_code_task` — Dispatch orchestrator spawn tools ONLY; sub-agent `Task`/`Agent` are deliberately out of scope per ADR-031 r7 Pin-1 as amended r8 / ADR-034, because they are AI-internal mechanics (peer review, verification, internal helpers), not branches of the user↔AI conversation the tree models) blocks an individual child-session spawn unless the conversation-tree state file exists, is valid JSON, carries an attestation-verified snapshot, is fresh relative to the gate's per-session spawn marker, and names — in `snapshot.nodes` — the branch the spawn's `tool_input` describes; `workstreams-stop-gate.sh` (Stop) scans the agent-uneditable `$TRANSCRIPT_PATH` and blocks session end if a spawn occurred this session without a corresponding verified state-file write. Those two hooks enforce **file freshness, JSON shape, branch-name presence, and "spawned in transcript ⇒ must have written before Stop."** Everything below that — that the tree the orchestrator wrote is *semantically true* (the branches it names actually exist, their states actually reflect reality, the items genuinely waiting on the operator are the ones in the actions pane) — is the Pattern this rule documents. The orchestrator self-applies it; no hook can verify truth, only integrity.

**Ships with:** ADR-031 r7 (`docs/decisions/031-conversation-tree-ui-architecture.md`, "Enforcement design" + "Plan-safety pins") — the ACCEPTED Option-2 file-mediated state contract and the Mechanism+Pattern split — and ADR-032 §8 r2.1 (`docs/decisions/032-conversation-tree-state-schema.md`) — the enforcement-shaped layout and the snapshot-integrity attestation primitive that makes the branch-presence check a single delegated library call.

## Why this rule exists

The conversation-tree UI is a passive observer (ADR-031 r7 Option 2): the Dispatch orchestrator writes the conversation-tree / backlog / decision-action state to a well-known local file as it works; the GUI reads that file to surface the tree, "what's waiting on the operator," and which threads are still open. The contract is file-mediated — the GUI never spawns, never steers, never becomes the chat. Its only input is what the orchestrator wrote.

That makes the *writer's honesty* load-bearing. A GUI is only as truthful as the state file behind it. The two enforcement hooks raise the mechanical bar substantially for local Dispatch — they can mechanically force a fresh, well-shaped, branch-naming write before every spawn and before session end — but ADR-031 r7 is explicit about the ceiling: **a hook checks freshness/shape, not truth.** The strongest mechanical proxy ADR-032 §8 buys is requiring the spawned branch name to appear as a live key in an attestation-verified `snapshot.nodes` (raising the bar from "wrote anything" to "wrote an entry naming this branch" — non-gameable because the spawn `tool_input` supplies the branch independently of the file the writer controls). It cannot force the *content* to be correct, and it cannot reach genuine cloud sessions.

This rule codifies the unenforceable parts. It is the same Mechanism+Pattern split the rest of the harness uses (`spec-freeze.md`, `acceptance-scenarios.md`, `prd-validity.md`): the gates are the floor; the rule is the discipline that the floor cannot mechanize, with the operator's interrupt authority as the human backstop.

## The Pattern: write the *true* tree

When the orchestrator opens a branch, raises a decision, concludes a thread, promotes a backlog item, or takes any action the conversation-tree state schema (ADR-032) models, it MUST write the state file so that what it records is **semantically true**, not merely structurally well-formed and branch-naming.

Concretely, the orchestrator self-applies:

1. **The branch name written is the branch actually spawned.** The gate verifies a `snapshot.nodes` element exists naming the branch from `tool_input`. The Pattern requires that node to *be* the branch — same work, same intent, same scope — not a placeholder node titled to satisfy the key-presence check while the real spawned work is something else. The §8 bar ("a live node naming this branch") is satisfiable by a writer that writes *anything* and titles it after the spawn; the Pattern is what makes that not happen.
2. **Node states reflect reality.** A node marked `open` is genuinely a thread the operator could continue; a node marked `archived` is genuinely done/abandoned; a `decision-raised` event corresponds to a decision genuinely awaiting the operator. The GUI surfaces these verbatim — a node mis-stated as open clutters the operator's view with a dead thread; a live thread mis-stated as archived silently disappears from his tracker (the exact FR-24 "a minute or a month later, 100% of open branches are surface-able" failure the whole system exists to prevent).
3. **Items waiting on the operator are the items genuinely waiting on the operator.** The actions-list pane is driven by what the orchestrator wrote. An item that needs his decision but was never written as a decision-raised event is invisible to him; an item written as needing his follow-up that actually does not is noise that erodes trust in the pane.
4. **Schema designed so the most semantically-meaningful property reduces to a key-presence/shape check.** This is the writer-side corollary of ADR-031 r7's design instruction and ADR-032 §8's load-bearing obligation: the schema is shaped so the strongest mechanical proxy (branch-name-as-required-key in an attestation-verified snapshot) is as close to "the tree is true" as a shape check can get. The Pattern is the residual gap between that proxy and actual truth — and it is the orchestrator's job, not a future hook's, to close it. That residual gap is the documented ceiling of the mechanical layer (ADR-031 r7; ADR-032 §8 "Costs / accepted ceilings (a)").

The discipline is not "write something the gate accepts." It is "write the tree such that, if the operator read the GUI cold a month later, every branch / state / waiting-item it shows him is *true*." The gate guarantees he is shown a fresh, well-shaped, branch-naming write; this rule guarantees that write is honest.

## Accepted gaps (verbatim from ADR-031 r7 — surfaced, not hidden)

ADR-031 r7 enumerates three classes of work the mechanical layer cannot reach. They are accepted v1 limitations, rule-backed, with the operator's interrupt authority as the backstop — **not defects, and the GUI surfaces nothing false about them** (it shows only what it can see).

1. **Genuine cloud sessions are invisible and unenforced.** Claude Code on the web, `claude --remote`, and unattended `/schedule` Routines write **no local transcript** (`--teleport` only) and, separately, **do not load `~/.claude/` hooks** (cloud/Routines inherit project `.claude/` only, per Decision 011). Consequence: threads that run as true cloud sessions are invisible to the GUI and unenforced by the local hooks. This is an accepted v1 limitation, not a defect — the operator's actual workflow is desktop-app + phone-triggered Dispatch (all local, verified on his machine). Mitigation if cloud usage grows: commit the enforcement gates into the project's `.claude/` per Decision 011 Approach A so cloud sessions inherit them (this is enforcement-only, not visibility — a teleported transcript is still required for the GUI to see the thread); until then, cloud threads are out-of-tracker by construction.
2. **Sub-agent `Task`/`Agent` are deliberately OUT of scope (not a gap — a scoping decision); `Bash(claude…)` and `/schedule` remain an accepted gap.** Per ADR-031 r7 Pin 1 **as amended r8 (ADR-034, 2026-05-19)**, the enumerated set is **Dispatch-only**: (a) `mcp__ccd_session__spawn_task` / `mcp__ccd_session_mgmt__start_code_task` are **enforced** (both gates' matchers + the `workstreams-emit.sh --on-spawn` emitter cover exactly these two; the Stop-gate transcript scan detects only these); (b) **`Task` / `Agent` are explicitly NOT enforced and NOT emitted** — sub-agent invocations (code-reviewer, task-verifier, ux-designer, comprehension-reviewer, parallel `plan-phase-builder` dispatch, etc.) are AI-internal mechanics, not branches of the user↔AI conversation the tree exists to track. The conversation tree models the conversations Misha has *with AI via Dispatch*; anything a Claude Code session needs from him is already surfaced to him through Dispatch, so sub-agent workflow has no place in the tree (it would only pollute the operator's view with workflow noise and force spurious per-session waivers on every parallel-reviewer dispatch); (c) **`Bash(claude…)` and `/schedule` remain an accepted gap** — rare for the conversation-tree workflow and mapping to the same cloud/`--remote` blind spot accepted above; the GUI shows nothing false about them. Any change to the enumerated set (adding OR removing a surface) requires re-enumeration in ADR-031 r7 — the matcher is an explicit enumerated set, not an implicit pattern; this r8 narrowing is recorded in ADR-034.
3. **The documented semantic-correctness ceiling.** *Semantic correctness* of the tree is **NOT mechanically enforceable** — a hook checks freshness/shape, not truth. The strongest mechanical proxy is the branch-name-as-required-key bar (ADR-032 §8: verify the attestation, then assert a live `snapshot.nodes` element names the branch from `tool_input`). Above that proxy, truth is rule-class with the operator's interrupt authority as backstop. Directly binding spawned child sessions is also out of reach — the harness's own `spawn-task-report-back.md` convention exists precisely because the orchestrator has no mechanical callback into a spawned child, so enforcement binds the **orchestrator at spawn-time and Stop**, not the child. ADR-032 §8 records this as an accepted ceiling: "the §8 mechanical layer verifies branch-record-presence, not semantic tree correctness — the documented ADR-031 ceiling; semantic truth is the B3 rule-class layer + the operator's interrupt authority."

ADR-031 r7's answer to the operator's question ("is Dispatch beholden to the harness?"): **Yes, for local Dispatch — the harness can mechanically force a fresh, well-shaped, branch-naming write before every spawn and before session end. It cannot mechanically force the content to be correct, and it cannot reach genuine cloud sessions. Those gaps are rule-backed + accepted, not hidden.**

## How the two hooks compose (the Mechanism this rule documents the ceiling of)

- **`workstreams-state-gate.sh` — PreToolUse**, matcher `mcp__ccd_session__spawn_task|mcp__ccd_session_mgmt__start_code_task` (Dispatch-only; sub-agent `Task`/`Agent` no-op per ADR-031 r7 Pin-1 amended r8 / ADR-034). Blocks a child-session spawn unless the state file (a) exists, (b) is valid JSON, (c) carries an attestation-verified snapshot (ADR-032 §8 r2.1 — trust computed ONLY by the sole-normative state-library `verifySnapshotAttested` primitive, never by shell canonicalization), (d) has mtime newer than the gate's per-session spawn marker, and (e) has a live (`state != archived`) `snapshot.nodes` element whose `node_id` or `title` equals a branch identifier extracted **independently** from the spawn `tool_input` (ADR-031 r7 Pin-1 — the non-gameable bar). Pin-2 error partition: JSON-parse-fail → CLOSED; missing+prior-spawn → CLOSED; missing+no-prior → OPEN (bootstrap, a tree must be able to start); torn/stale → CLOSED; unknown schema major → CLOSED with the distinct "schema too new — upgrade" message; hook-internal error → OPEN (fail-open only on the gate's own malfunction). Release valve: a fresh `.claude/state/conv-tree-spawn-waiver-*.txt` (≥1 substantive line, <1h, mirrors `bug-persistence-gate.sh`).
- **`workstreams-stop-gate.sh` — Stop.** Scans `$TRANSCRIPT_PATH` (agent-uneditable) for a spawn/dispatch this session whose `tool_name` ∈ the Dispatch-only enumerated set (`mcp__ccd_session__spawn_task` / `mcp__ccd_session_mgmt__start_code_task`; sub-agent `Task`/`Agent` lines are ignored — a session whose only spawns were sub-agent reviewers/verifiers does NOT trip this gate, per ADR-031 r7 Pin-1 amended r8 / ADR-034); if ≥1 occurred, BLOCKs session end unless the state file passes the same `verifySnapshotAttested` check and its verified `snapshot.nodes` names a branch extracted independently from the transcript's spawn `tool_input`. Mechanically forces "spawned ⇒ wrote the true tree before Stop." Stop-hook decision/exit shape + fresh-`conv-tree-stop-waiver-*.txt` release valve + `lib/stop-hook-retry-guard.sh` integration mirror `bug-persistence-gate.sh` exactly. Fail-OPEN only on the gate's own malfunction; a torn/missing state file with a real spawn BLOCKs.

The PreToolUse gate catches an *individual* spawn lacking a true-tree write the moment it begins; the Stop gate closes the loop at the session boundary. Both delegate snapshot trust to the single sole-normative state-library primitive (ADR-032 §8 r2.1) — the gate never re-implements canonicalization. What neither can do is verify that the named, well-shaped, attestation-verified node is *semantically true* — that is this rule.

## Builder-dispatch work-item emission (ADR-054, 2026-06-10 — amends ADR-034's scoping for the WORK-ITEM tier)

ADR-034's conversation-BRANCH scoping stands: sub-agent `Task`/`Agent` (and `Workflow`) dispatches still produce **no tree node** — they are AI-internal mechanics, not branches of the user↔AI conversation. What ADR-054 adds is the **work-item tier**: every orchestrator builder dispatch now auto-emits ONE `action-added` work-item on the session's own `ss-*` node, mechanically, with no orchestrator discipline required (Misha's 2026-06-10 directive: tracking must happen automatically "even when you don't want to listen to me").

The mechanism (forcing-first, gate-second):

| Surface | Hook + mode | What it does |
|---|---|---|
| PreToolUse `Task\|Agent\|Workflow` | `workstreams-emit.sh --on-builder-dispatch` | Emits an idempotent creation batch via the `state.js` facade: root + session `ss-*` `branch-opened` (same deterministic event ids as `--on-session-start` — no duplication) + `action-added` (`wi-bd-<sha1(sid\|tool\|title)>`, title from `.description // .meta.name // .name // .title // first prompt line`) + `item-details-set` (`_category: "builder-dispatch"`, tool, subagent_type, background flag). Writer — never blocks; every path exits 0. |
| PreToolUse `Task\|Agent\|Workflow` (wired AFTER the emit) | `workstreams-state-gate.sh --builder-tracking` | THIN fail-closed substrate gate: BLOCKS the dispatch when tracking is possible (node + state library present) but the canonical state file is missing/unwritable — a dispatch may not proceed untracked when tracking is possible. Degrades OPEN when the subsystem itself is absent (bootstrap rule). Valves: fresh substantive `.claude/state/builder-tracking-waiver-*.txt` (<1h), `WORKSTREAMS_BUILDER_GATE_DISABLE=1`. Deliberately NOT a per-dispatch emit verification (that would false-positive-block on any transient flake — the reconciler is the per-dispatch catch-up). |
| PostToolUse `Task\|Agent\|Workflow` | `workstreams-emit.sh --on-builder-complete` | Foreground dispatches: tool return IS sub-agent completion → `action-done` (item leaves In-flight). Background dispatches (`Workflow`, `run_in_background: true`): launch-ack only → creation batch, NO done. |
| Stop (existing reconciler wiring) | `workstreams-emit-reconciler.sh` builder sweep | Re-derives every builder `tool_use` from the agent-uneditable transcript; catch-up-fires `--on-builder-dispatch` for each and `--on-builder-complete` for each with a present `tool_result` (idempotent event ids make re-fires no-ops). |

**Noise control:** `details._category: "builder-dispatch"` is outside the GUI's Misha-ask category set, so builder items can never land in Awaiting-me; unchecked + no explicit state derives `in-flight` — exactly the work-in-motion tier.

**The honest completion ceiling (Rule 7):** background dispatches have NO stable local completion contract — no per-workflow completion hook event exists, and the wake/notification message shape is undocumented. Their items honestly stay in-flight (FR-7 keeps the owning session node un-concludable — intentionally visible) until a later `--resolve-item`, the operator, or a future upstream hook surface resolves them. This is a named gap, not a solved problem. Cloud sessions that load no `~/.claude/` hooks remain out of reach (the same accepted blind spot as above).

## Scope

This rule applies in any project whose Claude Code installation has the two conversation-tree gates wired in `settings.json` AND the conversation-tree-ui module present. Adoption is implicit on harness install/sync (`install.sh` propagates `adapters/claude-code/settings.json.template` → live `~/.claude/settings.json` and the two hook scripts → `~/.claude/hooks/`). A project without the conversation-tree-ui module sees the gates degrade gracefully: with no state file and no prior spawn, the PreToolUse gate ALLOWs (bootstrap exemption), and the Stop gate ALLOWs when no enumerated spawn appears in the transcript. The Pattern binds the Dispatch orchestrator specifically — it is the sole writer of the file-mediated state contract; the passive GUI never spawns and therefore never triggers either gate.
