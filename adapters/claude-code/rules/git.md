# Git Standards

- Commit at natural milestones, not after every small change
- Clear commit messages: `<type>: <description>` (types: feat, fix, refactor, test, docs, chore)
- **Push commits when work reaches a natural completion point** — feature branch ready for review, atomic commit landed and verified, plan task task-verified PASS. Default is to push, not to wait. Use safe methods only.
- **Safe push methods.** **NEVER force-push.** No `git push --force`, no `git push -f`, no `git push --force-with-lease` — to any branch, protected or not. The full prohibition and three safe alternatives (merge instead of rebase; new-branch + cherry-pick + fresh PR; first-push if never pushed) live in `~/.claude/rules/git-discipline.md` Rule 1. Never use `--no-verify` to bypass pre-commit/pre-push hooks. Never use `--no-gpg-sign` if signing is enabled. Always honor branch-protection rules.
- **After every merge to master, sync the user's main checkout.** When a session has merged work to `master` (via `gh pr merge`, direct push, etc.), `cd` into the user's main checkout (NOT the worktree — the parent of `git rev-parse --git-common-dir`) and run `git fetch origin && git pull --ff-only origin master`. Handle uncommitted changes in the main checkout via `git stash push -u`, pull, then `git stash pop` (surface stash-pop conflicts in the final report; do not auto-resolve). Full mechanic in `~/.claude/rules/git-discipline.md` Rule 2.
- **Customer-tier branching policy.** Pre-customer projects (no real users yet): pushing feature branches and merging to master (which deploys to production) is acceptable; direct master commits are also acceptable when convenient (small docs fixes, ancillary edits during a session that's already on master). The cost of a bad push is mostly self-inflicted and reversible via revert + redeploy. **Once a project has real users**: all work must go through a `dev` or `preview` branch with deployment-validation gates passing before merge to the customer-facing branch (typically `master` or `main`). The harness expresses this via the `automation-mode.json` per-project config (see "Auto-merge" rule below): pre-customer = `mode: full-auto`; customer-facing = `mode: review-before-deploy`. The config is the mechanical signal; per-project judgment determines when to flip it.
- **Auto-merge feature branches to master when work is verified-complete.** Per pre-customer policy + user directive 2026-05-05 ("don't wait for me on this"): when a feature branch's work is fully task-verified, all hooks pass, and the branch is pushed, the agent merges to master autonomously — do not pause for user authorization. Use `--no-ff` merge commits to preserve the feature-branch audit trail (squash loses too much history for harness-dev work). Push master after merge via the multi-push origin remote. Skip only when the user has explicitly said "preview only" / "don't merge" / "wait" in their most-recent message, OR when the branch contains genuinely unreviewed risky work (irreversible operations, schema migrations, auth changes — Tier 3 in `planning.md`). Cross-reference: `~/.claude/rules/deploy-to-production.md` (the same default-to-deploy discipline applied to merge timing). **Mechanism:** the `automation-mode.json` per-project config governs whether deploy-class commands pass through (`mode: full-auto`) or pause for human review (`mode: review-before-deploy`). Set per-project via `<project>/.claude/automation-mode.json` (or via `/automation-mode full-auto --project`); inherited from `~/.claude/local/automation-mode.config.json` when no per-project file exists. Pre-customer projects opt into `full-auto`; flip back to `review-before-deploy` when a project launches to real users.
- **Direct master commits and pushes are acceptable on pre-customer projects** (per user directive 2026-05-05 — "please disable the rule that prohibits master pushes"). Feature branches remain the preferred shape for substantial work because they preserve atomic audit trails, but direct master operations no longer require explicit per-action authorization. Once a project crosses the customer-tier boundary, the prohibition returns automatically (handled at branch-protection level via GitHub, not at the harness rule level).
- PR descriptions: what changed, why, how to test; call out breaking changes
- Never leave uncommitted work at session end — commit or stash
