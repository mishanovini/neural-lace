# Frontend Conventions — compact
> Enforcement: Pattern — self-applied; verified by domain-expert-tester / ux-end-user-tester agents post-build. Sources: ux-standards, ux-design, ui-components, react.
> Applies: `src/components/**`, `src/app/**/page.tsx`, any `.tsx`/`.jsx`.

- **Color semantics:** Red = error/critical ONLY (never informational — use amber). **Purple = AI only** — every AI feature/badge/sparkle uses purple; never use purple for generic actions like Save/Create/Launch (use blue). Blue=info, Green=success, Amber=warning, Gray=inactive. Color is never the only signal — pair with text/icon.
- **Contrast, all modes:** buttons need a **filled background**, never outline-only (`bg-blue-600 text-white` primary, `bg-purple-600 text-white` AI actions). Every border/text class needs an explicit `dark:` variant (`border-gray-300 dark:border-gray-600` min; `text-gray-500 dark:text-gray-400` min). No opacity (`/20`,`/30`) on primary container backgrounds — use solid colors.
- **Every async data display needs all four states:** loading (skeleton + descriptive text), empty (icon + explanation + first-action button), error (specific cause + recovery action), success (confirm + next step). Every card is clickable (`cursor-pointer hover:shadow-md`) and links to its data — no dead-end numbers.
- **UX principles (all states guide to resolution):** errors suggest a solution; suggestions link directly to the action; empty states offer a first action; destructive actions confirm with reversibility info; success states reveal what's next; loading states describe what's loading; warnings by color (yellow=fixable, red=blocking).
- **Prerequisite unblocking — never a dead end:** when a user attempts an action whose prerequisite is unmet, the app never merely blocks. The surface either opens an **inline modal that satisfies the prerequisite right there** (self-contained prerequisites — a missing record, field, or one-step setup) or presents a **direct deep link to the page where it can be satisfied** (page-level configuration) — and in both cases states what is missing and why: "You need X to do Y." Anti-patterns (all defects): disabled controls with no explanation; toast-only rejections; error text naming a destination without linking it; silent no-ops. Enforcement: review checklist — **no new precondition ships without its unblocking affordance**; the feature's acceptance scenarios MUST cover the blocked-state UX. (Operator directive 2026-07-06; first codified downstream as that project's ADR 084.)
- **AI features:** AI-generated content gets a purple `✦` + "AI-generated" label; AI suggestions show an editable preview and NEVER overwrite original text until Accept is clicked.
- **Every prop/data field rendered in JSX must be traced**: source, exact API field name, optional/nullable + fallback. Conditional rendering: confirm at least one real DB record satisfies the condition and the test user has the required role. New interactive elements: confirm the click handler is wired AND visible; add `data-testid`.
- **Semantic HTML:** `button` not `div onClick`, `a` not `span onClick`. ARIA labels on icon-only buttons. Keyboard-navigable. Error boundaries so one broken component doesn't white-screen the app. Server Components by default in Next.js; minimize `"use client"`.
- Verify with: `node scripts/verify-ui.mjs http://localhost:3000/<page> '[data-testid="<id>"]'`. Before committing: glance test ("3-second understand"), all four states present, purple used consistently for AI, mobile responsiveness checked.
