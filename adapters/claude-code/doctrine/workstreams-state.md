# Workstreams State — compact
> Enforcement: workstreams-state-gate.sh (PreToolUse spawn), workstreams-stop-gate.sh (Stop), workstreams-emit.sh (writer). Full: doctrine/workstreams-state-full.md
> Applies: every Dispatch orchestrator spawn (mcp__ccd_session__spawn_task / start_code_task) — Dispatch-only, not sub-agent Task/Agent.

- The tree is only as truthful as what the orchestrator writes. Write the **semantically true** tree, not just a well-shaped one: gates verify freshness + JSON shape + branch-name presence in `snapshot.nodes`; they cannot verify the node IS the real work, that `open`/`archived` states reflect reality, or that waiting-on-operator items are genuinely waiting on the operator. That's the Pattern — orchestrator self-applies it.
- Snapshot trust flows through ONE sole-normative primitive: the `state.js` facade's `verifySnapshotAttested`. Never re-implement canonicalization in shell.
- Scope is Dispatch-only (ADR-034): `mcp__ccd_session__spawn_task` / `start_code_task` are enforced and emit `branch-opened`. Sub-agent `Task`/`Agent`/`Workflow` produce NO tree node — AI-internal mechanics, not conversation branches; polluting the tree with them would bury real waiting-items in workflow noise.
- ADR-054 builder-dispatch tier: every builder `Task`/`Agent`/`Workflow` dispatch auto-emits ONE `action-added` work-item on the session's own `ss-*` node (`_category: "builder-dispatch"`, outside the operator-ask set — never lands in Awaiting-me). Foreground completion auto-emits `action-done`; background (`Workflow`/`run_in_background`) has NO stable completion contract and honestly stays in-flight — a named gap, not solved.
- Four defense-in-depth emit layers: A (tool-wrapped PreToolUse/SessionStart/Stop writer, never blocks), B (Stop-hook transcript reconciliation, auto-fills missed emits), C (5-min heartbeat concluding stale sessions), D (agent discipline — the only enforcement genuine cloud Dispatch gets, since cloud loads no local hooks).
- Context-complete emission: every operator-facing `decision-raised`/`question-raised`/`action-added` SHOULD carry a sibling `item-details-set` validated by the sole-normative `decision-context-schema.js` (`background` + ≥1 actionable field per kind). Invalid/absent payloads never block the emit — item lands honestly detail-less + a WARN to the emit log. The render side (not this layer) refuses to present a contextless item as actionable.
- Accepted gaps, not defects: genuine cloud sessions (no local transcript, no `~/.claude/` hooks) are invisible and unenforced; `Bash(claude…)`/`/schedule` are out of scope; semantic correctness above the branch-presence proxy is never mechanically enforceable — operator interrupt authority is the backstop.
