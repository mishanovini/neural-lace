## Summary

Closes the inline-PR-body validation gap that was generating ~19 "PR Template Check failed" emails per week (~12 distinct branches) and forcing a 2-push amend cycle on every AI-spawned PR. Ships `pr-template-inline-gate.sh` (PreToolUse `Bash` hook) as a sibling to `vaporware-volume-gate.sh` — both fire on the same matcher, neither extends the other. The new gate intercepts `gh pr create` and `gh pr edit`, extracts the inline body from `--body` / `--body=` / `--body-file` / heredoc forms, and pipes it through the canonical `.github/scripts/validate-pr-template.sh` library already used by the CI workflow and the local pre-push hook (same regex, same stderr, single source of truth).

Tracked as HARNESS-GAP-43 in `docs/backlog.md`; failure class catalogued as FM-030 in `docs/failure-modes.md`.

## What changed and why

**Root cause.** The existing `pre-push-pr-template.sh` git hook only validates `.pr-description.md` files or the latest commit message body — never the inline `--body` argument that AI sessions virtually always use (`gh pr create --body "$(cat <<'EOF' ... EOF)"`). So the first push reaches GitHub, the server-side `PR Template Check` workflow correctly fails for missing template sections, an email is sent, and the AI session has to issue a second push amending the body. Pure noise; no signal.

**Files added/modified:**

- `adapters/claude-code/hooks/pr-template-inline-gate.sh` (NEW, 528 lines). PreToolUse `Bash` hook. Self-detects `gh pr create` / `gh pr edit`. Body extraction uses bash parameter expansion (not `sed` line-mode) so multi-line bodies parse correctly. Sources the canonical validator and surfaces its stderr verbatim on FAIL with structured remediation. 11-scenario `--self-test` flag (T1-T11). Pass-through cases: `--fill`, no body source, non-gh-pr-create/edit commands. Explicit block on `--body-file -` (stdin) with hint to use a real file.
- `~/.claude/hooks/pr-template-inline-gate.sh` (NEW, byte-identical mirror).
- `adapters/claude-code/settings.json.template` (+8 lines): wired the new hook on the same `Bash` PreToolUse matcher immediately after `vaporware-volume-gate.sh`.
- `~/.claude/settings.json` (+8 lines, byte-identical mirror).
- `docs/failure-modes.md` (+11 lines): new `FM-030 — Inline PR body bypasses local template validator; server-side CI is the first checkpoint`.
- `docs/backlog.md` (+5 lines): new `HARNESS-GAP-43 — Inline gh pr create --body bypasses local PR-template validator` entry; `Last updated:` line refreshed to 2026-05-24 v45 with the new shipment summary.
- `docs/harness-architecture.md` (+1 row): inventory entry for `pr-template-inline-gate.sh` immediately after the existing `vaporware-volume-gate.sh` row.

**Decision: sibling hook, not extension of `vaporware-volume-gate.sh`.** Different concerns (volume heuristic for describes-vs-executes ratio vs template-content validation), different escape hatches (PR title prefix vs validator rules), and conflating them would push `vaporware-volume-gate.sh` past 500 lines while making both checks harder to reason about in isolation. Sibling preserves clean self-test boundaries.

**Three-layer defense now in place:**
1. `pr-template-inline-gate.sh` — intercepts `gh pr create / pr edit` inline-body calls.
2. `pre-push-pr-template.sh` — covers `.pr-description.md` + commit-message paths at `git push` time.
3. `.github/workflows/pr-template-check.yml` — final server-side checkpoint (unchanged).

## What mechanism would have caught this?

**(b) New catalog entry proposed.** `FM-030 — Inline PR body bypasses local template validator; server-side CI is the first checkpoint` has been added to `docs/failure-modes.md` in the same commit. The entry catalogs the failure class (AI sessions constructing PRs via `gh pr create --body "$(cat <<EOF ... EOF)"` whose body is never seen by `pre-push-pr-template.sh` because that hook only checks `.pr-description.md` and commit messages), names the prevention mechanism (`pr-template-inline-gate.sh` PreToolUse `Bash` hook shipping in this PR), and provides Discriminator + Recovery fields per the standard six-field schema. The existing failure modes FM-001 through FM-029 did not cover this class — FM-006 and adjacent vaporware-shape entries gate on completion claims, but the inline-body bypass is a structural gap in the local validation surface, not a vaporware-narrative issue. The new entry sits chronologically after FM-029 and references the originating signal (~19 "PR Template Check failed" emails in one week before this gate landed).
