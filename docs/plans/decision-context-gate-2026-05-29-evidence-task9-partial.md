### Spec meaning

The two new hooks (`decision-context-gate.sh` Stop hook from Task 4 commit `a2c61a8`, `decision-context-reply-emit.sh` UserPromptSubmit hook from Task 6 commit `cd95b3b`) already exist as scripts on disk in both the canonical adapter tree and the live `~/.claude/hooks/` mirror, but they DO NOT fire automatically until Claude Code's hook dispatcher reads them out of `settings.json`. This task's job is purely the wiring step: append the two new entries to the right hook chains in BOTH layers of the two-layer config (`adapters/claude-code/settings.json.template` is canonical/committed/source-of-truth for downstream installs; `~/.claude/settings.json` is the per-machine live copy the dispatcher actually reads on THIS machine). The Stop hook is appended immediately AFTER `goal-coverage-on-stop.sh` (the last Gen-6 narrative-integrity Stop hook, position 8 of the existing chain), per ADR 047 and the `decision-context.md` rule's enforcement-map specification. The UserPromptSubmit hook is appended immediately AFTER `goal-extraction-on-prompt.sh` (the existing Gen-6 A1 first-message-goal-extraction prompt hook). Both layers must remain valid JSON post-edit, and the harness's existing `settings-divergence-detector.sh` must report no NEW divergence between template and live (the new entries must be present in BOTH files at the same positions).

### Edge cases covered

- **Two-layer config discipline.** Per `harness-maintenance.md`, edits to settings hooks must land in both the canonical template AND the live mirror. The Edit calls touched both files at the same lifecycle positions (Stop after `goal-coverage-on-stop.sh`, UserPromptSubmit after `goal-extraction-on-prompt.sh`).
- **JSON validity preserved.** Ran `jq . <file> > /dev/null` on both files post-edit; both validate. Bracket / comma placement matches the existing entry shape (`{ "type": "command", "command": "bash ~/.claude/hooks/<name>.sh" }`).
- **Settings-divergence non-regression.** Pre-wiring `diff -u <(jq -S template) <(jq -S live) | wc -l` = 195. Post-wiring = 195. The two new entries land in BOTH files so they do not contribute to divergence; the residual 195 lines are pre-existing baseline (live machine omits hooks like `principles-compliance-gate.sh`, `conv-tree-emit-reconciler.sh`, `conversation-tree-extract-pending.sh` per the operator's machine-local choices, plus various unrelated permission/env diffs).
- **Live settings.json is gitignored.** `git status -s` after both edits shows only `M adapters/claude-code/settings.json.template` — the live edit is NOT in the commit. This is correct: the canonical template is what propagates to other machines via `install.sh`; the live edit is the operator-machine-local copy. Downstream consumers will pick up the wiring via their next `install.sh` run.
- **Correct chain position.** Stop hook goes AFTER `goal-coverage-on-stop.sh` (Gen-6 position 8 — the LAST narrative-integrity hook in the canonical Stop chain), placing the new decision-context-gate at the "position 9" slot conceptually. UserPromptSubmit hook goes AFTER `goal-extraction-on-prompt.sh` so the first-message-goal extraction continues to fire first (decision-context-reply-emit must see the prompt AFTER goal extraction has tagged it).
- **Existing matcher pattern preserved.** Both target chains have a single entry with `"matcher": ""` (empty matcher = fire on every event). The new hooks slot into that same entry's `hooks` array, NOT into a new sibling entry — consistent with the convention every existing same-class hook follows.
- **Hygiene scanner clean.** `harness-hygiene-scan.sh` ran silently on the staged template diff (exit 0, no findings). The two new lines are pure `bash ~/.claude/hooks/<name>.sh` invocations with no identifiers, paths-with-usernames, or sensitive substrings.

### Edge cases NOT covered

- **Cross-machine propagation timing.** The live `~/.claude/settings.json` edit on THIS machine fires the new hooks immediately on the next Claude Code session start. On OTHER machines, the new hook entries arrive only when the operator there runs `install.sh` (which copies `settings.json.template` → `settings.json` only when the live file does not exist; existing live files are NOT auto-merged). Out of scope for this task — downstream operators are responsible for propagating template changes to their live configs, and HARNESS-GAP-14 already tracks the per-hook reconciliation methodology.
- **What the hooks DO at runtime is out of scope here.** Task 9-partial verifies the wiring fires; it does NOT exercise the hooks against a real session transcript. That's Task 10's Walking Skeleton job — this is its predicate.
- **No removal of the pre-existing 195-line baseline divergence.** Pre-existing divergence on other unrelated entries is not within Task 9-partial scope and would be a separate harness-maintenance pass.
- **No verification that the Stop chain still terminates in a sane order under failure.** All Stop hooks compose via separate processes; the new `decision-context-gate.sh` is independent of `principles-compliance-gate.sh` and `session-wrap.sh refresh` that follow it (template) / precede `session-wrap.sh refresh` (live). The Stop-hook retry-guard library backs all blocking Stop hooks; this hook inherits that substrate.

### Assumptions

- Claude Code's hook dispatcher reads `~/.claude/settings.json` on session start and fires hooks in the order they appear in each event's `hooks` array. Empty `"matcher": ""` means "fire on every event of this type." This matches the existing pattern of every other hook in the chain (Stop hooks like `pre-stop-verifier.sh`, UserPromptSubmit hooks like `goal-extraction-on-prompt.sh`) and is the documented behavior in `~/.claude/CLAUDE.md` "Enforcement substrate" + `docs/harness-architecture.md`.
- `bash ~/.claude/hooks/<name>.sh` is the canonical invocation form (every existing hook uses it). The `~` is expanded by bash, not by Claude Code's hook dispatcher, but every existing hook entry uses this exact form so it must work.
- The two hook scripts (`decision-context-gate.sh` and `decision-context-reply-emit.sh`) at `~/.claude/hooks/` are executable (`-rwxr-xr-x` confirmed) and already mirror the canonical adapter copies byte-identically (both stat at 46635 bytes / 28002 bytes respectively). Task 4 and Task 6 builders already ran the install/sync step; this task does NOT need to re-sync the hook bodies.
- The Stop-chain order from `pre-stop-verifier.sh` through `goal-coverage-on-stop.sh` represents the canonical "narrative-integrity comes last" ordering; placing `decision-context-gate.sh` immediately after is consistent with treating it as a sibling Gen-6 narrative-integrity hook (it enforces decision-context fence integrity, which is structurally adjacent to first-message-goal extraction).
- The `settings-divergence-detector.sh` script's "differ" signal (exit code + stderr message) is presence-of-difference, not a divergence count. The diff-line count via `wc -l` of `diff -u <(jq -S template) <(jq -S live)` is the operator-friendly proxy for divergence magnitude, and was used as the pre/post comparison metric.
