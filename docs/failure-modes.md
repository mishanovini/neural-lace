# Failure Mode Catalog

> **Purpose.** Canonical, sanitized catalog of known harness failure classes. Every new failure that surfaces during a session should either extend an existing entry (if it is the same class) or be added as a new entry (if it is a new class). Hooks, agents, and skills consult this file when evaluating uncertain claims and known-bad patterns. Every failure that is not encoded here is a failure that will repeat.

> **Scope.** Failure CLASSES, not individual incidents. Each entry generalizes from one or more concrete observations. Sanitized — no codenames, no personal identifiers, no real incident dates tied to a specific product.

> **Catalog plan.** This file is the artifact produced by the `failure-mode-catalog` plan (see `docs/plans/archive/failure-mode-catalog.md`). The companion `capture-codify-pr-template` plan wires this catalog into PR-time mechanism analysis — every PR's mechanism field references entries here by `FM-NNN` ID under answer form (a), or proposes new entries here under answer form (b).

## Schema

> **Cross-project convention.** This file is the *reference instance* of the harness-wide FM-catalog standard documented at `docs/conventions/failure-mode-catalogs.md` (ratified by Decision 033). Every project has exactly one such file at `docs/failure-modes.md`.

Every entry uses the same six required fields, plus two optional fields added by Decision 033. The two optional fields are **additive and backward-compatible**: entries that predate them (FM-001..FM-023) need no migration, because every consumer reads by `Symptom` phenotype or by `FM-NNN` ID, never positionally by field count.

Required:

- **ID.** `FM-NNN` ascending. Never recycled. Renaming an entry preserves the old ID.
- **Symptom.** What an operator or user observes when this failure manifests, in one or two sentences. Written so that a future session diagnosing a similar event can search the catalog by phenotype. **This is the primary grep target.**
- **Root cause.** What in the system actually produced the symptom. Names mechanism, not blame.
- **Detection.** Where this class can be caught — which hook, agent, skill, or review step is in the position to surface it. If detection is purely behavioral today, say so explicitly so the gap is visible.
- **Prevention.** What stops the class at the source — the hook, rule, agent, template, or workflow that closes the loop. If prevention is partial or aspirational, say so honestly.
- **Example.** One sanitized concrete instance of the class, in generic terms.

Optional (Decision 033 — populate whenever they add signal; they are the highest-leverage fields for the investigation-first use case):

- **Discriminator.** How to tell *this* FM apart from look-alike FMs that share surface symptoms — the single observation or command that distinguishes it. This is the field that shortcuts a multi-hour investigation's dead ends.
- **Recovery.** The immediate human steps to get *unstuck right now* (distinct from Prevention, which is mechanism-facing). What an investigator does in the next five minutes.

## How to extend

When a new failure surfaces during a session:

1. Read this catalog top-to-bottom. If the failure phenotype matches an existing **Symptom**, extend that entry's Example list rather than create a new one.
2. If the root cause is a new class, append a new entry with the next `FM-NNN` ID. Use generic terms — no codenames, no real names, no real incident dates tied to a specific product, no absolute paths containing usernames.
3. Update **Detection** and **Prevention** if the new instance reveals a new way to catch or stop the class.
4. Reference the catalog entry from any related rule, hook, or agent change in the same commit.

The diagnosis rule (`rules/diagnosis.md`, "After Every Failure: Encode the Fix") makes this an explicit step in the post-failure workflow. The `harness-lesson` and `why-slipped` skills check this catalog first before proposing a brand-new mechanism.

## FM-001 — Concurrent-session plan wipe

- **Symptom.** A plan file in `docs/plans/` that existed and contained substantive content earlier in a session is empty, missing, or reverted to a stub when read later. The session that authored the plan has no edit in its history that explains the change.
- **Root cause.** A second concurrent Claude Code session (often a sub-agent build, a UX testing run, or a parallel orchestration thread) overwrote or replaced the plan file's contents because the in-flight version was not yet committed to git. The shared filesystem has no per-session locking; whichever session wrote last won.
- **Detection.** `plan-lifecycle.sh` PostToolUse hook surfaces a loud `[plan-uncommitted-warn]` warning when a new plan file is written but not yet committed. `pre-stop-verifier.sh` re-warns at session end if uncommitted plans remain. Neither blocks; both raise the chance the author commits before the wipe-out.
- **Prevention.** Commit the plan file in the same edit cycle that creates it, before any sub-agent dispatch or parallel work begins. Treat an uncommitted plan as transient state at risk from any concurrent activity. The `planning.md` "Plan File Lifecycle" section codifies this as the creation stage discipline.
- **Example.** A plan was authored and saved to `docs/plans/<slug>.md`, then the orchestrator dispatched a UX testing sub-agent on a different surface. The sub-agent's filesystem activity coincided with a separate orchestrator pass that re-templated the plan path, leaving the original plan empty. The work was reconstructed from conversation transcript only because the session was still open.

## FM-002 — Mysterious effort-level reset

- **Symptom.** A session that began at one configured effort level (`high` or `xhigh`) is observed mid-session running at a lower effort (`medium` or `low`), without any user-visible setting change. The agent's reasoning quality, tool-call density, and willingness to verify drop accordingly.
- **Root cause.** Effort level is read from `settings.json` and is not re-validated mid-session against the user's declared minimum. If a hook, plugin, or background process rewrites `settings.json` during the session — or if a sub-agent inherits a different config path — the effort level can shift silently. The session has no in-context audit of "what effort level am I currently at?"
- **Detection.** `effort-policy-warn.sh` runs at SessionStart and warns when the configured effort is below the project-level (`.claude/effort-policy.json`) or user-level (`~/.claude/local/effort-policy.json`) declared minimum. There is no mid-session detector — the warning is start-time only, so a mid-session drop is invisible until the next SessionStart.
- **Prevention.** Declare a minimum effort level in `~/.claude/local/effort-policy.json` so that any future SessionStart catches drift. The shipped `settings.json.template` declares `effortLevel: "max"` as the default for fresh installs so that the failure mode is structurally rare. Mid-session detection remains an open gap; the partial mitigation is operator awareness via the SessionStart warning.
- **Example.** An automation task ran for an extended period at what appeared to be reduced reasoning capacity. Subsequent inspection of `settings.json` at session end showed the effort level had been changed during the session. The automation's outputs from the affected window had to be re-evaluated.

## FM-003 — Bug-persistence trigger fired without persistence

- **Symptom.** During a session, the agent uses a trigger phrase that signals a bug or gap was identified ("we should also build X", "as a follow-up", "this is missing", "for next session") — but no `docs/backlog.md` entry, `docs/reviews/YYYY-MM-DD-*.md` file, or plan-evidence-log update is written before session end. The next session has no record that the gap was ever observed.
- **Root cause.** Identifying a bug and persisting a bug are separate operations. The agent's working memory holds the observation while it continues the current task, and "I'll write it down later" loses the observation when context shifts. Without a persistence step in the same response as the observation, the gap evaporates.
- **Detection.** `bug-persistence-gate.sh` Stop hook scans the session transcript for trigger phrases. If matches exist AND no change to `docs/backlog.md` or new `docs/reviews/YYYY-MM-DD-*.md` file is present in the working tree or recent commits, the hook blocks session end. Escape hatch: a `.claude/state/bugs-attested-YYYY-MM-DD-HHMM.txt` file with per-match justification documents legitimate false positives.
- **Prevention.** The `testing.md` "Bug Persistence" rule mandates writing the backlog or review entry in the same response as the observation, not later. The trigger-phrase list in the rule mirrors the patterns the Stop hook scans. The gate makes the rule mechanical: forgetting to persist is caught at session end, not lost.
- **Example.** A session investigating a workflow surfaced approximately two dozen distinct issues across state machine integration, harness gaps, and prompt content. Most were mentioned in conversation but not written to the backlog. The list had to be reconstructed retroactively from chat history, with high risk of omission. After this rule landed, the same class of session writes each gap to the backlog as it is observed.

## FM-004 — Verbose plan with placeholder-only required sections

- **Symptom.** A plan file in `docs/plans/` has all seven required headings (Goal, Scope, Tasks, Files to Modify/Create, Assumptions, Edge Cases, Testing Strategy) but one or more sections contain only `[populate me]`, `TODO`, `...`, or template-default placeholder text. The plan reaches the implementation stage with hidden gaps.
- **Root cause.** The required-section structure was treated as a checklist of headings rather than a checklist of substance. The author copied the template, filled the easy sections (Goal, Tasks), and left the harder ones (Assumptions, Edge Cases) as placeholders intending to fill them later. The builder then started building against an under-specified plan.
- **Detection.** `plan-reviewer.sh` Check 6b enforces both presence (the heading must exist) and substance (each required section must contain at least 20 non-whitespace characters of non-placeholder content). Sections consisting solely of placeholder tokens are rejected at pre-commit time.
- **Prevention.** The `templates/plan-template.md` ships with prompts in each placeholder explaining what the section should contain, raising the friction of leaving them empty. The `verbose-plan` skill exists to fill gaps in thin plan files before commit. The mandatory rule in `planning.md` ("Verbose Plans Are Mandatory") makes the substance requirement explicit — empty Assumptions on a "trivial" plan is exactly the case where the section is most valuable.
- **Example.** A plan file was authored with all seven headings populated except Assumptions, which contained only `[populate me]`. The pre-commit gate blocked the commit; the author added two real assumptions ("the existing API behaves as documented", "the feature flag is enabled in the target environment") and the plan passed. Both assumptions turned out to be load-bearing during implementation.

## FM-005 — Untracked plan file location ambiguity

- **Symptom.** A hook, agent, or session looks for a plan file by slug at `docs/plans/<slug>.md` and reports "plan not found" — but the file exists at `docs/plans/archive/<slug>.md` because it has already transitioned to a terminal status. Verification, completion reports, or reminder hooks fail with a misleading error.
- **Root cause.** The plan-file lifecycle moves plans from `docs/plans/<slug>.md` to `docs/plans/archive/<slug>.md` when their `Status:` transitions to a terminal value (COMPLETED, DEFERRED, ABANDONED, SUPERSEDED). Callers that hard-code the active path don't find the file post-archive. Without an archive-aware resolver, every consumer has to re-implement the fallback.
- **Detection.** Symptom is observable via "plan not found" messages from hooks and agents. The `find-plan-file.sh` helper emits a `resolved from archive: <path>` stderr note when the match comes from the archive subdirectory, making archive-aware lookups explicit.
- **Prevention.** Use `~/.claude/scripts/find-plan-file.sh <slug>` for any code path that resolves a plan by slug. The script prefers active and falls back to archive transparently. Hooks and agents that previously hard-coded `docs/plans/<slug>.md` were migrated to the helper as part of the 2026-04-23 plan-lifecycle work. Hooks intentionally NOT made archive-aware (`pre-commit-gate.sh`, `backlog-plan-atomicity.sh`, `harness-hygiene-scan.sh`, `plan-edit-validator.sh`) are scoped to active-work enforcement; documenting that exception is part of the prevention strategy.
- **Example.** A post-tool reminder hook checked `docs/plans/<slug>.md` for an unchecked task matching an edited source file. After the plan was archived on completion, the hook stopped firing for any continuing edits — the slug no longer resolved at the active path. Migrating the hook to the archive-aware helper restored the reminder behavior without breaking active-plan semantics.

## FM-006 — Self-reported task completion without evidence

- **Symptom.** A plan task's checkbox is flipped from `- [ ]` to `- [x]` in the plan file, but no corresponding evidence block exists in the companion `<plan>-evidence.md` file, or the evidence block contains only plain-text manual verification ("checked in browser", "verified manually") with no replayable command.
- **Root cause.** Self-report failure mode: the builder marks its own work complete based on "I think this works" or "tests pass locally" without producing reproducible evidence that the user-observable outcome was exercised. Pre-Gen-4, the only enforcement was social — the builder was expected to be honest. In practice, multi-task plans accumulated checked tasks where the work was incomplete or untested.
- **Detection.** Three layers. (a) `plan-edit-validator.sh` PreToolUse hook blocks the checkbox flip unless a fresh evidence file with matching Task ID and a `Runtime verification:` entry exists within a 120-second window. (b) `runtime-verification-executor.sh` Stop-hook check actually executes every `Runtime verification:` entry — fakeable text fails. (c) `runtime-verification-reviewer.sh` Stop-hook check verifies the verification command corresponds to the modified files (curl URL hits a modified route, sql queries a modified table, test imports a modified source).
- **Prevention.** The `task-verifier` agent is the only entity authorized to flip checkboxes; it follows the evidence-first protocol enforced by `plan-edit-validator.sh`. The `planning.md` "Task Completion — Verifier Mandate" section makes this a hard rule. Plain-text manual verification is forbidden — every runtime task must produce a replayable command in one of five accepted formats (`test`, `playwright`, `curl`, `sql`, `file`).
- **Example.** A plan with 41 tasks was reviewed retrospectively; nine of the checked tasks were found to have no actual implementation matching the description. The builder had marked them complete based on "I built this earlier in the session" without re-verifying. The Gen 4 evidence-first protocol exists specifically to make this class structurally impossible — the validator can't be satisfied by intent alone.

## FM-007 — Behavior change stranded in plan analysis section

- **Symptom.** A plan's analysis sections (typically Section 6 observability, Section 7 failure-mode analysis, Section 8 idempotency, Section 9 capacity) prescribe new behaviors — logging, retry, error messages, encoding rules, recovery paths — that change how the existing code currently works. The Task description and the Files-to-Modify entry for the same file do NOT enumerate any of those changes. A builder reading just the implementation entry points will skip them.
- **Root cause.** Plan author writes Section 1-10 analysis thoroughly but doesn't run a final pass to verify every prescription is reachable from the entry points a builder actually reads (Tasks + Files to Modify/Create). The analysis sections are written with the planner's full mental model; the entry points are written more briefly; the gap between them swallows behavior changes.
- **Detection.** `plan-reviewer.sh` Check 8A (commit `10adac2`, landed via `docs/plans/pre-submission-audit-mechanical-enforcement.md`) blocks Mode: design plans whose `## Pre-Submission Audit` section is missing OR contains neither (a) the canonical full-sentence carve-out (`n/a — single-task plan, no class-sweep needed`) nor (b) at least 5 distinct sweep tokens (S1/S2/S3/S4/S5). This is a STRUCTURE gate — it does not verify whether the documented sweeps were honestly performed (acknowledged residual risk; bypass cost is ~30s of fake-line typing vs ~30min of honest sweeps, a 60× friction ratio). `systems-designer` agent then catches per-section substance issues during adversarial review; with the audit gate landed, the agent can focus on design-quality calls instead of finding sibling-class instances one at a time.
- **Prevention.** `rules/design-mode-planning.md` "Pre-Submission Class-Sweep Audit" rule, S1 (Entry-Point Surfacing). Sweep query: `rg -n 'add|emit|log|retry|fall back|skip|replace|advance|store|cap|new behavior' <plan-file>` cross-checked against the Tasks section AND the Files-to-Modify entry for each affected file. Document the sweep result in the plan's `## Pre-Submission Audit` section before invoking `systems-designer` — Check 8A enforces presence + structure mechanically.
- **Example.** An auth-refactor plan (eight-round originating review effort) had Section 7 row 9 prescribing "Retry once with 2s backoff" for LLM calls — present only in Section 7. The connector-refactor task and its Files-to-Modify entry did not mention any retry behavior. A builder following the entry points would have shipped without the retry. Surfaced as 9+ sibling instances across the same plan's later review rounds.

## FM-008 — Stale existing-code claim in plan

- **Symptom.** A plan asserts a behavior of an existing function, line number, or API — function signature, what a specific line does, what a return value contains — that contradicts the actual current source code. Cited line numbers point at different content than the plan describes.
- **Root cause.** Plan author dictates from memory of an earlier exploration without re-reading the actual file at audit time. Plans authored from memory drift from the codebase, especially when the same file has been touched between the original exploration and the plan revision.
- **Detection.** `systems-designer` agent reads cited line numbers and finds the claim does not match. No mechanical detection — the agent has to actually open the cited file. Self-detection requires the planner to re-read each cited file before submission.
- **Prevention.** `rules/design-mode-planning.md` "Pre-Submission Class-Sweep Audit" S2 (Existing-Code-Claim Verification). Sweep query: `rg -n 'line ?\d+|existing.*function|existing.*at\s+\d+|currently\s+(does|has|returns|reads|writes)' <plan-file>` — for each match, open the cited file at the cited location and confirm. If wrong, fix the claim before submission.
- **Example.** Round 1 of an originating review effort — plan claimed an existing function "stays largely intact" while the end-to-end trace showed a different signature with an added parameter. The existing function fetched its own data internally; the refactor removed that fetch and made the data a required parameter — a real signature change disguised as "stays intact."

## FM-009 — Cross-section contradiction within a single plan

- **Symptom.** Section X of a plan claims "the existing line-N check protects us"; Section Y of the same plan warns "actually that check might not be reliable post-refactor." Two sections contradict on the same code element with no resolution. Builder sees both, picks one view, the other view fails at runtime.
- **Root cause.** Sections written at different times with different mental models without a final reconciliation pass. The plan's structure encourages independent section-by-section thinking; nothing forces the planner to cross-check claims across sections before submission.
- **Detection.** `systems-designer` agent eventually catches the contradiction during section-by-section review. No mechanical detection today.
- **Prevention.** `rules/design-mode-planning.md` "Pre-Submission Class-Sweep Audit" S3 (Cross-Section Consistency). Sweep query: `rg -n 'reliable|unchanged|preserved|stays|protects|catches' <plan-file>` — for each match, check Edge Cases / Failure-mode analysis / Idempotency for any contradictory caveat about the same element. Reconcile by picking the more accurate view, updating the other section, or committing to a third option both sections agree on.
- **Example.** Round 1 of an originating review effort — Section 8 (idempotency) claimed an existing message-count early-exit check was reliable; Edge Cases of the same plan warned that protocol-level reordering introduced by the refactor might make the same check unreliable. Resolution: replace the count comparison with a UID-set subset comparison.

## FM-010 — Deferred design decision with interface impact

- **Symptom.** Capacity or design section of a plan says "either approach A or approach B" / "Mitigation: do X. Or: do Y." for a load-bearing choice and leaves the decision to build-time. The choice usually has interface impact — changes the shape of an API the plan defines, affects user experience, or has cost/complexity tradeoffs. Builder picks one approach alone, with no user visibility and no audit trail.
- **Root cause.** Plan author identifies the constraint but hopes the builder will pick. The choice usually has interface impact (shape of an API the plan defines, control flow, recovery semantics). Without forced surfacing to the user at plan time, the choice gets baked in without informed consent.
- **Detection.** `systems-designer` agent flags any "either/or" / "TODO" / "decide later" / "OR:" pattern in capacity or design sections. The planned `plan-reviewer.sh` extension would FAIL Decisions Log entries for "either/or" patterns unless preceded by a `Surfaced to user:` annotation.
- **Prevention.** `rules/planning.md` "Plan-Time Decisions With Interface Impact — Surface To User" — when the gate fires, plan author MUST use AskUserQuestion to surface the choice with alternatives + tradeoffs + recommendation BEFORE recording the decision in the plan. Picking alone — even with sound rationale — violates the rule because the user lacks the context to override later.
- **Example.** Round 1 of an originating review effort — Section 9 said "process LLM calls in parallel (Promise.all batches of 5-10). Or: cap threads_per_sync at 30 to fit within timeout" without committing. Resolution: a Decision Log entry committing to BOTH (belt-and-suspenders). User feedback on the same date explicitly required surfacing such choices: "Any decisions like this that need to be made, especially when found during planning, should be surfaced to me along with supporting information so that I can make an informed decision before we start building."

## FM-011 — Numeric-precision spec incomplete for ID encoder

- **Symptom.** An ID encoder/decoder spec uses `Number.toString(16)` (or equivalent JS Number arithmetic) on a value that may exceed `Number.MAX_SAFE_INTEGER` (2^53 ≈ 9×10^15). The encoder silently produces wrong output for large IDs; no exception thrown. Dedup against existing entities silently fails for any ID above the safe limit.
- **Root cause.** Plan author does not consider precision when the input is "an integer ID." Real-world 64-bit IDs (Gmail X-GM-THRID, Twitter snowflake, Discord snowflake, UUID v7 timestamps) routinely exceed `Number.MAX_SAFE_INTEGER`. Naive JS Number arithmetic loses precision in the low bits without throwing.
- **Detection.** `systems-designer` agent notices the example ID is large and asks about BigInt handling. No mechanical detection today; depends on the example value being concrete enough to surface the precision issue.
- **Prevention.** `rules/design-mode-planning.md` "Numeric-Precision Spec for ID Encoders" — any plan touching ID encoding must explicitly address `Number.MAX_SAFE_INTEGER`, specify BigInt handling end-to-end (parse, transform, render), and require an acceptance test that locks the precision trap closed via `assertNotEquals(naiveEncoder(input), correctOutput)`. Public API of the encoder should accept `string | bigint` only — never `number` — to prevent reintroduction.
- **Example.** Round 2 of an originating review effort — an ID-encoder Decision Log entry used `idAsInt.toString(16).padStart(16, "0")` without specifying BigInt for a 64-bit unsigned external ID. The illustrative value `19d08d90e074b50a` decodes to ~1.86×10^18 (~207× the safe limit). Without BigInt, the resulting hex would be wrong by the last digit, silently breaking dedup against existing entities.

## FM-012 — Task acceptance criteria incomplete vs section

- **Symptom.** A Task description in a plan claims a function's internals "stay identical" / "preserved" / "unchanged" but another section of the same plan requires changes inside that function. A builder following the task description literally skips the changes; a builder following the section gets the changes but does not know the task expected them.
- **Root cause.** Author writes Tasks first, then Sections, and does not propagate Section requirements back into Task acceptance criteria. The Task entry-point is the canonical builder reference; if it says "stays identical," the builder will not read every section to discover otherwise.
- **Detection.** `systems-designer` agent cross-references "stays identical" claims against per-section change requirements. The planned `plan-reviewer.sh` extension would WARN on "stays identical" / "preserved" / "unchanged" claims in Tasks unless followed by an enumerated whitelist of what is preserved AND what changes.
- **Prevention.** `rules/design-mode-planning.md` "Stays Identical Must Enumerate" — every "stays identical" claim must list specifically what is preserved AND specifically what changes per other sections. Blanket claims forbidden.
- **Example.** Round 2 of an originating review effort — a Task description said "internal logic of `processThread` stays identical" but Section 8 of the same plan required replacing the existing message_count check + adding a new properties field to the upsert payload, both inside `processThread`.

## FM-013 — Capacity claim without arithmetic check

- **Symptom.** Section 9 (load/capacity) of a plan says "X is within tier limits" / "well under Y" / "comfortably below Z" without showing the inline calculation. The reviewer does the multiplication and finds the comparative claim does not hold.
- **Root cause.** Author makes a comparative claim ("under the rate limit") without doing the multiplication that would prove it. The math feels obvious or the planner trusts a quick mental check; in practice, mental arithmetic is unreliable for multi-factor capacity calculations.
- **Detection.** `systems-designer` agent does the multiplication and finds it does not hold. The planned `plan-reviewer.sh` extension would WARN any "within X" / "under Y" / "comfortably below" assertion without inline numerics in the same paragraph.
- **Prevention.** `rules/design-mode-planning.md` "Quantitative Claims Must Be Validated, Not Asserted" — every comparative claim must show the multiplication AND be re-validated by the planner before submission. Preferred: commit a small re-derivation script (Deno one-liner, bash + bc, etc.) to the plan so the math becomes runtime-replayable evidence.
- **Example.** Round 2 of an originating review effort — Section 9 said "30 threads × 2 = 60 LLM calls in ~30s, within tier limits" against a 50 RPM cap. 60 > 50 — the claim was false. Faster execution made it worse, not better, because a denser burst still falls within one rate-limit window.

## FM-014 — Capacity claim contradicts its own math

- **Symptom.** A sentence in a capacity section says "comfortably under X" while a parenthetical hedge admits "(slight over)" or similar contradiction. Self-contradicting prose: the comparative phrase says one thing, the parenthetical admits the opposite. Readers cannot tell whether the design fits the budget or violates it.
- **Root cause.** Author wants to defend a previous design choice when the math has changed underneath it. The honest disclosure ("over the cap") would force a redesign; the hedge tries to avoid the redesign while still being technically accurate.
- **Detection.** `systems-designer` agent reads adjacent numerics and the comparative phrase together and flags the contradiction. Same mechanism as FM-013; this class is just the more honest variant where the author KNOWS the math does not fit.
- **Prevention.** Same as FM-013 — `rules/design-mode-planning.md` "Quantitative Claims Must Be Validated, Not Asserted" + the explicit rule that self-contradicting hedges are forbidden. When math shows the design is over a limit, write the disclosure honestly and pick a mitigation (lower the cap, accept rate-limit retries, upgrade tier).
- **Example.** Round 3 of an originating review effort — Section 9 said "comfortably under tier 1's 50K ITPM (slight over)" with 60K ITPM in scope. Resolution: lower the cap from 20 to 15 to fit both RPM and ITPM honestly.

## FM-015 — Numeric-parameter change not fully swept

- **Symptom.** A numeric design parameter changes during plan revision (e.g., a cap drops from 30 to 20 to 15 across review rounds). The author edits the section that surfaced the issue but sibling references to the old value remain in unrelated sections. Reviewer greps for the literal old number and finds it.
- **Root cause.** Authors edit reactively — fix the section that surfaced the issue, declare the revision complete, do not sweep the rest of the plan for sibling references to the changed parameter.
- **Detection.** `systems-designer` agent greps for the literal old number after revision. The planned `plan-reviewer.sh` extension would detect numeric-token changes between plan revisions and warn about un-swept sibling references.
- **Prevention.** `rules/design-mode-planning.md` "Pre-Submission Class-Sweep Audit" S4 (Numeric-Parameter Sweep) — when changing any numeric parameter, sweep with both literal-number AND prose-context greps across the entire plan. Document the swept parameters in the `## Pre-Submission Audit` section.
- **Example.** Round 4 of an originating review effort — a concurrency-cap parameter changed across review rounds (30→20→15) in the connector-refactor task, the Files-to-Modify entry, the Decisions Log entry, and Section 9 — but Edge Cases ("max 50 threads") and Section 8 ("scan latest 50 threads") were missed. The Section 8 instance was particularly dangerous — a 50-thread cold-start would have burst 100 LLM calls (2× the rate-limit cap).

## FM-016 — Scope vs analysis contradiction

- **Symptom.** Sections 1-10 of a plan prescribe a change to a file/component that Scope OUT explicitly excludes. Plan ships with internal contradictions where the analysis demands a change to a file the plan promises not to touch. Builder either skips the prescription (intent lost) or violates scope (new bugs).
- **Root cause.** Scope is set early in plan authoring; analysis sections evolve later without revisiting Scope. The author working on Section 6 does not re-check whether the changes Section 6 demands are actually in scope per the IN/OUT lists at the top.
- **Detection.** `systems-designer` agent cross-references Scope OUT against every "Add X" / "Modify Y" claim. The planned `plan-reviewer.sh` extension would FAIL plans where a Section's "Add/Modify" verb targets a file in Scope OUT.
- **Prevention.** `rules/design-mode-planning.md` "Pre-Submission Class-Sweep Audit" S5 (Scope-vs-Analysis Check) — every "Add" / "Insert" / "Modify" / "Replace" verb in Sections 1-10 must be checked against the Scope OUT list before submission. Contradictions resolved by either moving the target IN scope (and updating the analysis) or removing the prescription (and replacing with a non-code alternative).
- **Example.** Round 6 of an originating review effort — Section 6 of an auth-refactor plan prescribed adding a banner-log to several adjacent connectors; the same plan's Scope OUT explicitly excluded "any non-target connector behavior changes." Resolution: drop the banner-log code requirement; record the same observability signal as a SCRATCHPAD/Decisions Log entry per the final task (no code change to out-of-scope files).

## FM-017 — Cold-start path violates steady-state envelope

- **Symptom.** A steady-state design parameter (cap, batch size, timeout budget, rate-limit envelope) is constrained in the plan, but a cold-start, reset, fallback, or recovery code path retains the original (larger) value, blowing the constraint on first use after a state reset. The happy path is safe; the recovery path bursts past the budget.
- **Root cause.** Author thinks of "the happy path" as the place to apply the cap and forgets that recovery paths execute the same code under the same rate limits. Cold-start code is sibling-pattern code to steady-state code; constraints applied to one must apply to both.
- **Detection.** `systems-designer` agent reads cold-start prescriptions in Section 8 (idempotency / restart) against the steady-state cap in Section 9. No mechanical detection today.
- **Prevention.** `rules/design-mode-planning.md` "Cold-Start Paths Inherit Steady-State Constraints" — cold-start / reset / fallback paths must inherit steady-state constraints unless explicitly acknowledged in the capacity model as a separate rate-limit event with its own mitigation. If a cold-start path needs a HIGHER cap (legitimate use case: catching up after extended outage), Section 9 must model it as a separate event with rate-shaping.
- **Example.** Round 5 of an originating review effort — steady-state concurrency cap was lowered to 15 in an auth-refactor plan but the cold-start fallback path (when sync state is wiped) retained the existing connector's hardcoded 50. A 50-thread cold-start would burst 100 LLM calls (2× the 50 RPM cap) and ~150K ITPM (3× the 50K ITPM cap).

## FM-018 — Unfrozen-spec edit

- **Symptom.** A builder edits a file declared in an active plan's `## Files to Modify/Create` while that plan's spec is not yet frozen. The build proceeds against a still-mutating plan; mid-build the planner discovers the spec missed a load-bearing path (auth-token refresh, error envelope, retry semantics) and the builder's work has to be partially or wholly redone. Hours of build and verification effort are lost to drift the freeze gate would have prevented.
- **Root cause.** Spec was set `Status: ACTIVE` but never explicitly frozen. The builder treats `ACTIVE` as license to start editing without the planner-side final review that locks the declared-files list, the file-by-file contracts, and the failure-mode coverage. Without a frozen-vs-unfrozen distinction in the plan-header schema, "the plan exists" and "the plan is implementation-ready" collapse into one state.
- **Detection.** Pre-introduction: none — no mechanical detection that an ACTIVE plan was being edited mid-build. After Phase 1d-C-2: `spec-freeze-gate.sh` PreToolUse Edit/Write blocker iterates every `Status: ACTIVE` plan in `docs/plans/*.md`, parses each plan's `## Files to Modify/Create` section into a path list, and BLOCKS edits on any matching file when the claiming plan's header has `frozen: false` or missing.
- **Prevention.** `frozen: true` header field flips after the planner completes a final spec review (interface contracts re-checked, failure modes enumerated, capacity model validated). Only frozen plans permit declared-file edits. Bypass paths: (a) flip `frozen: true` after the final review, (b) move the file out of the plan's declared-files list. Captured in `adapters/claude-code/hooks/spec-freeze-gate.sh` (commit ffd7e2c) + `adapters/claude-code/rules/spec-freeze.md` + Decision 016 (`docs/decisions/016-spec-freeze-gate-c2.md`). Bundled with FM-020 since the schema field is the substrate the gate reads.
- **Example.** A plan declares `## Files to Modify/Create` with multiple source files for an auth-refactor. Builder begins editing immediately on plan-ACTIVE without a freeze step. Mid-build the planner discovers the spec missed the auth-token-refresh path; the partial work has to be redone against the corrected contract. With the freeze gate, the builder's first Edit attempt would have surfaced "plan not frozen — request final spec review before edits proceed," and the missing path would have been caught at planner-review time, not mid-build.

## FM-020 — Missing plan-header field

- **Symptom.** An ACTIVE plan is missing one or more of the 5 required header fields (`tier:`, `rung:`, `architecture:`, `frozen:`, `prd-ref:`). Downstream gates that read these fields cannot compute the right gate set, so calibration / acceptance-loop / DAG-review / spec-freeze / PRD-validity / behavioral-contracts machinery silently degrades — the missing field defaults to "the gate doesn't know if it should fire, so it doesn't," and the plan ships without the enforcement that field would have triggered.
- **Root cause.** A 5-field schema is paper-only without a mechanical check; planners default to whatever feels minimal at plan-creation time. Pre-introduction the plan template advertised the fields but no hook enforced their presence or value validity, so plans drifted toward 1-or-2-field headers depending on which gates the planner happened to remember.
- **Detection.** Pre-introduction: none — `plan-reviewer.sh` checked Mode (`code | design | design-skip`) but did not enforce the 5-field schema. After Phase 1d-C-2: `plan-reviewer.sh` Check 10 blocks plan-file edit on `Status: ACTIVE` plans missing or having invalid values for any of the 5 fields. Required values: `tier ∈ {1,2,3,4,5}`, `rung ∈ {0,1,2,3,4,5}`, `architecture ∈ {coding-harness, dark-factory, auto-research, orchestration, hybrid}`, `frozen ∈ {true, false}`, `prd-ref` non-empty. DEFERRED / COMPLETED / ABANDONED / SUPERSEDED plans don't need fresh schema and are exempted.

Captured in `adapters/claude-code/hooks/plan-reviewer.sh` Check 10 (commit 8ff6a0c) + `adapters/claude-code/templates/plan-template.md` + Decision 017 (`docs/decisions/017-plan-header-schema-locked.md`) + `adapters/claude-code/rules/planning.md` cross-reference.
- **Example.** A plan with `Status: ACTIVE` is missing the `rung:` field. Check 11 (behavioral contracts gate, FM-021) cannot determine whether to fire — the rung might be `3+` (requires `## Behavioral Contracts`) or `0-2` (no requirement). The gate's safe-default is "don't fire" to avoid false-positives, so a high-rung plan ships without a behavioral-contracts section, missing the idempotency / performance-budget / retry-semantics / failure-modes documentation that downstream sessions depend on. With Check 10, the missing `rung:` blocks the plan-file edit before it reaches Check 11, and the planner is forced to populate every field before any rung-gated check can drift silently.

## FM-021 — Missing behavioral contracts at rung 3+

- **Symptom.** A plan with `rung: 3`, `rung: 4`, or `rung: 5` ships without a `## Behavioral Contracts` section, OR with the section present but with placeholder / missing sub-entries for any of (Idempotency / Performance budget / Retry semantics / Failure modes). The builder writes code with implicit, undocumented invariants — a retry loop with arbitrary timeouts, an idempotency assumption that holds in the happy path but not on partial failure, a failure mode the next session does not know to preserve. Downstream sessions that touch the same code path use different parameters, breaking the invariant for callers that depended on the original behavior.
- **Root cause.** Behavioral contracts at higher autonomy rungs are paper-only without a section-presence + substance check; planners trade scope for speed at the moment of greatest pressure (the plan is already long, the section feels like overhead, the planner thinks "I'll capture the contract in the commit message instead"). Without mechanical enforcement, the trade gets made silently and the contract is lost.
- **Detection.** Pre-introduction: the `systems-designer` agent could catch missing contracts on adversarial pass for Mode: design plans, but did not enforce mechanically and did not run on Mode: code plans at higher rungs. After Phase 1d-C-2: `plan-reviewer.sh` Check 11 fires when `rung ∈ {3,4,5}` and requires `## Behavioral Contracts` section + 4 sub-headings (`### Idempotency`, `### Performance budget`, `### Retry semantics`, `### Failure modes`) each with ≥ 30 substantive non-whitespace chars (mirrors Check 6b's substance check, including HTML-comment stripping and placeholder-token stripping).
- **Prevention.** Check 11 in `plan-reviewer.sh` enforces the section + 4 sub-entries with substantive content. Concrete-invariants requirement (HARNESS-GAP-10 sub-gap E partially absorbed by this plan): sub-entries must be substance-checked, not just present — a sub-entry containing only `[populate me]` or `TODO` fails. Captured in `adapters/claude-code/hooks/plan-reviewer.sh` Check 11 (commit 8ff6a0c) + `adapters/claude-code/rules/design-mode-planning.md` cross-reference + `adapters/claude-code/rules/planning.md` cross-reference + the plan template's behavioral-contracts scaffold for rung-3+ plans. Originating decision: 017 (plan-header schema, which establishes the `rung:` field Check 11 reads).
- **Example.** A `rung: 3` (formalized) plan for an auth-refactor ships without specifying retry semantics. The first builder uses 3 retries with 2s exponential backoff; the next session refactoring the same code path uses 5 retries with 500ms linear backoff because the original parameters were never written down. Callers depending on the original semantics (idempotent retry within a 10s budget) break silently — the new retry envelope blows the budget on transient failures. With Check 11, the plan's rung-3 status would have forced the planner to write down "Retry semantics: at most 3 attempts, exponential backoff 2s/4s/8s, total budget ≤ 14s" before the plan reached implementation, and the next session would have a written contract to preserve.

## FM-022 — Unpersisted finding discovered mid-session

- **Symptom.** During a session, a gate fires emitting class-aware feedback (`Class:` + `Sweep query:` + `Required generalization:`), OR an adversarial-review agent (harness-reviewer / systems-designer / code-reviewer / security-reviewer / ux-designer / claim-reviewer / plan-evidence-reviewer) surfaces a class-aware finding in its return summary, OR a builder discovers a sibling regression / unanticipated invariant violation mid-build — but the finding is never persisted to durable storage. The next session has no record that the class was ever observed; promotion / demotion gates that key off the findings ledger cannot compute eligibility because the relevant instances aren't there to read.
- **Root cause.** Pre-Phase-1d-C-3, the `bug-persistence-gate.sh` Stop hook accepted three durable-storage targets (`docs/backlog.md`, `docs/reviews/YYYY-MM-DD-*.md`, `docs/discoveries/YYYY-MM-DD-*.md`). None of those is the right home for a class-aware finding from a gate or adversarial-review agent: backlog is for actionable open work the team plans to address, reviews are timestamped audit-pass output, discoveries are mid-process realizations. Class-aware findings have a different lifecycle (open → in-progress → dispositioned-act / dispositioned-defer / dispositioned-accept → closed) and a different audience (downstream gates that read the ledger to compute eligibility, not just human reviewers), but pre-introduction had no canonical substrate of their own. The result: agents writing findings into return summaries that disappeared when the session ended, gates emitting class-aware blocks with no place to land.
- **Detection.** Pre-introduction: none — the existing `bug-persistence-gate.sh` would have ALLOWed session end as long as some other persistence target was touched, even when the trigger phrases related to class-aware findings the gate didn't model. After Phase 1d-C-3: `bug-persistence-gate.sh` extension accepts `docs/findings.md` as the fourth legitimate persistence target alongside the existing three; `findings-ledger-schema-gate.sh` (PreToolUse Bash on `git commit`) mechanically validates the locked six-field schema (`ID` / `Severity` / `Scope` / `Source` / `Location` / `Status`, plus a non-empty `Description` body) on every entry, blocking malformed commits with a stderr message naming the failing entry + reason.
- **Prevention.** Three-layer enforcement. (1) `findings-ledger-schema-gate.sh` validates schema on every commit that touches `docs/findings.md` — malformed entries cannot land. (2) `bug-persistence-gate.sh` extension (Phase 1d-C-3) makes `docs/findings.md` the canonical home for class-aware findings; trigger phrases at session end must be matched by persistence to one of the four targets (backlog / reviews / discoveries / findings), with findings being the right substrate for class-aware observations from gates and agents. (3) Adversarial-review agents are encouraged to write findings as part of their return — every `Class:` block they emit becomes a candidate `docs/findings.md` entry, surviving session end. Captured in commit `3afa037` (Phase 1d-C-3 Tasks 3+4 — schema gate + bug-persistence-gate extension) and Decision 019 (`docs/decisions/019-findings-ledger-format.md`, which locks the six-field schema and dispositioning lifecycle).
- **Example.** A `harness-reviewer` agent run during plan review surfaces three class-aware findings (Class: `stale-existing-code-claim`, with sweep query and required generalization). Pre-Phase-1d-C-3, the agent's return summary names the three findings, the orchestrator reads them, addresses zero or one in the current build, and moves on; the remaining findings live only in the agent's return text and disappear when the orchestrator's context turns over. Post-Phase-1d-C-3, the orchestrator writes all three findings to `docs/findings.md` — one with status `open` (will fix in this commit), one with status `open` (later session), one with status `dispositioned-defer` (intentionally not acting now, with rationale captured in the Description). All three survive session end. The schema gate validates each entry on commit; the bug-persistence-gate accepts `docs/findings.md` as legitimate persistence at Stop time. A future C13 promotion/demotion gate reading the ledger sees all three instances and computes eligibility correctly.

## FM-023 — Vaporware: spec misunderstood by builder

- **Symptom.** A diff passes typecheck, lint, code-review, and even matches the spec textually — but the builder silently misunderstood an edge case, an assumption about callers, or a side effect. Failure surfaces later as "the feature was built but doesn't behave correctly under <unobserved condition>." Most common at `rung: 2+` (multi-file or behavioral-contract scopes) where the spec implies invariants that aren't enumerated line-by-line. The textual diff agrees with the prose; the runtime behavior does not agree with the prose's intent.
- **Root cause.** The builder's mental model diverged from the spec's intent at a point too small for code-review to catch but large enough to matter at runtime. Pre-Phase-1d-C-4, every adversarial-review agent (`code-reviewer`, `task-verifier`, `plan-evidence-reviewer`, `systems-designer`, `harness-reviewer`, `ux-designer`, `claim-reviewer`) verifies what was *written* against what was *specified*; none verifies that the builder *understood* what was supposed to be written. Comprehension was implicit — assumed to follow from "the diff matches the spec" — but the textual match does not capture builder-side misreadings of edge cases, caller assumptions, or side effects. The gap is structural: a builder can produce a passing diff while holding an internally-consistent but spec-divergent mental model, and no existing reviewer is positioned to detect the divergence.
- **Detection.** Pre-introduction: none — no agent or hook required builders to articulate their interpretation of the spec, and none cross-checked articulation against diff. After Phase 1d-C-4: `comprehension-reviewer` agent invoked by `task-verifier` at `rung: 2+` BEFORE the checkbox flip. The agent reads the builder's `## Comprehension Articulation` block in the evidence entry (four required sub-headings: `Spec meaning` / `Edge cases covered` / `Edge cases NOT covered` / `Assumptions`) and applies a three-stage rubric. Stage 1 (schema): all four headings present, none placeholder-only. Stage 2 (substance): each sub-section ≥ 30 non-whitespace chars of non-vacuous content; vacuous patterns ("does what the spec says", "handles all edge cases", "no assumptions") FAIL. Stage 3 (diff correspondence): each `file:line` citation in `Edge cases covered` maps to actual diff content; no `Assumption` is contradicted by the diff; the `NOT covered` list is honest (cited file:lines really are absent from diff). Returns PASS / FAIL / INCOMPLETE with class-aware feedback per gap. FAIL or INCOMPLETE blocks the checkbox flip until the builder revises the articulation.
- **Prevention.** Comprehension-gate (this mechanism) per Decision 020. Three load-bearing pieces. (1) `adapters/claude-code/templates/comprehension-template.md` — canonical four-heading articulation block builders paste into the evidence entry; substance prompts in each sub-section explain what to write so the builder cannot satisfy the schema with placeholder-only content. (2) `adapters/claude-code/rules/comprehension-gate.md` — Hybrid Pattern + Mechanism rule documenting when the gate fires (rung-2 cutoff: R0/R1 are bug-fix-shaped scopes where comprehension errors are typecheck-visible; R2+ is where the gap matters), the four-section schema, the rung-2 rationale. (3) `adapters/claude-code/agents/comprehension-reviewer.md` + the `task-verifier` extension — auto-invocation at `rung: 2+` with FAIL/INCOMPLETE blocking. Below R2 the gate is a no-op (legitimate carve-out — bug-fix scopes don't carry the same comprehension surface). Captured in Decision 020 (`docs/decisions/020-comprehension-gate-semantics.md`).
- **Example.** A `rung: 3` plan task ships a concurrent-session lock at file:lines `auth.ts:120-140`. The diff handles the happy path correctly (single session acquires lock, releases on success). The builder's articulation under `Edge cases covered` claims "Concurrent sessions handled via lock at line 125" — but file:125 in the actual diff is a comment block, with the real lock acquisition at line 132 and a missing release path on the error branch at line 138-140. Stage 3 (diff correspondence) catches both errors: the cited line 125 doesn't contain the claimed code, AND a sibling edge case ("error during locked section releases lock") is in the diff's behavior but not in the builder's `Edge cases covered` AND is not in `Edge cases NOT covered` either — the articulation is silently wrong about what the diff does. comprehension-reviewer returns FAIL with two gaps; `task-verifier` blocks the checkbox flip until the builder revises both the articulation and (likely) the diff to match.

## FM-024 — Spawn/state gate over-fires on parallel peer-review dispatch

- **Symptom.** A session dispatching multiple read-only review sub-agents in parallel (e.g., two adversarial peer-review Task calls in one response) is blocked by a state/spawn PreToolUse gate that expected a state-file write keyed to the spawn — even though the dispatched work is a stateless review that legitimately produces no state mutation. The operator works around it with an in-band waiver, every time, in every such session.
- **Root cause.** The gate's contract assumes every enumerated spawn is *productive work* that must be preceded by a fresh, shape-valid state-file write naming the spawned branch/work. Read-only peer-review dispatches are spawns under the same matcher but produce no branch and no state — there is nothing truthful to write, so the only path forward is a waiver. The gate cannot distinguish "spawn that should have written state" from "spawn that legitimately has no state to write."
- **Detection.** Behavioral only today: the operator notices the gate fired on a review dispatch and recognizes the waiver is reflexive, not exceptional. Repeated reflexive waivers for the same spawn-shape are the signal (a waiver that is written every time is not an exception — it is an unmodeled case).
- **Prevention.** Partial / aspirational. The principled fix is for the gate to recognize read-only-review spawns (no write-capable tools, no branch, review-agent subagent_type) as a distinct, exempt spawn class rather than forcing a waiver. Until that lands, the documented workaround is the substantive in-band waiver per `git-discipline.md` Rule 3 — but a reflexive waiver should itself be filed as a harness gap, not absorbed silently.
- **Example.** A docs session dispatched two adversarial reviewers in parallel to critique a draft. The spawn/state gate fired on each dispatch expecting a state write naming the spawned branch; the reviewers produced no branch and no state. The session wrote a one-line waiver to proceed. A sibling docs session the same day hit the identical gate on the identical spawn-shape and wrote the identical waiver — confirming a class, not an incident. *Further confirmed 2026-05-19:* the harness-development session that authored THIS entry then immediately hit the same gate when filing three `mcp__ccd_session__spawn_task` bootstrap chips — a harness-docs session with no conversation-tree state file, exactly the unmodeled spawn class. It applied the gate's own named remediation (a substantive `conv-tree-spawn-waiver`), proceeded, and recorded the recurrence here per this entry's Recovery field — the catalog dogfooding itself.
- **Discriminator.** Distinguish from a *genuine* missing-state-write (productive builder spawn that really should have written state): check the spawned `subagent_type` and tool grant. If it is a read-only review agent (no Edit/Write/Bash, no branch created) the gate fire is a false positive of this class; if it is a write-capable builder, the gate is correctly demanding the missing state write (a different, legitimate block).
- **Recovery.** Write the substantive per-`git-discipline.md` Rule 3 waiver to proceed *now* — but in the same session, file the recurrence as a harness gap (the reflexive waiver IS the bug report). Do not let "the waiver works" mask the unmodeled spawn class.

## FM-025 — Stop-hook freshness/attribution signal false-fires from worktree or cross-session-merge context

- **Symptom.** A Stop-hook freshness or attribution signal (SCRATCHPAD staleness, "plans touched this session", handoff-freshness) fires absurdly or repeatedly in a context the signal was not designed for: from inside a git worktree (no per-worktree SCRATCHPAD by convention → "stale by an impossible number of minutes"), or after a cross-session merge that pulled in another session's plan-archive rename (→ the signal attributes the merged session's plan churn to the current session and blocks it ~15+ times for work it has no honest basis to touch).
- **Root cause.** The signal computes against repo-global or current-HEAD state without scoping to the *current session's own* contributions or accounting for worktree topology. A worktree shares `.git` but has no SCRATCHPAD by orchestrator-pattern convention; a cross-session merge brings another plan's terminal-status rename into the current diff. The signal treats "visible in my tree" as "produced by my session."
- **Detection.** Partially mechanized: the SCRATCHPAD-from-worktree freshness shape was fixed in Decision 028 (`session-wrap.sh` `find_repo_root` worktree fall-back). The cross-session-merge transitive false-fire (Signal 3 / dependent Signals 5-6) remains a pending discovery (`docs/discoveries/2026-05-17-session-wrap-signal3-transitive-false-fire.md`) — detection there is still behavioral (operator notices the loop).
- **Prevention.** Decision 028 closed the worktree-SCRATCHPAD instance (S7 self-test locks it). The general class — scope every session-end attribution signal to the current session's own commits, plus a loop-break safety net so an unresolvable Stop signal downgrades instead of looping — is the recommended fix in the 2026-05-17 discovery, not yet shipped.
- **Example.** A worktree-cleanup session merged master; the merge included a stale roadmap plan's archive rename from a different session. `session-wrap.sh` Signal 3 attributed that plan to the current session and re-fired ~15+ times, blocking session end for work the session had no scope to touch. Separately (pre-Decision-028), every worktree session reported SCRATCHPAD "1666666 min stale," training operators to ignore the freshness signal everywhere.
- **Discriminator.** Distinguish from a *real* stale-state signal: check whether the cwd is a worktree (`git rev-parse --git-common-dir` ≠ `--git-dir`) and whether the flagged plan/SCRATCHPAD change appears in *this session's own commits* (`git log <session-start>..HEAD --name-only`). If the flagged artifact is not in the session's own commits, it is this false-fire class.
- **Recovery.** Write the substantive Stop-hook waiver per `git-discipline.md` Rule 3 BEFORE the retry-guard absorbs the loop (do not let it re-fire 15×). If pre-Decision-028 worktree-SCRATCHPAD shape: ignore the absurd freshness number for this session and confirm Decision 028's `find_repo_root` fall-back is installed.

## FM-026 — Config-reading gate is blind to config committed after the worktree's branch point

- **Symptom.** A gate that reads a project config file to decide its behavior (e.g., automation-mode / deploy-policy) reports the config "missing" or applies the conservative default, even though the config exists on master — because the session is in a worktree branched from a commit *before* the config file was committed, so the config is simply not present in the worktree's tree.
- **Root cause.** The Agent-tool worktree is rooted at master HEAD *at worktree-creation time* (or at the branch point), not at live master. A config committed to master after that point is invisible inside the worktree. The gate reads "config absent" and cannot tell "absent because not adopted" from "absent because my worktree predates its commit."
- **Detection.** Behavioral today: the operator notices a gate applying the wrong default in a worktree where master clearly has the config. No mechanical detector distinguishes the two absence causes.
- **Prevention.** Aspirational. Candidate fixes: (a) config-reading gates resolve the config from the parent repo's current master (`git show origin/master:<path>`) when the cwd is a worktree and the file is absent locally; (b) worktree creation rebases onto live master; (c) document that config-dependent gates require the worktree to be re-based past the config commit. None shipped — currently the operator merges/rebases master into the worktree to make the config visible.
- **Example.** A worktree branched before the automation-mode config landed on master; the deploy-class gate read "no automation-mode config" and applied the conservative `review-before-deploy` default, blocking an autonomous merge the project's actual (committed-on-master) config would have allowed.
- **Discriminator.** Distinguish from a genuinely-unconfigured project: `git show origin/master:<config-path>` (or check the parent checkout). If master has the config but the worktree tree does not, this is the worktree-branch-point blind spot, not an unconfigured project.
- **Recovery.** Merge or rebase current master into the worktree (`git merge origin/master`, never force) so the config becomes visible; re-run the blocked operation. Do not hand-fake the config inside the worktree.

## FM-027 — BLOCKED/PAUSING marker treated as session-stop instead of an orchestrator routing signal

- **Symptom.** A dispatched sub-session or builder returns a `BLOCKED:` (or `PAUSING:`) terminal marker; the orchestrator reads the marker and *stops the whole run* — rather than reading the blocker, deciding whether it is actionable, and either unblocking it or routing around it. Work that had a recoverable blocker is abandoned with the plan still ACTIVE.
- **Root cause.** The terminal-marker protocol (`session-end-protocol.md`) is designed so a session declares *why its own turn* ended. An orchestrator consuming a *builder's* return mis-reads the builder's `BLOCKED:` as the orchestrator's own stop condition, instead of as a routing input ("this task is blocked on X — can I supply X, re-dispatch narrowed, or must I surface to the user?"). The marker is a signal *to* the orchestrator, not a command *for* it.
- **Detection.** Behavioral today: the operator notices the run halted with an ACTIVE plan and unbuilt tasks because one builder returned BLOCKED and the orchestrator did not triage it. The orchestrator-pattern rule already says "on BLOCKED, stop dispatching and report the blocker" — but "stop dispatching this dependency chain" is being over-applied to "stop the entire session."
- **Prevention.** Partial. `orchestrator-pattern.md` step 5 already distinguishes "stop dispatching [the blocked chain]" from "abandon the run"; the gap is that the distinction is not reinforced at the marker-consumption point. Aspirational fix: orchestrator-pattern guidance (and a future check) that a builder `BLOCKED:` return requires an explicit orchestrator triage decision (supply-the-dependency / re-dispatch-narrowed / surface-to-user-and-continue-independent-tasks) before the run may end — a builder block is not transitively the orchestrator's block for *independent* remaining tasks.
- **Example.** An orchestrator dispatched a multi-task plan; task 3's builder returned `BLOCKED:` on a missing credential. The orchestrator stopped the entire run, leaving independent tasks 4-7 (no dependency on task 3) unbuilt and the plan ACTIVE — when the correct action was to surface task 3's blocker and continue dispatching the independent remainder.
- **Discriminator.** Distinguish from a *legitimate* full stop: is the blocker on the *critical path of every remaining task* (then stopping the run is correct), or only on *one task with independent siblings* (then continuing the independent siblings is correct)? Check the remaining tasks' `Depends on:` / file-set disjointness against the blocked task.
- **Recovery.** Triage the builder's blocker explicitly: if you can supply the dependency, do so and re-dispatch; if not, surface it to the user AND continue dispatching the tasks that do not depend on it. Only end the run if the blocker is on the critical path of all remaining work — then end with a `BLOCKED:` marker that names the blocker and what unblocks it.

## FM-028 — Investigation proceeds without consulting the failure-mode catalog first

- **Symptom.** A debug / root-cause / "why is X broken" session forms and chases hypotheses for hours before (if ever) discovering that the failure is a known, already-cataloged class. The same class is re-diagnosed from scratch at full cost in a later session.
- **Root cause.** The FM catalog was historically consulted only at *encode* time (`diagnosis.md` "After Every Failure", the skills' "Step 0" when *proposing a mechanism*) and never reflexively at the *start* of an investigation. Nothing in the diagnosis workflow said "grep the catalog before forming your first hypothesis," so the catalog's recognition value was unrealized exactly when it mattered most.
- **Detection.** The investigation-first reflex in `~/.claude/rules/diagnosis.md` ("Check the Failure-Mode Catalog Before Forming a Hypothesis") is the standing instruction; the proposed SessionStart auto-search hook (`docs/proposals/fm-catalog-auto-search-harness-integration.md`) would make detection mechanical by injecting matching FM IDs into investigation-class sessions at spawn. Until the hook ships, detection is the reflex (behavioral, but now explicit doctrine).
- **Prevention.** This convention (`docs/conventions/failure-mode-catalogs.md` + Decision 033) plus the `diagnosis.md` reflex make catalog-first the documented first step of any investigation; the additive `Discriminator`/`Recovery` fields make a matched entry immediately actionable. The proposed auto-search hook is the highest-leverage closure (reflexive use without relying on agent memory) and is the next execution.
- **Example.** An investigation consumed roughly twelve hours and several wrong leads to identify a framework + platform cold-init deadlock. The class was recognizable from its phenotype; a catalog grep at hour zero would plausibly have surfaced it (had it been cataloged) in the first hour. This FM, and this whole convention, exist because of that incident.
- **Discriminator.** This is the meta-class: if a session is more than ~30 minutes into hypothesis-chasing on a defect and has *not* run `grep -in '<symptom keywords>' docs/failure-modes.md`, it is exhibiting FM-028 regardless of the underlying bug.
- **Recovery.** Stop forming hypotheses. Run `grep -in '<keywords from the symptom>' docs/failure-modes.md` and read the `Discriminator`/`Recovery` of any match. If no match, proceed to diagnose — and when the root cause is found, add the new entry so the next session pays one cost, not N.

## FM-029 — Investigation proceeds from inferential evidence without first capturing runtime/error logs from the affected system

- **Symptom.** A debug / root-cause / production-failure investigation forms and pursues a confidence-sounding causal narrative (often across days) built entirely from inferential evidence — probe behavior, code reading, git history, bisect correlation, dependency analysis, schema reads, configuration diffs — without ever pulling runtime / error logs from the affected platform (`vercel logs`, Sentry, Datadog, Supabase logs, webhook delivery logs, job-runner execution logs, etc.). When logs are eventually examined, the actual error message reveals a different root cause that would have been visible in seconds. Engineering resources committed to the inferential narrative (multi-day plans, migrations, refactors) are wasted.
- **Root cause.** Inferential evidence is cheap to gather and feels rigorous in isolation; the platform-specific runtime/log pull is a different motor skill (knowing the right CLI, deployment ID, retention window) that is easy to defer. Once a plausible inferential narrative crystallizes, it anchors subsequent investigation; the deferred log pull becomes "we'll do that later if this doesn't pan out" and never gets done. Pre-2026-05-22 the `diagnosis.md` rule's FIRST step was the FM-catalog grep (FM-028), not the log pull — leaving a gap where a session could proceed all the way to plan-authoring without observing the actual error message the system emitted at the moment of failure.
- **Detection.** The DIAGNOSTIC-FIRST PROTOCOL section in `~/.claude/rules/diagnosis.md` is the standing instruction. The instruction-pair in `~/.claude/rules/claims.md` (every causal claim tagged PROVEN/HYPOTHESIZED; every hypothesis paired with a refutation criterion) catches the downstream symptom — confidence-sounding claims with no log citation are syntactically detectable. The instruction trio in `~/.claude/agents/plan-phase-builder.md`'s "Investigation-work mandate" section forces dispatched investigation sessions to surface log evidence (or an explicit inaccessibility acknowledgment) in their first tool call. Until a mechanical SessionStart-class detector ships, detection is reflexive (the agent self-applying the rules) and behavioral (the user noticing an untagged causal claim and pushing back).
- **Prevention.** The diagnostic-first protocol makes log-pulling the literal FIRST tool call of any production-failure investigation, upstream of the FM-catalog grep (FM-028's reflex). The hypothesis-vs-proof labeling rule (`claims.md`) prohibits naked confident claims without a tag; HYPOTHESIZED is the default-safer tag and requires a refutation criterion. The plan-phase-builder mandate forces dispatched investigation work to either start with a log pull or surface "logs are inaccessible because X" explicitly. The 30-minute trigger from FM-028 applies equally: if a session is past 30 minutes of hypothesis-chasing on a production failure and has NOT pulled runtime logs, it is exhibiting FM-029. Decision 035 (`docs/decisions/035-diagnostic-first-protocol.md`) locks the protocol.
- **Example.** `docs/lessons/2026-05-22-fm-001-misdiagnosis.md` documents 8+ days of misdiagnosis on the FM-001 cold-init incident. The Dispatch orchestrator built a "Lambda 10s INIT cap cold-init deadlock" narrative from bisect correlation, code audit, dependency graph analysis, and bundle composition reads — then authored a multi-day Fly.io migration plan on top of the narrative. Runtime logs were never pulled. A friend running `vercel logs <deployment-id> --no-follow --since 24h --limit 2000 --json` found the actual error (`You cannot use different slug names for the same dynamic path ('id' !== 'orgId')`) in ~30 seconds; the error appeared 1760 times in 2000 log lines on the broken deployment. The actual fix was a 5-10 minute directory rename. The orchestrator received chat-level course-corrections from Misha across multiple sessions; the corrections did not persist because chat is not the harness's durable rule layer. That observation motivated this catalog entry, the diagnostic-first protocol, the claims-labeling rule, and the plan-phase-builder investigation-work mandate.
- **Discriminator.** A causal claim about a production failure that does NOT cite a specific log line (file path + line range, deployment ID, Sentry event URL, query result) is exhibiting FM-029 regardless of how plausible the inferential evidence is. Inferential evidence does NOT close the discriminator — it expands the causal-model space but does not localize the actual failure signature. Concrete check: grep the session transcript for the causal claim and verify the same paragraph contains either a `vercel logs` invocation, a Sentry query, a Datadog query, a Supabase log read, a webhook-provider log read, or an equivalent platform-native log pull. If none, FM-029 applies.
- **Recovery.** Stop authoring plans. Stop bisecting. Stop reading code. Pull the runtime logs per the platform-class guidance in `~/.claude/rules/diagnosis.md`. Re-read the symptom in the log signature (precise error string, status-code distribution, timing pattern). Re-grep the FM catalog with the precise log keywords (FM-028 reflex composes downstream of this rule). Tag every existing claim in the in-flight investigation as PROVEN (with log evidence) or HYPOTHESIZED (with refutation criterion) per `claims.md`. If a structural fix was already proposed on top of the inferential narrative, declare the refutation criterion explicitly per the claims.md rule and look for refuting evidence before continuing. The migration plan in the originating case study would have been correctly abandoned the moment the slug-conflict error appeared in logs — but only because someone pulled the logs.

## FM-030 — Inline PR body bypasses local template validator; server-side CI is the first checkpoint

- **Symptom.** AI sessions constructing PRs via `gh pr create --body "$(cat <<'EOF' ... EOF)"` (or `--body=...`, or `--body "literal"`) produce a PR whose body never passes through the local pre-push validator. The PR is created, GitHub fires the server-side `PR Template Check` workflow, the workflow fails because the body is missing the `## What mechanism would have caught this?` section (or contains placeholder text, or selects no answer form), an email is sent to the repo owner, and the AI session has to issue a second push amending the body. The cycle is 2 pushes + 1 failure email per AI-spawned PR.
- **Root cause.** The pre-push hook (`adapters/claude-code/git-hooks/pre-push-pr-template.sh`) only validates two body sources: (a) a `.pr-description.md` file in the repo root, OR (b) the latest commit message body. AI sessions virtually never write `.pr-description.md` (their natural idiom is the heredoc-into-`--body` form), so the local validator is bypassed by construction. The server-side workflow is correctly enforcing the template contract; the gap is purely local.
- **Detection.** Operator-side: receiving > N "PR Template Check failed" emails per week (the originating signal — Misha logged ~19 across ~12 branches in the week before this catalog entry). Mechanically: any AI-spawned PR whose first push triggers the server-side check failure AND whose second push amends the PR body to add the missing section is exhibiting this class. A simple `gh run list --workflow=pr-template-check.yml --status failure --limit 30` plus correlation against the first-vs-second push of each branch surfaces the rate.
- **Prevention.** The `pr-template-inline-gate.sh` PreToolUse hook on `Bash` (added 2026-05-24) intercepts `gh pr create` and `gh pr edit` calls, extracts the inline body from `--body`, `--body=`, or `--body-file` (and the heredoc-into-`--body` form), pipes it into the same `.github/scripts/validate-pr-template.sh` library used by the CI workflow and the pre-push hook, and blocks the tool call when validation fails. Failure is surfaced inline with the validator's canonical stderr message and remediation guidance. Three-layer defense now: (1) this hook intercepts inline-body PR creation/edit; (2) the pre-push hook covers `.pr-description.md` + commit-message paths; (3) the CI workflow is the final server-side checkpoint.
- **Example.** An AI session opens PR #34 via `gh pr create --title "..." --body "$(cat <<'EOF' ... EOF)"` with a body that includes Summary + Testing performed but omits the `## What mechanism would have caught this?` section. First push: server-side check fails, email sent. Second push: agent amends the body via `gh pr edit 34 --body "..."` with the section added, check passes. The agent has produced two commits where one would have sufficed, and the operator has received an irrelevant failure email. This pattern recurred ~19 times in the week prior to 2026-05-24.
- **Discriminator.** A PR whose creation timestamp and first-passing-`PR Template Check` timestamp differ by more than a few seconds (i.e., the check failed on first push and only passed after an amend) is exhibiting FM-030. Alternatively: any `gh pr create` invocation in the session transcript that uses `--body` (rather than `--body-file`) AND does NOT include the literal string `## What mechanism would have caught this?` in the body is symptomatic before the push even happens.
- **Recovery.** Install the inline-body gate: `pr-template-inline-gate.sh` wired as a PreToolUse `Bash` matcher in `~/.claude/settings.json`. The gate blocks the offending `gh pr create` / `gh pr edit` call locally, surfacing the same canonical diagnostic the server-side workflow would have emitted, so the AI session sees the failure on the same turn it tried to create the PR and can amend the inline body before re-attempting. No second push needed; no failure email generated.

## FM-031 — Decision-soliciting prose surfaced without structured payload escapes audit

- **Symptom.** An agent surfaces a decision / question / action item / autonomous action to the user as freeform prose in its reply (e.g., "Should we use approach A or B?" buried mid-paragraph; "I went ahead and deleted the dead-letter queue" as a parenthetical aside; "Could you tell me your DB connection string?" embedded in a status update). The content does not reach the conversation-tree's "what's waiting on me" view, no `decision-raised` / `question-raised` / `action-item-raised` / `autonomous-action-logged` event is emitted, the item carries no `id` so it cannot be re-surfaced later, and the user must rely on memory to track what's pending across sessions. A session later — sometimes much later — the question / decision evaporates: the user does not remember it was asked, or the agent does not remember it asked it, and the loop never closes.
- **Root cause.** No mechanical floor between the agent's outgoing prose and the user-facing message surface. There is no `mcp__ccd_session__send_user_message` tool exposure that would let a PreToolUse hook intercept the message at send-time, and Claude Code provides no PostMessage hook. The agent's self-applied discipline (Pattern) is the only enforcement absent a Stop-hook reactive gate. Under context pressure, deadline pressure, or in late-session multi-turn fatigue, the discipline drifts: prose-shaped surfacing is the cheapest form to produce and the easiest to skip the fence-grammar overhead on.
- **Detection.** Stop-hook scans the last assistant message in `$TRANSCRIPT_PATH` for the Tier-1/Tier-2/Tier-3 triggers documented in `~/.claude/rules/decision-context.md` (option-presenting verbs + "?" near the message tail for Tier-1 decisions; interrogative-shape sentences for Tier-2 questions; first-person past-tense action verbs for Tier-3 autonomous-action; "could/can you …?" for action-items). When a trigger fires and no matching `::: <category> ... :::` fence is present in the same message, the gate's tiered-scan taxonomy classifies the gap and BLOCKs Stop with the schema-as-the-error message; the agent must redo the message as a structured fence before session can end.
- **Prevention.** `~/.claude/hooks/decision-context-gate.sh` (Stop hook, ADR 047 / DEC-1) is the reactive enforcement layer. The companion substrate is four-hook: `decision-context-reply-emit.sh` (UserPromptSubmit) emits the `answered` event when the user replies with the `reply_with` phrase; `decision-context-pending-surfacer.sh` (SessionStart) re-surfaces items still unresolved at the start of the next session; `decision-context-replay.sh` (SessionStart) drains the fallback queue of fences that failed to land during a facade-down window. Together the four hooks close the round-trip without a PreToolUse intercept on user-facing prose: emitted fences land structured events; un-fenced trigger-shaped prose blocks Stop; pending items persist across sessions; facade-down emissions are not lost. Composes with `conv-tree-orchestrator-emit.md` Layer D (orchestrator discipline binding every Dispatch tool surface to a structured tree event).
- **Example.** `docs/plans/decision-context-gate-2026-05-29.md` — the originating plan that shipped the substrate. The Walking Skeleton at `neural-lace/conversation-tree-ui/state/walking-skeleton-decision-context.sh` demonstrates the round-trip end-to-end: a `::: decision id=WS-1 ... :::` fence emits `decision-raised` + `item-details-set` events, the GUI surfaces the item in "what's waiting on me," a synthetic user reply containing the `reply_with` phrase emits the `answered` event, and the item resolves. The reverse case — prose surfacing without a fence — would have been caught at Stop by the gate's Tiered-Scan and blocked until the agent re-issued the surface as a fence.
- **Discriminator.** A message in the session transcript that contains (a) an option-presenting verb cluster ("either A or B", "should we use X or Y") followed by a "?", OR (b) an interrogative sentence directed at the user ("how should we …?", "what would you like …?", "would you prefer …?"), OR (c) a first-person past-tense action verb describing a state-changing action the agent took ("I deleted …", "I migrated …", "I disabled …"), AND does NOT contain a `::: decision ...` / `::: question ...` / `::: action_item_for_user ...` / `::: autonomous_action ...` fence in the same message, is exhibiting FM-031. The `decision-context-gate.sh --self-test` exercises this discriminator across the Tier-1/2/3 trigger taxonomy.
- **Recovery.** For an already-shipped un-fenced prose decision: manually author a backfill fence in a follow-up assistant message — the gate accepts new fences in any message, the tree picks up the item via the same `decision-raised` event path, and the original prose is left in-place as the historical record. For session-end-blocked cases: read the gate's stderr (which prints the offending message's classified Tier + the canonical fence schema for that category), rewrite the message with the fence, and re-attempt Stop. For chronic re-occurrence within a session: invoke the schema as the canonical reminder rather than relying on memory — copy `~/.claude/rules/decision-context.md` into the working context and use the category-specific fence skeletons as templates.

## FM-032 — Hook resolves target repo from session cwd, not from the operated-on subject

- **Symptom.** A hook that shells out to git (mv, add, ls-files, HEAD reads, staged-file listing) acts on the WRONG repository whenever the session's cwd differs from the repo containing the path or command the hook is processing. Observable phenotypes: a file physically moved out of its owning repo into the session repo's tree and staged there; a commit-scope gate evaluating staged files against an unrelated repo's plans (blocking `cd <other-repo> && git commit` while letting `git -C <other-repo> commit` bypass entirely); HEAD-content reads silently returning empty so pre/post comparisons misfire.
- **Root cause.** Hooks run as child processes with the SESSION's cwd, and bare `git <cmd>` / `git rev-parse --show-toplevel` resolve against that cwd. Any hook whose subject is a path (`tool_input.file_path`) or an embedded command target (`cd X && git ...`, `git -C X ...`) must derive the operating repo from the SUBJECT — but the natural way to write the hook uses bare git calls, which are correct in the common single-repo case and silently mis-target in the cross-repo case (a session rooted in one repo editing files in a sibling project's worktree, or committing into a sibling repo inline). The failure compounds: once repo-root resolution is wrong, downstream tracked-file checks (`git ls-files --error-unmatch`) also fail in the wrong repo, routing execution to plain-`mv` fallbacks that happily cross repo boundaries.
- **Detection.** Code-review check on any hook that shells out to git: every `git` invocation must either be `git -C <root-derived-from-the-subject>` or be cwd-by-design (SessionStart sweeps that intentionally operate on the session repo — and even those must root their per-file git operations in each file's own repo). Mechanically: a cross-repo self-test scenario — fixture file in a second temp repo while the test's cwd stays in the first, asserting the operation lands in the fixture's repo and the cwd repo is untouched — catches regressions of this class per-hook.
- **Prevention.** Derive the operating repo from the operated-on subject. Path-subject hooks: `git -C "$(dirname "$file_path")" rev-parse --show-toplevel`, then `git -C "$root"` for every subsequent operation (`plan-lifecycle.sh`, fixed — see its "Target-repo resolution" header section + self-test scenario 10). Command-subject hooks: parse `git -C` flags and `cd`/`Set-Location` prefixes from the command segment and resolve THAT directory (`scope-enforcement-gate.sh`, fixed — see its "Target-repo resolution" header section + self-test scenarios 26-31). SessionStart sweeps scanning `$PWD` are exempt by design but follow the per-file rooting discipline (`plan-status-archival-sweep.sh`'s `archive_plan()` resolves `git -C "$plans_dir" rev-parse --show-toplevel`, confirmed clean).
- **Example.** A session rooted in the harness repo flipped `Status: ACTIVE → COMPLETED` on a plan file living in a sibling product repo's worktree. The PostToolUse archival hook resolved its repo root from its own cwd (the harness repo); the cross-repo path fell through relative-path resolution unchanged, the tracked-check ran in the wrong repo and reported "untracked", and the plain-`mv` fallback physically moved the sibling repo's plan into the harness repo's `docs/plans/archive/` and staged it there — deleting the plan from the repo that owned it. A second instance of the same class, weeks earlier: the commit-scope gate blocked a `cd <other-repo> && git commit` against the SESSION repo's plan scope (the target repo had no `docs/plans/` and should have been skip-allowed), while `git -C <other-repo> commit` bypassed the gate entirely.
- **Discriminator.** Run both resolutions and compare: `git -C "$(dirname <operated-on-file>)" rev-parse --show-toplevel` vs `git rev-parse --show-toplevel` from the hook's process cwd. If the two toplevels differ AND the hook's git operations used bare `git` (no `-C`), the mis-targeting is this class. If the toplevels match, a wrong-repo effect has a different cause (e.g., FM-025 worktree attribution or a stale index).
- **Recovery.** Locate the misplaced artifact in the wrong repo (`git -C <session-repo> status --porcelain` shows the unexpected `A docs/plans/archive/<slug>.md` or similar), unstage it (`git -C <session-repo> restore --staged <path>`), move the file back into the repo that owns it, and re-stage it there (`git -C <owning-repo> add <path>`). Then check whether the hook that mis-fired has the subject-derived `git -C` fix; if not, apply the fix pattern from this entry's Prevention field before the next cross-repo session hits it.

## FM-033 — Compound Bash command's fix is silently discarded when the trailing commit is pre-screened and blocked

- **Symptom.** A single Bash tool call of the shape `<fix-command> && git add <files> && git commit -m "..."` is submitted; a PreToolUse commit-gate (findings-ledger-schema-gate.sh, scope-enforcement-gate.sh, or another member of the `pre-commit-chain` manifest entry) inspects the CURRENTLY-staged state, finds a violation, and blocks. The agent sees a block message and assumes only the commit failed — but Claude Code blocks the ENTIRE compound command before any of it executes, so `<fix-command>` never ran either. The agent re-runs, sees the same pre-existing state, and the actual fix silently never lands; observed 3 consecutive block/retry cycles before the compounding was diagnosed (NL-FINDING-016).
- **Root cause.** PreToolUse hooks intercept the tool call BEFORE the shell executes any part of it — a `&&`-chained Bash string is one atomic tool invocation from the hook's perspective, not a sequence the hook can partially allow. A gate's block message describing "the commit was blocked" is technically true but incomplete: it implies the commit alone was skipped, when actually nothing in the string ran. An agent editing-and-committing in one call structurally cannot tell, from the block message alone, that its fix was discarded along with the commit.
- **Detection.** Grep any PreToolUse commit-gate's block-message text for language that could be misread as "only the commit was skipped" (e.g., "commit blocked", "commit rejected") without an explicit "the entire command, including any fix commands before it in this chain, did not run" clause. A gate lacking that clause is exhibiting this class regardless of whether a concrete incident has been observed yet.
- **Prevention.** Structural: NEVER chain a fix and its commit in one Bash call when a PreToolUse commit-gate is active on the repo — issue the fix as its own Bash call, confirm it landed (re-read the file or re-run the check), THEN issue `git add && git commit` as a separate call (NL-FINDING-016 disposition; also independently required by the harness's fix≠commit discipline, NL-FINDING-016 cross-referenced in the constitution's git doctrine). Every commit-gate's block message now states explicitly that the WHOLE command did not execute, not just the commit segment, so a compound-command mistake is caught at the first block rather than after 3 silent retries.
- **Example.** A session running `sed -i 's/old/new/' hooks/foo.sh && git add hooks/foo.sh && git commit -m "fix: foo"` in one Bash call, blocked by `findings-ledger-schema-gate.sh` on an unrelated staged file; the `sed` never ran, `hooks/foo.sh` was unchanged, and the agent re-attempted the same compound command twice more before separating the edit from the commit.
- **Discriminator.** If the file the "fix" was supposed to touch is unchanged on disk AFTER a blocked compound Bash call that included both an edit and a `git commit`, this class applies — the block prevented the edit, not just the commit. If the file WAS changed but the commit still didn't happen, the failure is an ordinary gate block, not this class.
- **Recovery.** Re-issue the fix alone as its own Bash call; verify with a read/grep that the change landed; only then issue the add+commit as a separate call.

## FM-034 — Structurally-correct scope-update remedy is itself read as full-plan ownership, trapping the session between two gates

- **Symptom.** A session commits work outside its active plan's declared file scope. `scope-enforcement-gate.sh` blocks and names the documented remedy: append a line to the plan's `## In-flight scope updates` section. The session does exactly that — but this Edit touches the plan file, which `work-integrity-gate.sh` check (a) treats as "the session took ownership of this ENTIRE plan." At Stop, check (a) then demands every one of the plan's unchecked tasks be completed or the plan be marked `Status: ABANDONED` — neither of which is correct for a session whose only plan-file edit was the one-line scope update the OTHER gate required. Check (a) has no waiver valve (unlike check (b)), and `session-honesty-gate.sh` refuses a `DONE:` marker while any block is outstanding, so the session cannot legitimately end.
- **Root cause.** Two independently-reasonable gates compose into a trap: scope-enforcement-gate's remedy for an out-of-scope commit is "touch the plan file"; work-integrity-gate's ownership heuristic is "any transcript Edit/Write under `docs/plans/` this session ⇒ full ownership of that plan." Neither gate is wrong in isolation, but their composition punishes the exact structurally-correct response the first gate demands, with no escape valve on the second.
- **Detection.** Any session whose ONLY plan-file diff this session is confined to the `## In-flight scope updates` section, yet is blocked at Stop by work-integrity check (a) demanding completion of unrelated unchecked tasks, is exhibiting this class. Mechanically: diff the plan file's session-start vs session-end content; if the diff is fully contained within the in-flight-scope-updates section and check (a) still fires, this is the trap.
- **Prevention.** `work-integrity-gate.sh` check (a) exempts plan touches whose diff is confined to the `## In-flight scope updates` section from full-plan-ownership classification (ADR 059 D4), OR the per-session waiver valve available to check (b) (`acceptance-waiver-<slug>-*.txt`-style, freshness + substantive-reason rules) is extended to check (a) — shipped on master via the NL-FINDING-020 waiver-valve commit, with ADR 059's D4 honesty scoping preserved (a COMPLETED-but-unchecked plan, or a checked box with no evidence, remains unwaivable even under this valve). The longer-term structural fix (session end-manifest replacing transcript-forensics plan-touch derivation entirely) is task E.12.
- **Example.** A session shipping PR #74 appended one line to `.pr-description.md`'s owning plan's in-flight-scope-updates section (the documented remedy for a scope-enforcement-gate block), then found itself unable to produce a `DONE:` marker: work-integrity check (a) demanded the operator's entire 20-task Waves-E/F program plan be completed or abandoned, because the one scope-update Edit touched that plan file.
- **Discriminator.** Distinguish from a genuine full-plan-ownership case: if the session's ENTIRE plan-file diff is contained within the `## In-flight scope updates` section (grep the diff hunk boundaries against that section's line range), it is this trap, not real ownership. If the diff also touches task checkboxes, decisions log entries, or task descriptions, real ownership applies and the block is legitimate.
- **Recovery.** Use the per-session waiver path (substantive reason naming the scope-update-only nature of the plan touch) if check (a)'s waiver valve is present on your harness version; otherwise, confirm the diff is scope-update-only and treat the block as a known false-positive pending the exemption landing, escalating to the operator rather than fabricating task completion or a false `Status: ABANDONED`.

## FM-035 — Fixed synthetic session IDs in self-test scenarios accumulate real retry-guard state across runs, eventually masking the block the scenario asserts

- **Symptom.** A hook's `--self-test` suite includes scenarios that exercise a blocking code path keyed by session ID. If those scenarios reuse the SAME fixed synthetic session ID across runs (rather than a fresh one per invocation, or a properly sandboxed state directory), each run increments the SAME retry-guard counter on disk. After enough accumulated runs (locally, in CI reruns, or across a flaky suite retried by a developer), the retry-guard's downgrade-to-warn threshold trips — and the scenario that was supposed to assert "this blocks" silently starts asserting "this warns," passing green while testing nothing. The suite reports PASS; the mechanism it was meant to verify is not actually being exercised as a block anymore.
- **Root cause.** `lib/stop-hook-retry-guard.sh`'s 3-retry downgrade-to-warn logic is a cross-invocation, disk-persisted counter by design (it exists to stop an actually-stuck session from infinite-looping on a gate). A self-test that does not isolate its retry-guard state directory per-run (`RETRY_GUARD_STATE_DIR` unset or shared) is indistinguishable, from the guard's perspective, from a real session retrying the same block 3+ times — so the guard does exactly what it is designed to do: downgrade. The self-test author's intent (assert the BLOCK fires) and the guard's actual behavior (stop blocking after N accumulated retries) silently diverge.
- **Detection.** Any hook self-test asserting a blocking scenario without exporting a fresh `RETRY_GUARD_STATE_DIR` (or `HARNESS_SELFTEST`-scoped equivalent) into a per-run tempdir is exhibiting this class, whether or not it has yet been observed to flip. Mechanically: `grep -L 'RETRY_GUARD_STATE_DIR' <hook>.sh` for any hook that both sources `stop-hook-retry-guard.sh` and ships blocking self-test scenarios is a hit.
- **Prevention.** Every self-test suite that exercises a retry-guard-protected block exports `RETRY_GUARD_STATE_DIR` (and `HARNESS_SELFTEST=1`) into a fresh `mktemp -d` for the DURATION of that suite's run, guaranteeing the guard's counter starts at zero every invocation regardless of how many times the suite has run before, in CI or locally (findings 025/028/034 sandboxing discipline, generalized as a standing requirement for every NEW self-test in this program per the builder dispatch protocol).
- **Example.** A suite's "block fires on the 1st retry" scenario used a hardcoded `SESSION_ID=selftest-fixed-001` with no `RETRY_GUARD_STATE_DIR` override; after the suite had been run manually ~4 times during iterative debugging (each incrementing the same on-disk counter), the scenario started reporting the WARN-mode output instead of the BLOCK output, and the assertion (which grepped for the warn-mode string as a fallback-tolerant match) passed anyway — silently testing the wrong code path.
- **Discriminator.** If a previously-reliable blocking self-test scenario starts intermittently reporting warn-mode output instead of block output, and the flip correlates with HOW MANY TIMES the suite has been run recently (not with any code change), this class applies rather than a genuine logic regression.
- **Recovery.** Add `RETRY_GUARD_STATE_DIR=$(mktemp -d)` (and clean up in a trap) at the top of the affected self-test block; re-run the suite in isolation to confirm the block-mode assertion is exercised fresh every time regardless of prior run count.

## FM-037 — Hook-inheriting automation-spawned `claude` process + no reentrancy guard ⇒ unbounded spawn cascade

- **Symptom.** A machine running the harness experiences runaway process growth (dozens of bash/hook/`claude` processes, unbounded memory growth, eventual crash or forced reboot) with no single obviously-runaway loop visible in any one script's logic. Process listings show many DIFFERENT parent PIDs for the same script basename (e.g., many `nl.sh` or hook-shaped processes each parented by a different session/node process), suggesting fan-out across multiple independent triggers rather than one script looping.
- **Root cause.** No mechanism anywhere in the harness tells a hook "you are running inside an automation-spawned or already-reentrant invocation — do minimal work, do not spawn further." Any `claude`/`claude -p` process launched by automation (a resumer/watchdog script, a scheduled task, any future headless launcher) inherits the FULL live hook suite exactly as a human-interactive session would: every SessionStart hook (which can itself do expensive scanning and spawn further subprocesses — cache refreshes, GUI-mirror writers, heartbeat touches) and every Stop hook (which, in a manifest with a combined-dispatcher architecture, can fork several member-gate subprocesses PER Stop) fires again in the child, with no reentrancy signal to no-op any of it. Independently, any generic-oracle CLI that is polled on a timer by a long-running consumer (e.g., a local dashboard/cockpit server) multiplies its own spawn rate by however many instances of that consumer happen to be running concurrently (one per active worktree checkout, one per crashed-and-restarted dev server, etc.) — a second, `claude`-independent amplification path that can look identical in a process listing (many child processes, no obvious single owner).
- **Detection.** `grep -rn 'CLAUDE_HOOK_DEPTH\|reentran\|<harness's chosen reentrancy env var>' <hooks dir>` returning ZERO hits is itself the detection signal for the root gap — it means no hook anywhere can tell it is in a re-entrant/automation-spawned context. For the timer-poll amplification path: any consumer (dashboard, cockpit, watcher) that shells out to a CLI on a `setInterval`/cron-like timer, checked for (a) per-instance in-flight dedup (necessary but not sufficient — it only bounds ONE running instance) and (b) whether MULTIPLE instances of that consumer can legitimately run concurrently on the same machine (multiple worktrees, crashed-and-relaunched dev servers) without any cross-instance coordination.
- **Prevention.** (1) A shared, fail-safe reentrancy-guard library: a function every heavy/spawning hook calls at the top of its real work, returning true when an explicit env-var signal is set (exported into a child's environment by whatever automation spawns it); absent-lib or forgotten-check-site degrades to prior (unprotected) behavior, never to a crash — this is additive safety, not a new failure surface. Keep this signal SINGLE and explicit: a generic "depth counter" branch is theater unless something actually increments the counter in production (§10 anti-theater — a defense nothing reaches is not a defense). (2) Any script that itself spawns further live child processes for automation purposes (not just inherits hooks) needs an INDEPENDENT hard spawn breaker bounding concurrent/windowed spawns — separate from any pre-existing rate limiter that only bounds ITS OWN actions-per-window, since that does not bound the total number of live processes already on the machine from ALL sources. (3) Any such spawning script should default to inert/shadow/dry-run behavior until an explicit, durable, file-presence-based opt-in marker exists — "the script runs live the moment something calls it" is the wrong default for anything that can spawn further automation. (4) A generic, cheap, env-overridable process-count breaker at the entry point of any frequently-invoked CLI that a poller might call in a tight loop, independent of the reentrancy-guard signal.
- **Example.** A watchdog script that resumes dead/throttled sessions via `claude -p --resume <id>` had no ceiling on concurrent live spawns beyond a rolling-hour ACTION-count rate limiter (which bounds the watchdog's own action rate, not the total live-process count on the machine from any source) — a resumed child that hangs or ends mid-work looks "dead-with-work" to the next scan and gets re-resumed, producing branching growth with many different parent PIDs. Independently, a local dashboard server polling an observability CLI on a 30-second timer, with correct within-process in-flight dedup but no cross-instance coordination, produced dozens of CLI-subprocess spawns purely from having many worktree-scoped dashboard instances running at once — a fully sufficient explanation for "many child processes, no claude in sight" in the same incident.
- **Discriminator.** If a process-tree snapshot shows many SAME-basename child processes under MANY DIFFERENT parent PIDs (not one parent forking repeatedly), suspect the timer-poll/multi-instance-consumer path over a single-script infinite loop. If the snapshot shows `claude`/interactive-CLI processes with a plausible resume/nudge command line, suspect the automation-spawn-inherits-hooks path. The two are not mutually exclusive and can compound (a resumed `claude` child, itself lacking a reentrancy signal, re-triggers the SAME Stop-chain fan-out that produced the original session's own process growth).
- **Recovery.** Add the reentrancy-guard library and wire it into every heavy/spawning hook (SessionStart feeds that do real scanning or subprocess spawns; Stop-chain dispatchers that fork member gates); export the guard's signal into every child-process environment at every automation spawn site; add an independent hard spawn breaker (windowed spawn count + live-process count, both env-overridable, both failing OPEN on an undetermined count) to any script that spawns further automation; default that script's live path to inert until an explicit armed-marker file exists; add a generic process-count breaker to any frequently-polled CLI entry point.

## FM-036 — A hook's default fallback window silently masks the real invocation shape it exists to check

- **Symptom.** A hook that should read a specific, narrow input (e.g., `--shipped-since <ref>` naming exactly what changed this session) is never passed that argument by its caller in the REAL, flagless invocation path — only in its own self-test fixtures. Lacking the argument, the hook falls back to a wide default window (e.g., `HEAD~20`), which happens to still produce a passing/plausible result against the fixture's small commit history, but against the live repo's actual history silently pulls in dozens of UNRELATED prior commits/plans. The self-test suite is 100% green throughout, because it never invokes the hook the way its real caller does.
- **Root cause.** The hook's caller (a dispatcher, a wrapper script, an orchestrator) was changed or newly authored without updating the invocation to pass the narrow argument the callee's newer/stricter logic actually expects — a wiring gap between two components that are each individually well-tested. Self-tests exercise the CALLEE directly with the argument supplied by the test author, never through the REAL caller's actual argument-construction code path, so the gap between "what the caller passes in production" and "what the test passes" is invisible to every existing green suite.
- **Detection.** For any hook accepting an optional scope-narrowing flag with a wide default fallback, grep every REAL (non-test, non-fixture) call site for that flag; a call site invoking the hook WITHOUT the flag, when the flag exists specifically to avoid the wide-default fallback, is exhibiting this class regardless of whether the wide fallback has yet caused an observed false positive.
- **Prevention.** Every self-test suite for a hook with a caller-supplied narrowing argument MUST include at least one scenario that invokes the hook via the SAME shape its real (flagless, or real-flag-supplied) production caller uses — not just the hook's own richest, most-parameterized fixture invocation. This is the "masked-live-shape" self-test requirement (program-standing directive from findings 034/035/036): a self-test that only ever calls the hook with test-author-chosen flags cannot catch a caller that forgot to pass one of them.
- **Example.** `stop-verdict-dispatcher.sh` never passed `--shipped-since <ref>` to `end-manifest.sh`, so `end-manifest.sh` fell back to its `HEAD~20` default; in a real session with a deep, multi-plan commit history, this meant a live Stop gate could evaluate — and potentially block on — 20 commits' worth of UNRELATED prior plans instead of just the current session's shipped work, while every self-test (which always passed a narrow fixture ref directly to `end-manifest.sh`) stayed green throughout.
- **Discriminator.** If a hook's self-test suite is 100% green but the hook has never been invoked, in any test, via the EXACT argument list its real production caller uses (verified by reading the caller's own invocation line, not the hook's fixtures), this class applies — the green suite is proving the callee works when parameterized correctly, not that the system as wired actually parameterizes it correctly.
- **Recovery.** Read the real caller's invocation of the hook (the dispatcher/wrapper source, not the self-test); diff its argument list against what the callee's logic expects for the narrow (non-fallback) path; if the caller omits an argument the callee needs to avoid a wide default, fix the caller to pass it, then add the masked-live-shape self-test scenario so the gap cannot silently reopen.
