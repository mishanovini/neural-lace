# ADR 047 — Decision-Context Gate Enforcement Surface (Stop-Hook Reactive Model)

**Date:** 2026-05-30
**Status:** Active
**Stakeholders:** Misha (decision authority); the Dispatch orchestrator (downstream consumer — the agent that self-applies the fence grammar at message-author time); the maintainer of the conversation-tree substrate (downstream — composition with ADR-031 r7/r8, ADR-032 §8 r2.1, and ADR-034 governs how this gate's emitted events compose with the spawn-side mechanical floor).

## Context

The original brief for the decision-context-gate assumed a **Pre-SendUserMessage hook** — a PreToolUse-class hook that would fire BEFORE the agent's message reaches Misha, allowing the gate to BLOCK the send and force a redo before the un-fenced prose was ever seen. That hook surface does not exist. The MCP tool `mcp__ccd_session__send_user_message` is not exposed at all in the harness's tool surface — the upstream precedent for "we do not currently wrap these MCP send tools" is captured in `docs/plans/conv-tree-auto-emit-enforcement-2026-05-23.md` under the Decision "don't wrap send_message / send_user_message yet" (the conversation-tree auto-emit work explicitly deferred wrapping them on the same grounds: the tools are not surfaced as PreToolUse-matchable in the current harness, and Layer A's per-tool emit can only fire from surfaces that ARE matchable).

The verbal-vaporware gap this ADR closes is catalogued verbatim in `~/.claude/rules/vaporware-prevention.md` under "Residual gap (honest)":

> **Verbal vaporware in conversation is not mechanically blocked.** Claude Code has no PostMessage hook. `claim-reviewer` is self-invoked and can be skipped. This is the single unclosed gap from Generation 4. The mitigation is behavioral: every feature claim must cite file:line, and the user retains interrupt authority when they see an uncited claim.

The decision-context surface is the specific sub-case of that broader residual gap where the agent's prose SOLICITS A DECISION from Misha (enumerated options, a terminal-`?` plus a list, or explicit "pick one" / "your call" / "which do you want" phrasing) without emitting the structured fence the conversation-tree substrate requires to surface the decision in the "what's waiting on Misha" view. Decisions surfaced as prose only — without the fenced JSON payload — escape audit: they never appear as `decision-raised` events in the tree (per ADR-032 §8 r2.1's event-type enum) and the operator's tracker shows the branch as making progress when it is in fact stuck on an un-cataloged decision.

The conversation-tree substrate (ADR-031 r7 → r8 / ADR-034 → ADR-032 §8 r2.1) already enforces spawn-side discipline: the conv-tree state file must contain a fresh, attestation-verified, branch-naming snapshot before a Dispatch spawn proceeds; the Stop-gate verifies the same property at session boundary. Layer D of `~/.claude/rules/conv-tree-orchestrator-emit.md` is Pattern-only for the **message side** — the agent self-applies the fence-first discipline when surfacing decisions to Misha, with no mechanical floor backing it. That is precisely the asymmetry this ADR closes: spawn-side has Mechanism+Pattern; message-side gets Mechanism+Pattern here.

## Decision

Enforcement surface for the decision-context-gate is the **Stop-hook reactive model**.

After the agent's message lands (Stop event fires), the new `decision-context-gate.sh` Stop hook:

1. Reads the last assistant message from `$TRANSCRIPT_PATH` (the agent-uneditable JSONL — the Gen-6 narrative-integrity property that makes Stop-hook scans non-bypassable by the agent's own edits).
2. Runs a cheap pre-filter regex over the message body to detect Tier-1 triggers: enumerated options (`(a) ... (b) ...` or `1. ... 2. ...` lists framed as choices), a terminal `?` followed by a bulleted list, or one of the explicit phrases `pick one` / `your call` / `which do you want`.
3. If a Tier-1 trigger fires AND no fence block (the canonical decision-context JSON fence introduced alongside this gate) is present in the message: the gate invokes a `node` subprocess to validate against the sole-normative Zod schema, then BLOCKs Stop with the schema as the stderr error message. The agent must redo the message as a properly-fenced surface before session-end can complete.
4. If a Tier-1 trigger fires AND a fence IS present: parse the fence via the same Zod module, validate it, emit `decision-raised` / `question-raised` / `branch-note-add` events via the `state.js` facade (which composes with ADR-032 §8 r2.1's attestation primitive — the gate never bypasses canonicalization), and ALLOW Stop.
5. Tier-2 triggers (weaker signals — bare `?` without a list, hedged language like "I think" / "maybe", references to alternatives without explicit enumeration): no Stop block, but append a follow-up marker for `discovery-surfacer.sh`-style SessionStart pickup so the next session sees the un-fenced decision-shaped prose and can reconcile.
6. Tier-3 triggers (rhetorical "does that make sense?" / "right?" / "sound good?"): whitelisted no-op — these are conversational acknowledgments, not decision solicitations, and treating them as gate-fire input would produce constant false positives.

The pre-filter regex is the cost-control mechanism: the `node`-subprocess Zod validation only runs when a Tier-1 or Tier-2 trigger is detected. Stop hooks fire on every session-end; the regex pre-filter ensures the expensive path runs only when the gate's contract is actually in scope. Pattern-shared with the six sibling Stop hooks cited under Consequences below.

## Alternatives Considered

**(b) Convention-only with post-hoc transcript-mining emit.** The agent self-applies the fence discipline; a separate post-hoc job (e.g., a cron, or a SessionStart sweeper) mines transcripts for missed decision-context emissions and writes them to the tree retroactively. Rejected: no enforcement at the moment the message lands; the verbal-vaporware gap stays open in the most-load-bearing window (Misha reading the message and deciding what to do); relies on principles-compliance-style discipline without a mechanical floor. This is exactly the failure mode `vaporware-prevention.md` flagged as "the single unclosed gap from Generation 4" — adopting it as the answer to itself is circular. Post-hoc mining is also lossy in practice: the transcript JSONL is the agent-uneditable record, but a session that ended on a Stop without the gate firing leaves no signal that retroactive mining was needed; the sweeper has to scan every transcript every session, which is the cost-of-pre-filter pushed to a worse place.

**(c) Wait for `mcp__ccd_session__send_user_message` to be exposed by Anthropic.** Hold off on building any enforcement until Anthropic ships the tool, then wrap it with a PreToolUse hook (the originally-imagined design). Rejected: indefinite delay (Anthropic-controlled timeline; the upstream-bug list in `~/.claude/rules/agent-teams.md` shows how long such timelines run); the verbal-vaporware gap stays open meanwhile; the dependent foundation work (the Zod schema, the fence parser, the state-emit composition) all block on a tool that may never ship in the expected shape. Design-for-portability mitigates the long-term risk: the Zod schema, the fence parser, and the state-emit composition are factored as a hook-event-agnostic module invoked with `(message_text)` as input that returns `{verdict, parsed_payload, error}`. Migrating from Stop-hook to PreToolUse-on-`send_user_message` if/when Anthropic exposes it is a ~10-line `settings.json` matcher swap plus removing the Stop-hook entry — the load-bearing logic is reused unchanged.

**(d) Both Stop-hook AND post-hoc — defense-in-depth.** Wire the Stop hook (as in the chosen design) AND additionally run a post-hoc transcript-mining sweeper as belt-and-suspenders. Rejected: both layers depend on the same single point of failure — the transcript JSONL is the input to both. If a future class of decision-shaped prose escapes the Stop-hook's pre-filter regex, it will escape the post-hoc sweeper's regex equally (they share the trigger logic). The post-hoc layer adds runtime cost (re-mining the entire transcript on every Stop or every SessionStart) without surfacing a different failure class than the Stop hook already does. If a future audit reveals the Stop hook misses a category, that's the moment to add a separate layer keyed to a DIFFERENT signal (e.g., a PostToolUse on a related tool, or a SessionStart reconciler reading state files) — not to duplicate the same signal at a different lifecycle point. YAGNI applies: ship the single-layer design, observe failure modes, add layers when the failure modes name them.

## Consequences

**Enables:**
- Mechanical floor for the failure class **"decision-soliciting prose surfaced without structured payload — escapes audit"** — catalogued as the FM-NNN entry shipped with this plan's PR (Task 11 of the parent plan). The conversation tree's "what's waiting on Misha" view becomes structurally truthful: every decision-shaped prose surface either has a fence (event emitted, tree updated) or BLOCKs Stop (agent redoes before session-end). The "uncatalogued decision in flight" steady-state becomes impossible.
- Layer D of `~/.claude/rules/conv-tree-orchestrator-emit.md` migrates from Pattern-only to Mechanism+Pattern. The Mechanism is the Stop-hook gate; the residual Pattern is the agent's self-applied fence-first discipline at message-author time. The redo-friction incentive (BLOCK + schema-as-error + author-must-redo) is what makes the Pattern self-reinforcing: the cost of forgetting the fence is a visible Stop block and a redo, not a silent miss.
- Composition with the spawn-side substrate is clean: the Stop hook emits events through the same `state.js` facade and the same ADR-032 §8 r2.1 attestation primitive that spawn-time and Stop-time conv-tree gates already use. No parallel write path; no schema drift.

**Costs / accepted:**
- **Message-lands-once-before-redo.** The reactive surface means Misha sees the un-fenced message at least once before the agent's fenced redo lands. This is the cost-vs.-pre-send-hook trade. Acceptable per the redo-friction incentive logic (the cost of being SEEN doing it wrong is itself the corrective pressure) and per the sibling-pattern precedent below (all six shipped Stop-hook BLOCK-with-redo gates accept the same trade and none have produced operator-trust regressions in routine use).
- **`node`-subprocess on every Stop fire when triggers present.** Zod validation runs in a `node` child process. Mitigated by the pre-filter regex: most Stop events have no Tier-1/2 trigger and skip the node invocation entirely. When the trigger fires, the cost is bounded — single-file Zod schema, single-message-body input, sub-second typical wall time.
- **Gate fires on EVERY session-stop regardless of whether a decision was solicited.** This is intrinsic to the Stop-hook surface (the hook has no way to know in advance whether the upcoming Stop will need its attention). The pre-filter regex keeps the no-trigger path cheap (single `grep`-equivalent scan over the message body). Acceptable cost for universal coverage.

**Sibling-pattern precedents** (the Stop-hook reactive model with "scan last assistant message in transcript JSONL; BLOCK-with-redo-required" is well-established in the harness, with six shipped instances and a shared loop-break library):

1. `continuation-enforcer.sh` — Stop hook; scans final message for `DONE:` / `PAUSING:` / `BLOCKED:` marker; BLOCKs Stop on missing or contradictory markers.
2. `narrate-and-wait-gate.sh` — Stop hook; scans final message for permission-seeking trail-off phrases under a keep-going directive; BLOCKs Stop.
3. `goal-coverage-on-stop.sh` — Stop hook; cross-checks goals extracted at UserPromptSubmit against tool-call evidence in the transcript; BLOCKs Stop on uncovered goals.
4. `deferral-counter.sh` — Stop hook; pattern-matches deferral phrases against the JSONL; BLOCKs Stop unless the final message has a `## Deferrals not executed in this session` block.
5. `imperative-evidence-linker.sh` — Stop hook; links strong imperatives in user's recent messages to tool-call evidence; BLOCKs on silently-skipped imperatives.
6. `principles-compliance-gate.sh` — Stop hook; checks final message against principles-compliance contract; BLOCKs on violations with the contract as the error.

All six have shipped, are wired in `settings.json.template`, and have not produced runaway lockout failures in routine use. The Stop-hook retry-guard library at `~/.claude/hooks/lib/stop-hook-retry-guard.sh` is the shared loop-break mechanism: after 3 identical-failure retries with no new commits, the block downgrades to a warn and the unresolved gap is logged to `.claude/state/unresolved-stop-hooks.log`. The decision-context-gate inherits this loop-break: if the agent genuinely cannot satisfy the gate (e.g., the schema as designed is malformed for the case at hand), the gate degrades after 3 retries rather than locking the session.
