The plan-and-build phase is a separate phase requiring Misha's explicit greenlight after he has seen this adopted record.

## Adopted decision (r7 — Option 2, file-mediated state contract)

**Architecture (ACCEPTED by Misha 2026-05-16):** a **passive, read-mostly tracking GUI** that observes a **file-mediated state contract**. The Dispatch orchestrator (a harness-bound local Claude Code session) writes the conversation-tree / backlog / decision-action state to a well-known local file as it works; the GUI reads that file to surface the tree, "what's waiting on Misha," and which threads are still open. Dispatch (the Claude desktop app) remains Misha's primary chat surface; the GUI never becomes the chat. Options 1/1b (chat replacement) and Option 3 (active control) are rejected-with-rationale (over-scoped — see Alternatives; retained per Tier-5 option-history discipline).

### Forward-compatibility-conscious refinements (part of the adopted design)

1. **Versioned schema.** The state file carries an explicit `schema_version`. Readers refuse-with-a-clear-message on an unknown major version rather than mis-parsing. Rationale: the underlying Claude Code transcript format is undocumented/unversioned and can change without notice — *our* contract must be the stable layer, decoupled from it.
2. **Append-only event log + periodic snapshot.** State is an append-only event log (branch-opened, decision-raised, answered, concluded, …) plus a periodically-written snapshot for fast reads. The log is the source of truth; the snapshot is a derived cache. Rationale: append-only is crash-safe and auditable (mirrors the harness's own audit-log pattern), and lets the GUI reconstruct state even if a snapshot write is torn.
3. **Well-known local path.** A single documented path (resolved per the per-project vs global tree model, FR-18/FR-25) so the GUI and the orchestrator agree without configuration. Exact path is ADR-032's to fix.
4. **Written schema contract doc.** The state schema is a committed contract document (ADR-032 + a schema file), not tribal knowledge — both the orchestrator (writer) and the GUI (reader) build to it. This is the "our contract is the stable layer" guarantee made concrete.

### Cloud blind spot (explicitly acknowledged, accepted)

The GUI reads **local** state. Genuinely-cloud work — Claude Code on the web, `claude --remote`, unattended `/schedule` Routines — writes **no local transcript** (manual `--teleport` only) and, separately, **does not load `~/.claude/` hooks** (cloud/Routines inherit project `.claude/` only, per Decision 011). Consequence: threads that run as true cloud sessions are **invisible to the GUI and unenforced by the local hooks**. This is an accepted v1 limitation, not a defect — Misha's actual workflow is desktop-app + phone-triggered Dispatch (all local, verified on his machine). Mitigation if cloud usage grows: commit the enforcement gate into the project's `.claude/` per Decision 011 Approach A so cloud sessions inherit it; until then, cloud threads are out-of-tracker by construction and the GUI surfaces nothing false about them (it shows only what it can see).

### Enforcement design (mechanical where possible, rule-backed for the rest)

Research (2026-05-16) established Dispatch-orchestrator sessions are harness-bound: `~/.claude/` PreToolUse/Stop hooks **do** fire on them (confirmed structurally and by the very session that produced this ADR). Therefore the file-update is **substantially mechanically enforceable for local Dispatch**, via two new hooks mirroring existing harness precedents (`teammate-spawn-validator.sh`, `bug-persistence-gate.sh`):

- **`conversation-tree-state-gate.sh` — PreToolUse**, matcher `mcp__ccd_session__spawn_task|mcp__ccd_session_mgmt__start_code_task`. Blocks a child-session spawn unless the state file (a) exists and is valid JSON, (b) has mtime newer than the gate's own per-session spawn marker, and (c) contains an entry whose branch/id matches the branch the spawn's `tool_input` describes. **Fails closed** on a missing/stale state file; fails open only on hook-internal error. (One assumption flagged for the build phase: that Claude Code populates `tool_name` for MCP tools the same way as built-ins — verify empirically before relying on it.)
- **`conversation-tree-stop-gate.sh` — Stop.** Scans `$TRANSCRIPT_PATH` (agent-uneditable) for spawn-tool invocations; if spawns occurred this session without corresponding state-file writes, blocks session end with a remediation message + a justification escape-hatch marker (mirrors `bug-persistence-gate.sh` exactly).

**Mechanically enforceable:** file freshness, JSON shape, branch-name presence, and "spawned in transcript ⇒ must have written before Stop." **NOT mechanically enforceable (rule-class, with Misha's interrupt authority as backstop):** (a) *semantic correctness* of the tree — a hook checks freshness/shape, not truth; the strongest mechanical proxy is requiring the spawned branch name to appear as a key, raising the bar from "wrote anything" to "wrote an entry naming this branch"; (b) the cloud path (above); (c) directly binding spawned child sessions — the harness's own `spawn-task-report-back.md` convention exists precisely because the orchestrator has no mechanical callback into a spawned child, so enforcement binds the **orchestrator at spawn-time and Stop**, not the child. The unenforceable parts are codified in a new Pattern-class rule `~/.claude/rules/conversation-tree-state.md` (the orchestrator self-applies "write the *true* tree"; design the schema so the most semantically-meaningful property reduces to a key-presence/shape check — that is the ceiling of the mechanical layer). This is the same Mechanism+Pattern split the rest of the harness uses.

**Answer to Misha's question** ("is Dispatch beholden to the harness?"): **Yes, for local Dispatch — the harness can mechanically force a fresh, well-shaped, branch-naming write before every spawn and before session end.** It cannot mechanically force the *content* to be correct, and it cannot reach genuine cloud sessions. Those gaps are rule-backed + accepted, not hidden.

## Context

The Conversation Tree Management UI (PRD: `docs/prd.md`, passed Phase-2 substance review at 8b1453e; materially expanded during the interactive Stage-A/B intake to also be a **backlog/to-do surface for not-yet-started Claude-Code work** and to require **partial-answer parallelization** and **NFR-11 persistence/continuity** as a hard constraint) is an optional harness module that makes the Misha↔Dispatch conversation *tree* — and the backlog feeding it — durable, visible, and navigable. The load-bearing open question (PRD OQ-8) is the architecture. r1/r2 considered four options and recommended Option 4; **r3 records the stakeholder striking Option 4**, leaving Option 1 / 2 / 3, weighed against a new stakeholder-supplied ideal (below). Evidence base: two read-only research agents, 2026-05-15 (Agent SDK capability profile + desktop/Dispatch external-bridge investigation).

### Cloud vs local — definitions (r5, plain)

Three distinct things, often conflated (the conflation is what r2–r4 got wrong):

- **Claude Code on the web (cloud):** the task runs on Anthropic-managed cloud infrastructure at claude.ai/code. This is the actual "cloud" product.
- **Remote Control:** a session that runs **locally on the user's machine** but is *driven from* a remote device (phone/web is "just a window into that local session"; nothing moves to the cloud).
- **Dispatch (Cowork):** triggered from the Claude mobile/desktop app, but **executes on the user's LOCAL desktop** ("Claude runs on: Your machine (Desktop)"; the desktop must stay awake or work stops). Dispatch is *not* cloud execution.

**Misha's actual pattern is Dispatch = phone/app-triggered, local-desktop execution** — not cloud. (r2–r4 incorrectly called this "cloud Dispatch.")

### Hard constraint that drives the decision (CORRECTED r5)

**Persistence/continuity (PRD FR-24, formerly NFR-11) is the binding hard constraint:** whatever architecture is chosen must make continuing a thread seamless regardless of elapsed time — no user-visible session boundary. An architecture that cannot do this is reopened. This is unchanged and architecture-agnostic.

**What r5 corrects:** r2–r4 added a *second* "hard constraint" — *"Misha works via cloud Dispatch, so an external GUI can't observe/control it, so Options 2/3 are ruled out."* The 2026-05-16 research shows the premise is false: **Dispatch executes locally, not in the cloud.** The earlier research's own finding was that an external GUI **CAN** observe a *local* session (tail the session JSONL transcript) and **CAN** inject via the documented **Channels** mechanism (localhost MCP broker). The "cannot observe/control" finding applied to *Claude-Code-on-the-web cloud tasks* — a different product Misha does not use as his Dispatch path. **Therefore Options 2 and 3 are NOT ruled out by a cloud constraint; they are reopened and must be re-derived on the corrected local-execution premise** in the pending `systems-designer` pass. (Caveats that remain real even locally: Channels is research-preview; the transcript format is undocumented/unversioned; injected input is a queued turn, not a true mid-turn interrupt — GitHub #35072 open. These are genuine costs, not disqualifiers.)

**Separately (Q-A, decisive for the variant):** external software cannot *launch* a Dispatch session at all — Dispatch is app-initiated only; the spawn tools live only inside a running Dispatch session (GitHub #53494, #39889). So no option can rely on "start a Dispatch session from the GUI." An Agent-SDK app can, however, spawn its **own** Claude Code child sessions in-process (Q-B, documented) — which is what Option 1 / Option 1b actually do.

### Stakeholder-supplied ideal (r3 hard input) — "built into Dispatch"

Misha's stated ideal, verbatim: *"I'm picturing this in my mind as an abstraction layer on top of dispatch. although I also see dispatch already as an abstraction layer on top of Claude code. ideally this UI would essentially just be built into dispatch. but I'm also open to reconsidering if there might be a better way to do this."*

**Honest constraint: "built into Dispatch" is not buildable by us.** Dispatch is an Anthropic-owned product surface; we cannot modify it, embed into it, or ship a native Dispatch feature. So the literal ideal is unreachable, and the ADR must say so plainly rather than imply otherwise. The decision therefore becomes: **which of Options 1/2/3 most closely *approximates* "the tree management is just part of how I drive Claude Code," and at what cost?** Two evaluation axes follow from his framing:

- **Approximation of "built-in":** does the option make the tree feel like an intrinsic part of the driving surface (one place), or a separate bolted-on tool (two places)?
- **What it loses vs. desktop Dispatch's existing features**, and **what it gains in built-in tree/backlog management** — weighed explicitly per option below.

NFR-11 (persistence/continuity — no user-visible session boundary, seamless continuation) is the sharpest test of "approximation of built-in": an option that cannot make session boundaries invisible cannot feel built-in, by construction.

### Agent SDK research (the Option-1 substrate)

The Claude Agent SDK (Python + TS) provides every orchestration primitive first-class: the autonomous tool-use loop, sub-agents, ~18 lifecycle hooks, MCP, session resume/fork, streaming-input with `interrupt()`, and a `SessionStore` adapter that is a clean near-real-time event tap for an external observer. Official prior art exists (`anthropics/claude-agent-sdk-demos`: a ~27 KB React/WS chat UI; a Python multi-agent orchestrator). **But:** building on the SDK means re-implementing the product shell (chat UI, permission UX, durable persistence) and — critically — **Dispatch is not in the SDK at all** (the phone↔desktop remote-orchestration layer is a desktop/Cowork product feature). Commercial: from 2026-06-15, SDK usage draws a **separate metered Agent SDK credit** and **claude.ai-login auth is not permitted for third-party agents — API-key only** (`https://code.claude.com/docs/en/agent-sdk/overview`). Building on the SDK therefore changes Misha's billing and auth model and forfeits the Dispatch UX he relies on. There is **no out-of-process mid-run control channel**: to steer an SDK agent the GUI must *be* the `query()` host process.

## Alternatives considered

> **Option 4 — Tree-as-durable-state + fire-and-forget Dispatch — STRUCK by stakeholder directive at r3** (*"let's remove (ADR-031): Option 4"*). It is no longer in the option space. The r1/r2 analysis of Option 4, the Phase-3 hardening findings against it, and the Option-4 per-scenario table are retained only as a historical audit record in `## Phase-3 review hardening (HISTORICAL — r2, Option 4 struck)` below; none of it is load-bearing for the r3 decision.

### Option 1 — Custom orchestrator on the Claude Agent SDK ("Dispatch++") — [r6: REJECTED — over-scoped; Misha does not want a chat replacement]
- **What:** Build the chat + tree + backlog as one program hosting the Agent SDK `query()` loop. The GUI *is* the orchestrator; the tree/backlog are authored from inside the loop. There is no external surface to bridge to.
- **Approximation of "built in": highest of the three.** One place — tree, backlog, conversation, continuity all in the program Misha drives. This is the only option that is not structurally "a second tool beside Dispatch."
- **NFR-11: satisfiable.** Because it owns the surface, it can make session boundaries invisible (no "resume," no perceptible startup, seamless continuation) — the only option that can.
- **Loses vs desktop Dispatch:** phone/remote driving, the Cowork surface, push notifications, scheduled tasks/Routines, IDE integration — anything Dispatch ships that Option 1 does not re-build. Plus skills/CLAUDE.md/MCP carry over (SDK loads `.claude/`), but the *product* shell does not.
- **Gains in built-in tree/backlog:** total — the tree model and the agent loop are the same program; partial-answer parallelization, backlog→session activation, and persistence are native, not bridged.
- **Cost (not minimized):** largest build (product shell + durable persistence + permission UX); a standing maintenance liability; **billing/auth change** (metered Agent SDK credit 2026-06-15, API-key auth not subscription). r3's stance: these costs are the stakeholder's to accept, and his stated ideal implies he may; r1/r2's "disqualifying for v1" judgment is explicitly revised, not silently dropped.

### Option 1b — GUI owns tree/continuity AND spawns its own Agent-SDK Claude Code child sessions — [r6: REJECTED — over-scoped; same chat-replacement objection as Option 1]
- **What:** the GUI is the tree/backlog/continuity surface AND the orchestrator that **spawns its own Claude Code child sessions via the Agent SDK** (`query()` per work-thread, `resume`/`fork` for continuity, native subagents for decomposition — all documented, Q-B). **Dispatch is not in the execution path at all.** This is what the r4 "launch real Dispatch sessions" variant *becomes* once Q-A (external software cannot launch Dispatch) is applied.
- **Approximation of "built in": high** — like Option 1, one place; the distinction from full Option 1 is mostly framing/scope, not mechanism (both host the SDK and own the surface). In practice **Option 1 and Option 1b largely converge**: once Dispatch can't be in the loop, "Dispatch++" *is* "GUI spawns its own SDK sessions." Option 1b is the honest name for it.
- **NFR-11/FR-24: satisfiable** — owns the surface, so seamless continuation is achievable.
- **Loses vs desktop Dispatch:** same as Option 1 — phone/app-triggering, Cowork, scheduled tasks, push; **the user's existing Dispatch workflow is not used for this work's execution.** This is the sharpest tension with Misha's stated "don't replace normal Claude Code/Dispatch behavior" — Option 1b *is* a parallel execution path, not an overlay on Dispatch. (Misha's earlier correction said overlay-framing doesn't *rule out* an own-orchestrator option; the tension is still his to weigh.)
- **Gains:** total tree/backlog/continuity nativeness; multi-project + global tree + partial-answer parallelization are in-process, not bridged. For *cloud-hosted* execution specifically, the **Managed Agents REST API** is the documented external-create path — but it is a **separate Anthropic product, not Dispatch/Cowork**, with its own session model and the same billing/auth caveats.
- **Cost:** same magnitude as Option 1 (full product shell + persistence + permission UX + standing maintenance + billing/auth model change).

### Option 2 — Passive observer reading local session transcripts — [r6: SELECTED (pending systems-designer pass + Misha final ack)]
- **What:** GUI tails session JSONL to observe; relays follow-ups via a Channels MCP server. Two loosely-coupled interfaces.
- **Approximation of "built in": low.** Structurally two places — a tree tool beside Dispatch. The opposite of Misha's "one place" intent.
- **FR-24: hard for Option 2 to fully satisfy** — it doesn't own the chat surface, so making continuation perfectly seamless across a surface it only observes is the structural objection (a real cost; whether it's *disqualifying* is for the systems-designer pass + Misha, given FR-24 is a hard constraint).
- **Research verdict (CORRECTED r5):** **Dispatch executes locally**, and the research shows an external GUI **can** observe a local session (JSONL tail) and inject via **Channels** (localhost MCP broker). So Option 2 is **VIABLE for Misha's actual (local Dispatch) usage** — *not* ruled out as r3/r4 claimed (that was the cloud-conflation error). Real caveats remain: transcript format undocumented/unversioned; Channels is research-preview; injected input is a queued turn not a true interrupt (#35072 open).
- **Approximation of "built in": low** — structurally two places, a tree tool beside the Claude app. **Preserves Misha's normal Dispatch workflow entirely** (its main advantage vs Option 1/1b: it's a true overlay, not a replacement). Lowest build cost.

### Option 3 — Hybrid: GUI as control surface for Dispatch via a bridge — [r6: REJECTED — over-scoped; adds active-command capability the passive purpose does not need]
- **What:** Tree + backlog in the GUI; chat still in the desktop app; GUI sends "focus node X" via the Channels bridge.
- **Approximation of "built in": low-to-medium.** Still two places; the bridge makes it feel slightly more connected than Option 2 but the conversation surface is not owned.
- **FR-24: same structural objection as Option 2** (doesn't own the continuation surface) — a real cost to weigh, not auto-disqualifying; systems-designer pass + Misha decide given FR-24 is hard.
- **Research verdict (CORRECTED r5):** the Channels bridge is a **documented, first-class** mechanism for **local** sessions — and Dispatch *is* local. So Option 3 is **VIABLE for Misha's actual usage**, *not* "ruled out for cloud" (the r3/r4 cloud-conflation error again). Same real caveats: research-preview; queued-turn not interrupt (#35072 open).
- **Approximation of "built in": low-to-medium** — two places, but the Channels bridge makes the tree able to *steer* the chat (more connected than Option 2's relay). **Preserves Misha's normal Dispatch workflow** (overlay, not replacement). Distinct from Option 2 in *direction*: Option 2 passively observes + relays text; Option 3 actively commands the running desktop session via Channels.
