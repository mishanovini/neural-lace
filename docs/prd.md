## Problem

<!-- Stage A — Problem framing. -->

**Root problem (Misha's framing, Stage A — verbatim basis).** Conversations between Misha and Dispatch are **inherently multi-threaded and branch off in multiple directions, yet are managed as if they are a single thread.** The current workflow does not make it easy to keep track of what was previously being discussed, what still needs to be discussed further, what decisions still need to be made, and what questions have not been answered. A specific, concrete friction compounds this: **Claude (Dispatch) often asks several questions at a time, but Misha's response to any one of them requires conversation — not a bulk answer — so a multi-question turn is unmanageable in a single linear thread.** The decision to go deep on one question necessarily defers the others, and there is no surface that holds the deferred ones as live, trackable branches.

In Misha's words: *"conversations are multi-threaded and branch off in multiple directions, yet are managed as if they're a single thread … Claude often asks me several questions at a time and my response requires conversation. It's not easy to manage a conversation like this around multiple questions simultaneously."*

**Second root dimension (Misha's framing, Stage B — verbatim basis): the unmanaged, capacity-limited mental backlog.** The problem is not only managing conversations *already in flight* — it is also the **set of things Misha wants to work on with Claude Code but has not yet started**. Today that backlog lives only in his head, and it is capacity-limited: it is challenging to put enough thought into each prospective item to start parallel sessions simultaneously, so items queue mentally instead of being captured, contextualized, and parallelized. The solution should also be a to-do/backlog surface where Misha enters and tracks these items, adds context to them, and moves them into sessions — with each item's goals and progress represented in the same tree structure as it advances. Enabling **more parallelization** is an explicit primary goal, not a side effect.

In Misha's words: *"I keep a task list in my head of all the things I want to be working on with Claude Code because it's challenging to put enough thought into each one to start parallel sessions simultaneously. I imagine this solution as also being a to-do list where I can enter and track the items I need to work on with Claude Code. This solution should help me manage those items and move them into sessions, and manage the conversations and decisions of each of them in this tree structure as they go. This tree structure seems to be the most logical way to represent the goals and progress being made toward each."*

So the problem is structural, not cosmetic, on **two dimensions**: (1) the medium (a single linear chat) cannot represent the actual shape of in-flight work (a branching tree of open threads and pending Q&A); and (2) there is no durable surface for the not-yet-started backlog, so prospective work stays trapped in a capacity-limited mental list instead of being captured and parallelized. Everything below is a **consequence** of that two-part root mismatch.

**Consequences of single-thread management of a multi-threaded reality.** The branching work-tree today exists in only two volatile places — Misha's head and the agent's ephemeral context — neither of which survives a session boundary. That produces:

- **Lost threads.** When a session closes, open branches that weren't explicitly resumed evaporate. Misha has no durable surface that says "these branches are still open and waiting on you."
- **Stale decisions.** A decision Misha was asked to make several sessions ago is still pending, but nothing surfaces it; it silently rots until the work it gated is irrelevant.
- **Half-built efforts pile up.** A branch gets most of the way done, the session ends, and the remainder is invisible until someone re-derives it by reading scrollback — the exact failure mode the harness's SCRATCHPAD/plan/backlog discipline fights at the code level, with no equivalent at the *conversation-tree* level.
- **No "waiting on me" surface.** Misha cannot answer "what decisions are blocked on my input?" or "what questions are still unanswered?" without manually reconstructing the tree from memory and chat history. The information exists but is not *surfaced* and not *linked back to the structure that produced it*.
- **Multi-question turns are unmanageable.** A single Dispatch turn raising several questions forces Misha to either answer shallowly in bulk (which his answers don't support — they require conversation) or answer one well and silently drop the rest. Today there is no place for the dropped ones to live as tracked, resumable branches.
- **The prospective backlog is trapped in his head and capacity-limited.** Items Misha wants to work on with Claude Code queue mentally because contextualizing each well enough to start a session is effortful; the result is fewer parallel sessions than he could run, and prospective work that is never captured at all.

Misha explicitly does not do manual git/code work — the system and Dispatch drive execution. So the problem is not "give Misha tools to manage code"; it is "give Misha a durable, visible, navigable model of (a) the *conversation tree* of in-flight work — its open threads, pending decisions, and unanswered questions — and (b) the *backlog* of not-yet-started items he wants to work on with Claude Code, with context attachable and items movable into sessions — both represented in the same tree structure, with the things he is the bottleneck on surfaced and linked back to where they came from, in service of more parallelization."

**Invisible-knowledge surfaces (N-R-B prompt — user-confirmed at Stage A).** Two surfaces, confirmed by Misha. *(A third was proposed by the AI — "implicit Dispatch check-offs are untrusted, and that distrust is the feature" — and Misha **explicitly rejected it at Stage C**: tracking open items is the need, not distrust of check-offs. It is removed; the agree/disagree mechanism survives only as a low-emphasis safety net, not a primary motivation — see FR-9.)*

1. **Node granularity is a tacit heuristic, not a rule.** "A node is roughly the same criteria I'd use to spawn a separate Claude Code session" — a coherent piece of work, not a per-message unit. *(Refined at Stage B: the branching trigger is a conversation going off in a new direction — see FR-2. The "cost of a separate Claude Code session" heuristic stands.)* The model must encode this branching trigger, not invent its own granularity.
2. **The stale-and-forgotten friction is real but the wanted remedy is far narrower than a naive design would build.** The actual need is minimal: notify a parent branch *only* when one of its children concludes, plus let Misha defer his own action items with an optional condition. An aggressive nag/alert engine is a known anti-pattern Misha pre-empted.

## Scenarios

<!-- Stage B — Scenarios and stories. Primary + edge + explicit not-scenarios. -->

### Scenario 1 — A request branches and the branches persist (primary)

Misha asks Dispatch to investigate a harness gap. Dispatch works, then surfaces three decisions. Misha answers two and the conversation goes deep on the third. The tree now shows: a parent node (the original request), an open child branch for the deep-dive, and the two answered decisions auto-checked on the parent. **Whether Misha continues immediately or walks away and comes back a day later, nothing changes from his perspective** — the open deep-dive branch is right there, exactly as it was, with no scrollback to read and no sense that "a session ended." Elapsed time is invisible; everything persists (FR-24).

### Scenario 2 — "What's waiting on me?" (primary)

Misha opens the decision-list side surface. It shows every unanswered decision across the entire tree, each linked back to the node that raised it. He clicks one; the GUI focuses the conversation at that node with that branch's context reloaded. He answers. The decision auto-checks and disappears from the list.

### Scenario 3 — Click a node to resume a dormant branch (primary)

A branch has not been touched in a while (the elapsed time is irrelevant — a minute or a month is the same to Misha). He clicks its node. The branch's context reloads (the parent question, the decisions raised, the work done). He types a follow-up at that node. The previously-focused leaf auto-concludes (with the ability to re-open it). The clicked branch is now the active conversation context. There is no "resuming a session" moment — the branch was simply always there, persisted.

### Scenario 4 — A branch auto-collapses when complete (primary)

A branch has a checklist of 3 decisions and 2 actions. As Dispatch completes actions and Misha answers decisions, items auto-check. When the last item checks, the branch visually collapses (still expandable). Its parent receives a single alert: "child branch *Y* concluded — resume?"

### Scenario 5 — Override a check-off, either direction (secondary safety net)

*(Reframed at Stage C: this is NOT a primary motivation — Misha clarified the need is tracking open items, not distrusting check-offs. It survives as a low-emphasis safety net so a wrong check-off is recoverable, nothing more.)* (a) A check-off Misha disagrees with: he un-checks it and adds a note; the item reopens with the note attached, **and that note can itself become a conversation thread** (Stage B, Misha-confirmed) — not just a static label. (b) Misha marks something complete and Claude can flag a specific "X marked complete but Y may not be covered" note. Either way the disagreement is visible rather than silently resolved — but this is a recovery affordance, not the product's reason for existing.

### Scenario 6 — Conclude a branch, kick off a session (primary)

A branch's decisions are answered. Misha concludes it. Concluding **does not necessarily start a new Dispatch session** (Stage B, Misha-confirmed): the gathered context can EITHER (i) prompt a brand-new session, OR (ii) be fed to an *already-running* session to let it continue. Moreover, a session may be allowed to **start work with only a subset of its questions answered** while the still-open questions continue to be discussed in the GUI as live branches — partial-answer parallelization. A question raised by a running session appears as a new child node under the branch that spawned it. The explicit primary goal of this scenario is **more parallelization** — enabling concurrent progress, not serializing it behind full answer-sets.

### Scenario 7 — Defer my own action item with a condition (edge)

Misha has an action item "decide on hosting provider." He can't decide yet — it depends on a quote arriving. He defers it with the condition "after 2026-05-20." It disappears from the active action list and reappears (or is re-surfaced) when the condition resolves.

### Scenario 8 — Dispatch writes the state while Misha is editing it (edge)

Dispatch is mid-work writing branch annotations to the JSON state file. Simultaneously, Misha drags a node to re-parent it in the GUI. Both writes target the same state file. The system must not corrupt the tree or silently drop either change.

### Scenario 9 — Backlog item → start a session (primary; Stage B, Misha-confirmed)

Misha adds an item to the **backlog** — a thing he wants to work on with Claude Code that has not been started — optionally with a short note. It sits in the backlog list (a surface distinct from the action-items list, per OQ-10 resolved). Later he selects it and starts a Dispatch conversation about it; the item becomes a **root node** of a new tree. Nothing about this differs whether he activates it minutes or weeks after capturing it (FR-24).

### Scenario 10 — Partial-answer parallelization (primary; Stage B, Misha-confirmed + refined)

A running conversation has several open questions. Misha does not have to answer them one-branch-at-a-time: **multiple questions can be discussed within the same conversation thread** — that is still one branch, not many. He gives enough answers for the work to make progress, and the session **runs in parallel with those partial answers** while the remaining questions stay under open discussion in the same thread. A **new branch forms only if the conversation actually goes off in a new direction** (e.g., one question opens a sub-investigation that warrants its own thread) — not merely because a question is still open. The point is more concurrent progress, not a branch per question.

### Scenario 11 — Add context to a backlog item before activating (primary; Stage B, Misha-confirmed)

Before turning a backlog item into a conversation, Misha selects it and attaches context — research notes, files, prior decisions. When he later activates it, that context is carried into the conversation that the item's root node opens. (Misha noted he has not fully thought through the context-attachment mechanics; this scenario fixes the user-observable intent, not the mechanism — mechanics are an Open Question for later stages.)

### Explicitly NOT scenarios

- **Misha hand-edits the raw state.** The persistent shared state (FR-11) is an internal representation; its format is ADR-032's to decide. Every manipulation Misha does is through the GUI (drag-drop, check-off, promote-to-branch, reorder, archive). He never opens the raw representation.
- **Multiple humans collaborating on one tree.** Single-user. No presence, no shared cursors, no per-user permissions.
- **The GUI as a code editor / git client.** Misha does not do manual git/code work; the GUI never shows a diff editor or a commit UI.
- **A *generic* project-management tool.** *(Reconciled at Stage B.)* The system IS a backlog/to-do surface for Claude-Code work items (Scenarios 9/11) — but it is not a Jira/Linear/Asana: no sprints, story points, assignees, arbitrary ticket types, or due-date workflows. It models exactly one thing — Claude-Code work items and the conversation tree they grow into.

## Functional requirements

**Guiding principle (not an FR — symmetric interface, Stage C, Misha-stated).** Anything Misha can modify through the UI, Claude can also modify (via instruction). The interface is symmetric: there is no UI-only or Claude-only mutation. Every FR below that grants Misha a modify capability implicitly grants Claude the same, and vice-versa.

- **FR-1. Tree model.** The system maintains a strict tree (each node has exactly one parent). The root is a top-level work item / request. Acceptance: any node has a single resolvable parent path to the root; no node has two parents. (Non-hierarchical relationships are handled by FR-3 cross-links, not by multi-parenting.)
- **FR-2. Node granularity — branching is about CONVERSATIONS, not questions/decisions (Stage B, Misha-corrected).** A branch forms when **a conversation goes off in a new direction** — not per-question, not per-decision, not per-message. Multiple questions or decisions discussed *within one conversation thread* are **one branch, not many**. In Misha's words: *"the branching concept is branching of conversations, not specifically questions or decisions or anything specific like that."* Heuristic: roughly the cost/coherence of a separate Claude Code session — not per-question granularity. Acceptance: discussing several questions in one thread produces ZERO additional branches; a branch is created only when the conversation demonstrably diverges into a distinct line of work.
- **FR-3. Manual cross-links via tags.** Beyond the strict parent tree, the user (or Claude — symmetric principle) can create non-hierarchical cross-links between nodes via tags, **including cross-links between a per-project tree and the global tree** (see FR-18/FR-25). Acceptance: a tag-based link between two nodes (same tree or across trees) renders a visible non-tree association without changing either node's parent. *[interp — confirm: exact cross-link UX/semantics were a terse tag.]*
- **FR-4. Decision-list surface.** A view listing every unanswered decision across the tree(s), each linked back to its originating node. Acceptance: answering a decision anywhere removes it from this list within one state refresh; selecting an entry navigates to its node.
- **FR-5. Action-list surface — user follow-up items.** A view listing open **action items requiring the user's follow-up** on in-flight work, each linked back to its node — distinct from the decision-list (FR-4) and the backlog (FR-20). Item types are the OQ-4 fixed set {decision, question, action}; the action-list surfaces the follow-up-requiring ones, the decision-list (FR-4) the decisions/questions. A deferred item stays here, tagged (FR-13). Acceptance: completing an action removes it within one state refresh; deferred items remain visible with their tag. *[interp — confirm: "FR-5 user actions" was a terse tag; interpreted as "action items requiring Misha's follow-up," scoped distinct from backlog + decisions.]*
- **FR-6. Click-to-continue.** Selecting any node makes its branch the active conversation and **continues it seamlessly** — no session-lifecycle UI, no perceptible "starting" (FR-24). Context reload format per OQ-3. Acceptance: after selecting a node, the user's next message threads under that node; no session-start affordance or latency the user would attribute to "starting something" is presented for continuing an existing branch.
- **FR-7. Auto-conclude only when all checklist items are checked (Stage C, Misha-corrected).** A branch auto-concludes **only when all its checklist items are checked** — NOT on navigate-away. Navigating away from a branch leaves it open and untouched. (This supersedes the earlier "auto-conclude on navigate-away" entirely.) Acceptance: navigating away changes no branch's concluded-state; checking the final checklist item is the sole auto-conclude trigger; a concluded branch can be re-opened with no data loss.
- **FR-8. Branch checklist.** Each branch carries a checklist of its decisions and action items; entries auto-check when their decision is answered / action is done. When all are checked the branch auto-concludes (FR-7) and collapses visually but remains expandable to full history. Acceptance: the checklist reflects exactly the branch's open decisions+actions; all-checked ⇒ concluded+collapsed; expand restores full history. *[interp — confirm: "FR-8 checklist clarification" was terse; interpreted as the decisions+actions composition + the FR-7 coupling.]*
- **FR-9. Check-off override — secondary safety net (Stage C, reframed).** A check-off either side disagrees with can be un-checked/flagged with a note (and that note can become a conversation thread). This is a **recovery affordance, not a primary mode** — it exists so a wrong check-off is recoverable; it is explicitly NOT a distrust mechanism (invisible-knowledge surface #2 removed at Stage C). Acceptance: an override produces a visible non-silent state; the feature carries low UI emphasis.
- **FR-10. Parent notification on child conclusion.** When a child branch concludes, its parent receives **one** notification (not a stream). The user can also navigate to any parent manually at any time. Acceptance: child conclusion → exactly one parent notification; manual parent navigation always works regardless of notifications.
- **FR-11. Persistent shared state.** A single persistent shared state represents all trees, branches, backlog, and conversation structure. Both the GUI and Claude read and write this shared state; the user manipulates it only through the GUI (drag-drop, check-off, promote-to-branch, reorder, archive, etc.), never the raw representation. Implementation-agnostic: the storage format and concurrency model are ADR-032's to decide (reopened per ADR-031 r3). Acceptance: a GUI mutation and a Claude mutation are both reflected in the shared state and in the GUI; the raw representation is never shown to or hand-edited by the user.
- **FR-12. Lifecycle annotations.** Claude emits explicit lifecycle markers as it works ("opening branch: X", "concluding branch: Y, returning to Z"). These form the authorship/observability trail. Acceptance: every node creation and branch conclusion the system performs has a corresponding annotation; the annotation trail reconstructs the structural changes made.
- **FR-13. Defer is a tag; the item stays visible (Stage C corrected, Stage F finalized — OQ-5).** Deferring an action item **tags it "deferred"** (with an optional scheduled time); it is NOT deleted and **NOT removed from the action list** — it stays on the list, visibly tagged deferred. If a scheduled time is set, at that time the system **highlights the item and notifies Misha — and does nothing else** (it does NOT auto-clear the tag, does NOT auto-act, does NOT move the item). **Misha manually clears the deferred tag** when ready to act. Acceptance: a deferred item remains present on the action list throughout, tagged-deferred; at the scheduled time (if any) exactly one highlight+notification fires and no other state change occurs; the tag clears only by explicit user action; nothing is ever destroyed or auto-resolved by deferral.
- **FR-14. Continue-or-spawn.** Concluding/continuing a branch may EITHER feed accumulated context into ongoing Claude Code work OR start new Claude Code work; additionally, work may proceed with a *subset* of questions answered while the rest stay under discussion (partial-answer parallelization, FR-23). A question raised by running work appears as a new child node under the originating branch. Implementation-agnostic — no orchestration mechanism is baked here (that is ADR-031). Acceptance: both the feed-existing and start-new paths exist; a question from running work creates exactly one correctly-parented child node.
- **FR-15. Many-to-many branch ↔ session binding (Stage C, Misha-corrected).** A branch may be bound to **many** Claude Code sessions, and a Claude Code session may relate to **many** branches (many-to-many, not one-or-more). Acceptance: a branch with zero bound sessions is valid (not-yet-started); a branch may resolve multiple bound sessions; a session may be referenced by multiple branches; the binding model enforces neither a 1:1 nor a 1:N restriction.
- **FR-16. The GUI is the optional module.** The module *is* this GUI plus its persistent shared state; it is installable and removable. With it absent, the harness behaves identically to one that never had it — no required hook in core chains, no broken reference, no orphaned state read. Acceptance: harness behavior with the module absent == harness behavior with it disabled == a harness that never had it. *[interp — confirm: "FR-16 GUI = the optional module" was terse; interpreted as "module boundary == GUI + its state, nothing injected into core chains."]*
- **FR-17. Empty / first-run state.** On first run with no tree yet, the GUI shows an informative empty state explaining what populates it, not a blank screen or error. Acceptance: fresh install, no state ⇒ informative empty state.
- **FR-18. Multi-project support — separate tree per project.** Each project has its own separate tree. (Resolves OQ-6 toward per-project trees; *merged with the earlier-proposed FR-25 multi-project FR — see note at end of section*.) Acceptance: work in project A appears only in project A's tree; switching projects switches trees; no cross-project leakage within the per-project trees (cross-project relationships are FR-3 cross-links / the FR-25 global tree, not multi-parenting).
- **FR-19. — REMOVED (Stage C, Misha directive).** The former spawn-guardrail FR is deleted entirely. Number retired; not reused.
- **FR-20. Backlog surface, tied to project backlogs.** A backlog list of not-yet-started Claude-Code work items, **tied to the project backlogs** (the per-project backlog of work), distinct from the FR-5 action-list. Each item carries a **priority of {high, medium, low}** (single categorical field — OQ-11). Both Misha and Claude can add/modify items (symmetric principle). Covers Scenario 9. Acceptance: an item can be captured to the backlog with minimal friction by either Misha or Claude; the backlog is a surface separate from the action-list; backlog items map to the project's backlog. *[interp — confirm: "backlog tied to project backlogs" was terse; interpreted as "the GUI backlog reflects/links the project's backlog of work," not a wholly separate list.]*
- **FR-21. Add/modify context on a backlog item (both Misha and Claude).** A backlog item can carry attached context (notes, files, prior decisions) that is carried into the conversation when the item is activated. Both Misha and Claude can add or modify this context (symmetric principle). Covers Scenario 11. Acceptance: context attached pre-activation is present in the conversation the item's root node opens; either party can add/modify it.
- **FR-22. Backlog item → tree-root activation.** Selecting a backlog item starts a conversation about it; the item becomes a root node of a new tree. Covers Scenario 9. Acceptance: activation creates a new tree whose root is the item, carrying its FR-21 context.
- **FR-23. Partial-answer parallelization.** Multiple open questions may be discussed in one conversation thread (one branch, not many); work may proceed in parallel with a partial answer-set while remaining questions stay open in the same thread; a new branch forms only on genuine conversational divergence (FR-2). Covers Scenario 10.
- **FR-24. Persistence / continuity as a first-class system property (HARD CONSTRAINT — absorbs former NFR-11, Stage D).** The system SHALL persist all tree, backlog, and conversation state such that user-perceived state is **identical regardless of elapsed time between interactions**. There is no user-visible session boundary, no "next time you sit down to work," no session-start moment, no "resume" affordance, no scrollback to re-read — responding immediately vs. returning days later changes nothing the user sees. In Misha's words: *"there should not be any concept of 'the next time you sit down to work'. … everything should persist."* Implementation MAY use Claude Code session boundaries underneath, but they MUST be invisible at the UX layer — **no user-visible session-startup latency and no per-interaction session authorization when continuing an existing branch**. This is a binding UX constraint on **whatever architecture is chosen** (ADR-031, architecture-agnostic — the struck Option 4's "fresh session per click" framing is gone; the constraint binds Options 1/2/3 equally): if the chosen architecture cannot make continuation seamless, that architecture is reopened. Acceptance (the absorbed NFR-11 testable bound): state observed after an arbitrary elapsed gap is byte-identical to state at pause; a continuation interaction presents no session-lifecycle UI and no latency the user would attribute to "starting something"; the only spawn-confirmation permitted anywhere is for *concluding-into-new* work, never for *continuing* an existing branch.
- **FR-25. Global tree alongside per-project trees.** In addition to FR-18 per-project trees, a **global tree** holds cross-cutting work that spans multiple projects. Cross-links (FR-3) are allowed between the global tree and any per-project tree. Acceptance: cross-cutting work can live in the global tree; a global-tree node can cross-link to a per-project node and vice-versa; per-project trees remain separate (FR-18) — the global tree is an addition, not a merge.
- **FR-26. — NOT ADDED (Stage C, Misha directive).** A structural FR forcing multi-question-turn disaggregation was considered and explicitly rejected by Misha (he does not need it; FR-2 + FR-23 cover the underlying need). Number intentionally skipped to preserve this decision's traceability; not reused.
- **FR-27. Per-branch draft persistence + visual indicator.** An unsent draft tied to a branch is preserved when the user switches to another branch and restored when they switch back. The branch's tree node displays a visual indicator (highlight/badge) showing "draft pending"; the indicator persists until the draft is sent or cleared. Acceptance: switch-away-then-back preserves+restores the exact draft; the node shows a draft-pending indicator throughout; sending or clearing the draft removes the indicator.
- **FR-28. Branch / node archival.** The user (or Claude — symmetric) can archive branches/nodes individually. Archived branches stop appearing in the active tree view but remain accessible via an archive surface. **Archival does NOT close associated Claude Code sessions** — closing a session from outside the desktop app is not currently possible; only the user can close it in the Code tab. Archival affects only the GUI's representation. Acceptance: an archived branch is absent from the active view, present in the archive surface, restorable; no Claude Code session is terminated by archival.
- **FR-29. Backlog + action-list sorting and manual reordering.** Backlog items (and action-list items) can be sorted by any of: **priority** (a single categorical field, exactly {high, medium, low} — OQ-11; no finer granularity), date added, effort, tag. In priority-order view the user can drag-drop to manually reorder (mouse-only acceptable v1 per NFR-6). Both Misha and Claude can modify priority, ordering, and any other field (symmetric principle). Acceptance: each sort key produces a correct ordering; priority sorts on the 3-value categorical; manual drag-drop reordering persists in priority view; either party can modify orderable fields.

FR-19 (spawn guardrail) and FR-26 (multi-question disaggregation) are intentionally retired/skipped per explicit Misha directives, their numbers not reused, so the decisions stay traceable in the numbering itself.

- **NFR-1. State durability.** The persistent shared state (FR-11; format ADR-032's) is the crown-jewel artifact across **all per-project trees, the global tree, and the backlog**. Every write is atomic; the prior version is recoverable; last-N versions retained. A corrupt/unparseable state degrades to "last good version + surface the corruption", never to data loss or a blank view. **Unsent drafts (FR-27) are best-effort durability, NOT crown-jewel** (Misha-confirmed Stage D): draft loss on an abrupt crash is acceptable; tree/backlog loss is not.
- **NFR-2. Co-edit safety.** A concurrent GUI edit (by Misha) and a Claude write to the same state must not corrupt it or silently drop a change (symmetric — both parties write, per the guiding principle). The NFR states only the **safety property**; the conflict-resolution mechanism and conflict-unit are **ADR-032's to decide** (was OQ-1; no longer pinned per ADR-031 r3).
- **NFR-3. Render/interaction performance — calibrated to single-user, single-Dispatch-session-equivalent scale (Misha-confirmed Stage D).** Realistic ceiling: **well under 100 branches total across all per-project trees + the global tree**; backlog/action lists trivially small at that scale. - **NFR-4. State read/write latency.** A state change by either party is reflected to the other within a stated, testable budget. The exact number depends on OQ-2's resolution (snapshot vs. real-time), which is now architecture-coupled (ADR-031 r4) — the NFR is that *whichever* model is chosen carries an explicit testable bound, not the bound's value here.
- **NFR-5. Security / locality + project isolation.** Single-user, local: the GUI binds to localhost only; no remote network surface. The persistent shared state contains no credentials/tokens/secrets — only tree/backlog structure and references. **Project isolation:** no cross-project data leakage between per-project trees; cross-project relationships exist only via explicit FR-3 cross-links / the FR-25 global tree. Acceptance: a LAN port scan finds no listening GUI port; a state scan matches zero credential patterns; project A's tree never surfaces project B's content except through an explicit cross-link.
- **NFR-6. Accessibility — scoped subset (Misha-confirmed Stage D).** Tree and lists must be keyboard-navigable (traverse/focus/collapse) and must not rely on color alone (concluded/deferred/draft-pending each have a non-color signal). **Mouse-only drag-drop (reorder FR-29, re-parent) is acceptable for v1**; keyboard-accessible drag-drop equivalents are an explicit **v2 nice-to-have, not a v1 requirement**. Full WCAG-AA remains **N/A** (single private user, no external surface).
- **NFR-7. Observability.** Claude's lifecycle-annotation trail (FR-12) is the primary observability surface. Additionally, an append-only audit log records **every mutation class** — tree, backlog, archival (FR-28), reorder (FR-29), drafts — by **either party** (Misha or Claude). Acceptance: given only the audit log + annotation trail, current state across all trees + backlog is reconstructable.
- **NFR-8. Harness coupling ceiling.** Per FR-16 the GUI *is* the optional module. It touches the harness only through a documented, minimal integration surface (branch↔session binding, context-load, continue-or-spawn per FR-14, annotation-read). No required hook in core Stop/PreToolUse chains. **Explicit non-coupling:** FR-28 archival never terminates a Claude Code session. Acceptance: integration points enumerable on one page; module absent == module disabled == never-installed; **the hook-firing diff between a module-absent run and a module-disabled run is empty** (this is the measurable property formerly tracked as SM-5, relocated here at Stage E because it is a development-quality guarantee, not a user-outcome metric).
- **NFR-9. Reliability of session integration (many-to-many, FR-15).** If a Claude Code session a branch references no longer exists, the branch degrades **per missing session** and remains usable if any binding still resolves; a node with no resolvable session still renders and is still navigable ("session unavailable — context may be partial"). Archival (FR-28) never causes this (it doesn't close sessions).
- **NFR-10. Internationalization.** Explicitly **N/A** — single English-speaking user, no localization requirement.
- **NFR-11. — ABSORBED into FR-24 (Stage D, Misha-confirmed).** The former Continuity NFR is merged into FR-24, which now carries both the functional assertion and the measurable/testable bound, architecture-agnostically. Number retired; not reused (traceability).
- **NFR-12. Compliance.** Explicitly **N/A** — single-user local tool; no regulatory or internal-policy data-handling obligation; the only data is Misha's own work content, which never leaves localhost (NFR-5). Stated to close the category (no silent skip).

## Out-of-scope

- **Multi-user / collaboration** — single known user; presence, shared cursors, and per-user permissions are 10x the model complexity for zero current value.
- **Real-time collaborative editing** — only one human and one Dispatch write the tree; the co-edit case is narrow (NFR-2), not a general CRDT/OT problem.
- **Mobile-native client** — Misha drives Dispatch from desktop; a mobile tree client is not in the friction path. Revisit only if Dispatch usage moves to mobile primarily.
- **JSON hand-editing surface / schema docs for the user** — the JSON is internal; exposing it as an editable surface contradicts FR-11. The schema is an engineering artifact, not a user-facing one.
- **Replacing the full Claude desktop app feature set** — whether the GUI subsumes the chat interface at all is the architecture decision (ADR-031); even in the most ambitious option, parity with every desktop-app convenience feature is explicitly not a goal of the first release.
- **Generic project-management tooling** — *(reconciled at Stage B — see note).* The system **IS** a to-do/backlog for items Misha wants to work on with Claude Code (Stage B scope expansion: capture, contextualize, move into sessions, track per-item goals/progress in the tree). What remains out of scope is *generic* PM: sprints, story points/estimates, arbitrary ticket types, assignees, due-date workflows, and any field not in service of "a thing to work on with Claude Code and its conversation tree." It is not a Jira/Linear/Asana; it is a Claude-Code-work backlog whose items become conversation-tree roots.
- **Aggressive nag/alert engine** — per invisible-knowledge surface #2, notification is intentionally minimal (parent-on-child-conclude + Misha's own deferrals). A notification/reminder system beyond that is explicitly excluded.
