# Conversation Tree UI — UX Audit (2026-05-23)

**Auditor:** Claude Code (autonomous run on Misha's feedback)
**Scope:** entire GUI surface served at `http://127.0.0.1:7733`
**Method:** static reading of `web/index.html`, `web/app.css`, `web/app.js`, `web/responsive.selftest.js`, plus simulated viewport behavior at 1920 / 1440 / 1280 / 1024 / 800 px, plus the screenshots and verbal feedback Misha provided 2026-05-23.

---

## TL;DR

1. **The tree pane wastes most of its horizontal real estate** — it occupies 57% of a 1920 px viewport (~1095 px) while typical node-row content needs ~280–320 px. ~70% whitespace on the right. The pane width was sized for "tree sits next to two columns of side panels" but only one column is actually load-bearing.
2. **Side panels stack BELOW the tree at 1024–1439 px** (Layout B, the default grid). Misha's narrow-window screenshot is exactly this: tree top, Waiting + Backlog jammed below, ctx-panel and doc-pane overlaying on top. He explicitly does NOT want content below the tree.
3. **Five surfaces compete for the same horizontal slot at once** (Tree, Waiting, Backlog, ctx-panel, doc-pane) — and three of those are persistent docked panes at ≥ 1440 px. Misha wants the detail / doc / dispatch-composer surfaces to be **modal overlays**, not panes that crowd the persistent layout.

The redesign that follows lands on: **narrow fixed-width tree column + single tabbed right side panel + modals for detail / doc / dispatch composer + drawer collapse on narrow windows.**

---

## 1. Layout problems (with measurements)

The current responsive system has three named layouts (`app.css:132-197`):

| Layout | Width | Tree width | Right side | Misha's feedback |
|---|---|---|---|---|
| (A) ≥ 1440 px | wide | **57% (~1095 px @ 1920)** | Actions (top 55%) + Backlog (bottom 45%) docked right | Tree too wide, too much whitespace |
| (B) 1024–1439 px | mid | **100% top half** | Actions | Backlog stacked **below** tree, bottom half | "I prefer to keep all additional panels to the side instead of taking up space below" — VIOLATED |
| (C) < 1024 AND < 1024 | small | full pane, **tabs** to switch | Same pane | Acceptable but only fires when BOTH dimensions < 1024 |
| (B') < 700 px wide | mobile | full row | actions row, backlog row, all vertically stacked | Same vertical-cramming problem |

### Per-viewport behavior under the current build

| Viewport | What happens | Problem |
|---|---|---|
| 1920 × 1080 | Layout A. Tree gets 1095 px. Each tree row uses ~280 px of left-edge content. Remaining ~815 px on every row is empty. | ~74% horizontal whitespace inside the tree pane. |
| 1440 × 900 | Layout A. Tree gets 820 px. Same node-row width. ~62% whitespace. | Still wasteful; ctx-panel docked at ~30 rem (480 px) eats into actions/backlog. |
| 1280 × 800 | Layout B fires. **Tree fills 1280 × 400 (top half). Actions + backlog jammed in 640 × 400 quadrants below.** | This is the canonical "Misha screenshot" failure. Tree has plenty of room but actions/backlog are claustrophobic and require scroll. |
| 1024 × 768 | Layout B again (h < 1024 alone is not enough — needs w < 1024 too). | Same crowding. |
| 800 × 1200 | Layout B' fires (w < 700 doesn't trigger here either — needs w < 700). Actually layout B at 800 px wide stays B because grid-template-columns is 1fr 1fr, so each side gets 400 px. **At 800 px wide the bottom-row halves are unusable.** | Wrong breakpoint — bottom halves go cramped well before 700 px. |
| 800 × 800 | Layout C single-pane tabs (both < 1024). | Usable but very abrupt mode switch; modals overlay full screen. |

### Crowding when overlays open

At 1920 × 1080 with both ctx-panel AND doc-pane open:
- ctx-panel: 30 rem (480 px) docked right
- doc-pane: 32 rem (512 px) docked right (sits left of ctx-panel via `body[data-doc-pane=open][data-ctx-pane=open]`)
- main `#layout` margin-right: **992 px** (480 + 512)
- Remaining layout width: 928 px, of which 57% (~530 px) is tree and 43% (~398 px) is actions/backlog stacked

The actions pane gets 398 × 600 px. Backlog gets 398 × 480 px. Both panes still have headers + sort dropdowns + badges in that width. **They're functional but visibly cramped, and the persistent docked-pane stacking is exactly what's "trying to cram itself in horizontally."**

## 2. Information density per panel

What lives in each panel today (`web/index.html:41-110`):

### Tree pane (`#treePane`)
- Title "Tree" + crumb (selected node breadcrumb)
- View toggle: 👁 Show concluded
- Zoom controls: − / + / ⊙ fit
- Tree canvas (node rows with: twist, title, state badge, count badge, archive badge)
- Empty / first-run / loading states

**Density verdict:** Low. Most rows are 1 line of text + 2 small badges (~12% of available row width at 1095 px column).

### Waiting (`#actionsPane`)
- Title + "+N new" arrival badge + sort dropdown
- List of items, each carrying:
  - Type tag (action/decision/question)
  - Priority badge (P1–P4)
  - Title text
  - Jump-to-tree icon
  - 2–5 action buttons (Done, Respond, Defer, Promote, Assign-priority, etc.)
  - Optional ▾ details disclosure (renders full What/Why/Options/Links)
  - Optional defer popover, priority popover, response inline form

**Density verdict:** High. Each item can grow to ~10 visible elements when expanded. **All controls live inline on the card** — no item-level modal.

### Backlog (`#backlogPane`)
- Title + "+N new" badge + sort dropdown + "+" add button
- Capture form (text / priority / context / save)
- List of backlog items (text, priority chip, activate button)

**Density verdict:** Medium-low. Simpler than actions.

### Context panel (`#ctxPanel`) — currently DOCKED at ≥ 1440 px
- Parent chain
- Diverging sub-branches
- Selected-item rich details (item 39, ~3 paragraphs)
- Open items summary + "show more"
- Cross-links + add cross-link
- **Staged note / Send to Dispatch** (the big one — textarea + 5 buttons)
- Archive / annotate / restore

**Density verdict:** Very high. This is more like a modal/full-page detail view than a side pane.

### Doc pane (`#docModal`) — DOCKED at right at all widths
- Doc title + "Open in editor" + close
- Markdown body (full doc render)

**Density verdict:** Medium-high. The user actively reads doc text in this surface — long-form content should be a modal, not a docked sliver.

### Docs browser panel (`#docsPanel`)
- Filter input + close
- Project → folder → file tree

**Density verdict:** Medium. Already overlays as a side panel — works fine as a modal/drawer.

### Redundancy
- Detail pane shows "Open items (N)" AND clicking an item in actions/backlog already shows the same item's details inline via ▾.
- The ctx-panel's "parent chain" duplicates information the tree already conveys via indentation.
- "Cross-links" is rarely populated for most nodes (informational density very low).

## 3. Interaction model

Click flows today:

| User action | What happens | Side effect |
|---|---|---|
| Click a tree node | `selectNode(id)` → tree row highlights → **ctx-panel opens** | At ≥ 1440 px: layout shrinks margin-right by 30 rem. Backlog/Actions panes shrink. |
| Click a Waiting item title | Item highlights, jumps to tree node, opens ctx-panel | Same shrink. |
| Click ▾ on a Waiting item | Details disclosure expands INLINE | Inline (good). |
| Click a `[doc.md]` link inside details | Doc viewer opens as a docked side pane (margin-right shifts again) | Two stacked side panes now. |
| Click 📁 Docs in header | Docs browser opens as overlay panel (full-height right drawer with scrim) | Three side surfaces possible (docs browser + ctx + doc-pane). |
| Open a defer popover | Inline below the item (good) | Inline. |

**Issues:**
- Clicking ANYTHING in the tree opens the ctx-panel, which then triggers layout shrinkage. There's no way to "just navigate without opening the side pane."
- The ctx-panel has TWO modes: docked (≥ 1440) and modal-with-scrim (< 1440). Misha's narrow-window screenshot shows the docked-vs-modal cutover at exactly the worst width.
- Modals and panels are not visually differentiated — a user can't tell at a glance which is dismissible-with-Esc, which is persistent, which auto-shrinks layout.

## 4. Modal candidates

Surfaces that should be **modal popups** (centered overlay with backdrop, Esc/click-outside-to-close), per Misha's direction:

| Surface | Today | Should be |
|---|---|---|
| `ctxPanel` (node detail) | Docked side at ≥1440, modal overlay at <1440 (inconsistent) | **Modal overlay always**, centered, ~640 px wide × 80vh, dismissible. |
| `docModal` (markdown viewer) | Docked side drawer at all widths | **Modal overlay**, centered, ~800 px wide × 85vh, dismissible. Reading long-form text should NOT shift the persistent layout. |
| Staged-note / Send to Dispatch (inside ctxPanel today) | Buried at bottom of ctx-panel's body | **Dedicated modal** triggered from a node's "send" action — composer-style. |
| Backfill notes (the missing-rich-details fallback the ctx-panel sometimes renders) | Inline ctx-panel section | Promote to its own modal when triggered from the "incomplete metadata" badge. |
| `docsPanel` (cross-project docs browser) | Right-edge drawer with backdrop | Keep as drawer (acceptable) OR convert to modal — either works. |

Surfaces that should be **persistent panes**:
- Tree (full vertical, narrow)
- Waiting on you
- Backlog

That's THREE persistent surfaces, not five. Everything else is on-demand.

## 5. Narrow-window simulation

At **1024 px wide** today:
- Layout B fires (because width < 1440 AND not both dimensions < 1024).
- Tree gets 100% of top half (1024 × 540 ish).
- Actions + Backlog quartered into 512 × 540 each below.
- Click a node → ctx-panel pops as 34 rem modal-with-scrim (~544 px).
- Doc viewer also opens as 32 rem side pane — but at 1024 px the layout shifts margin-right 32 rem (512 px) and EVERYTHING ELSE gets 512 px to share. Actions becomes 256 px. Unreadable.

At **800 px wide**:
- Layout B + B' fires (h < 700 → stacked actions over backlog). Tree top, then actions, then backlog — three full-width vertical rows.
- Tree gets 800 × ~44% = 800 × 396. Actions + backlog each get 800 × 224.
- Click a node → ctx-panel pops as 92vw modal (~736 px) over everything.
- This is the only mode where the doc-pane doesn't cause severe re-pack — but reading docs in a 92vw modal is still chunky.

**Conclusion:** there's no width between 700 and 1440 px where the layout doesn't suffer.

## 6. Other audit dimensions

### Accessibility
- ✓ Header has `<h1>`; panes have `aria-label`; live regions for status / last-read / freshness.
- ✓ `prefers-reduced-motion` honored (`app.css:657-663`).
- ⚠ `<select>` widget for tree-switcher is fine, but the project add `+ project` button only has `title=`, no `aria-label`. Minor.
- ⚠ Tab bar at narrow widths is keyboard-navigable but doesn't announce active tab to screen readers (no `role="tab"`/`aria-selected`).
- ⚠ ctx-panel close button (`✕`) has `title=` only. Screen reader gets "button" + title. Should be `aria-label="Close detail panel"`.
- ⚠ Modal/drawer focus management: opening ctx-panel does NOT trap focus or move focus into the panel. Esc dismisses (good).

### Keyboard navigation
- ✓ Esc dismisses ctx-panel + doc modal + docs panel.
- ⚠ Tab order is browser-default. No skip-links. Tree canvas is not keyboard-navigable (no arrow-key node walk).
- ⚠ No global keyboard shortcuts (e.g., `/` to focus search, `g w` to jump to Waiting, etc.).

### Performance (static analysis)
- Tree renders by full DOM clear + rebuild on every snapshot (`renderTree()` calls `clear(treeCanvas)` then appends). With ~50+ nodes this is fine; at ~500 nodes it'd jank. Not currently a problem.
- SSE keep-alive every 15 s + state file watch + 40 ms debounce: solid.
- Doc markdown render is client-side regex-based (`mdRender`) — fast for typical doc lengths.
- No virtualization for actions/backlog lists. At 100+ items it'd start to lag.

### State persistence
- ✓ Active tree, zoom, collapsed set, drafts, show-concluded all persisted to `localStorage`.
- ✓ ctx-panel and doc-pane widths persist via `--ctx-pane-w` and `--doc-pane-w` CSS vars.
- ⚠ Active tab (when at narrow widths) does NOT persist between page loads.
- ⚠ When you reload at narrow width, you always start on the Tree tab — even if you were last looking at Waiting.

### Browser compatibility
- Vanilla CSS Grid + custom-property `--var()`. IE11 dead, all evergreen browsers fine.
- SSE: not supported in IE11; fine elsewhere.
- No build step, no polyfills, ~5800 LOC total.

### Visual quality
- Dark mode only (intentional, per harness UX standards).
- Type tags as tinted-badge (item 36): legible.
- Per-card 5% type-color tint + left accent stripe: nice density signal.
- Tree node rows have 3 px left accent + 1 px border — visually busy at low widths because of vertical padding around the row.
- `--bg` (#111827) vs `--panel` (#1f2937) contrast OK but subtle; on a glossy monitor the pane boundaries can blend.

## 7. Findings summary

| ID | Severity | Finding |
|---|---|---|
| **UX-VR-01** | P0 | Tree pane wastes ~70% of horizontal real estate at ≥ 1440 px (used as a 1095 px column when content needs ~300 px). |
| **UX-VR-02** | P0 | Layout B (1024–1439 px) stacks actions+backlog BELOW the tree, against Misha's explicit "additional panels to the side, never below" directive. |
| **UX-VR-03** | P0 | Up to 5 persistent surfaces (tree, waiting, backlog, ctx-panel, doc-pane) crowd the same horizontal layout; should be ≤ 3 persistent + N modal. |
| **UX-VR-04** | P0 | ctx-panel and doc-pane should be modal overlays, not docked panes that shrink the persistent layout. |
| **UX-VR-05** | P1 | No drawer / off-canvas behavior at < 1024 px for the side panel — currently jumps to full-screen-pane-with-tabs (Layout C). The drawer pattern is more conventional. |
| **UX-VR-06** | P1 | Narrow-window breakpoints are too coarse: Layout B' at < 700 px is too late (cramping starts at ~900 px); Layout C only fires when BOTH dimensions < 1024 (excludes tall narrow windows). |
| **UX-VR-07** | P1 | Send-to-Dispatch composer is buried at the bottom of a long ctx-panel. Should be its own modal triggered from the node header. |
| **UX-VR-08** | P2 | Tree node rows are visually busy at low widths (3 px left accent + 1 px border + per-row padding). Could simplify. |
| **UX-VR-09** | P2 | No item-level keyboard shortcuts; tree canvas not keyboard-navigable. |
| **UX-VR-10** | P2 | ctx-panel close `✕` lacks `aria-label`; tab bar lacks `role=tab`. |
| **UX-VR-11** | P2 | Narrow-mode active tab not persisted across reload — always lands on Tree. |
| **UX-VR-12** | P2 | Backlog capture form lives ABOVE the list when expanded — pushes existing items down. Could overlay or anchor at bottom. |
| **UX-VR-13** | P0 | **Context is captured via a single-line `<input>`, not a textarea.** A title-only entry isn't useful — context is critical and needs space. Applies to backlog capture AND to inline edits of existing items. Existing items without context should show an explicit "Add context →" affordance, not appear empty. Context should be inspectable in any detail/modal view of the item. |

## 8. Recommended redesign (shipped in the same branch as this audit)

### Persistent layout (ALL widths ≥ ~480 px)

```
┌─────────────┬─────────────────────────────────┐
│             │  Right Side Panel               │
│   Tree      │  ┌─────────────────────────┐    │
│   Pane      │  │ [Waiting] [Backlog]     │    │
│             │  ├─────────────────────────┤    │
│  280-320px  │  │  (selected tab body)    │    │
│  fixed      │  │                         │    │
│  full       │  │  scrolls internally     │    │
│  vertical   │  │                         │    │
│             │  └─────────────────────────┘    │
│             │  400-480px (or drawer < 1024)   │
└─────────────┴─────────────────────────────────┘
```

- **Tree column:** fixed width `clamp(260px, 22vw, 340px)`. Full vertical. Internal scroll. Subtitles compressed (count badges only, no full crumb text inline).
- **Right panel:** width `clamp(380px, 32vw, 520px)`. Two tabs at the top (Waiting / Backlog) with `+N new` badges still showing on inactive tab. **Active tab persists across reload.**
- **At < 1024 px wide:** Right panel becomes an off-canvas drawer that slides in from the right with scrim. A small "Waiting (3)" pill at the top-right shows when collapsed. Tree gets the full window minus pill.
- **At < 480 px wide:** Tree becomes a drawer too, with a hamburger at top-left. Edge case; tablet/phone parity.

### Modal overlays (on-demand)

- `#detailModal` (replaces `#ctxPanel`): centered overlay, 640 × 80vh, with the full ctx-panel body. Click a node → modal opens. Esc / outside-click / ✕ dismiss. **Does NOT shift any persistent layout.**
- `#docModal` (already exists): centered modal at ~800 × 85vh instead of docked side pane.
- `#dispatchModal` (NEW): the Send-to-Dispatch composer extracted into its own modal. Triggered from the detail modal's "Send to Dispatch" button — opens a smaller focused composer (~520 × 60vh).

### Tabs in the right panel
- Replace the always-stacked `.right-col` with a single panel containing `<nav>` with two buttons (Waiting / Backlog) and a single `<section>` whose contents swap.
- `body[data-pane-tab="waiting"]` / `body[data-pane-tab="backlog"]` toggles visibility.
- Persisted in `localStorage` as `ctree-pane-tab`.

### Detail modal trigger affordance
- Clicking a tree node SELECTS the node (highlight) but does NOT auto-open the modal. A small `(i)` icon appears in the tree-pane title bar showing "details for [node title]" — click it to open the modal. Reduces noise for "just navigating the tree."
- Alternative: auto-open the modal on second click (single-click selects, second click opens). Decision: go with explicit `(i)` button — it's clearer.

### Drawer for < 1024 px
- New collapse / expand handle on the right edge.
- Tree gets full width minus a 36 px right gutter.
- Drawer slides in over the right portion with a scrim.

### Backlog capture form
- Move to overlay-anchored at the bottom of the backlog panel instead of pushing the list down.

### Context-as-textarea pattern (UX-VR-13)
- Capture form: the `context note` input becomes a `<textarea>` that grows as the user types (autoresize) and supports multi-line entry. Default visible rows: 3.
- Existing items in the backlog list expose an inline `▾ context` disclosure. If `item.context` exists, click expands a read-mode markdown render with an "edit" affordance; click expands an editable textarea with save/cancel buttons. If `item.context` is absent, the disclosure shows "Add context →" as a clear affordance to add one.
- Save emits a new additive event `backlog-context-set` (`item_id`, `context_text`) — schema version stays 1. Reducer applies last-writer-wins on the target item's context field. Idempotent on event_id.
- The same context surface is exposed in the dispatch composer modal when a backlog item is the target (context shown above the composer body so the operator has full background while composing).
- Waiting-on-you items already have the equivalent surface via the inline `▾ details` disclosure (`renderItemDetails`) — that path is preserved; no new pattern needed there.

## 9. Out of scope for this redesign

Captured here so they don't get lost. These are NOT in the redesign PR; they go to `docs/backlog.md` as follow-ups:

- Tree node keyboard navigation (arrows + Enter to select).
- Global keyboard shortcuts (`/` for tree search, `g w` for waiting, etc.).
- Backlog/actions virtualization for 100+ items.
- Tree node visual simplification (border / accent re-think).
- Full a11y audit (focus trap in modals, `role=tab` for tab bar, `aria-label` on `✕` buttons).
- Tab-active state persistence at narrow widths (covered in this redesign).
- Modal animation easing (currently no enter/leave transitions on `#ctxPanel`).
- Filter/search inside the tree (Misha hasn't asked, but at 200+ nodes it'll matter).

## 10. Validation checklist for the redesign

The redesign PR must satisfy:

- [ ] Tree pane has a fixed width between 260 px and 340 px at all viewports ≥ 480 px.
- [ ] No actions/backlog content appears BELOW the tree at any viewport ≥ 480 px.
- [ ] Detail (ex-ctx-panel) is a centered modal that does NOT shift `#layout` margins.
- [ ] Doc viewer is a centered modal that does NOT shift `#layout` margins.
- [ ] Send-to-Dispatch composer is its own modal triggered explicitly from the detail modal.
- [ ] Right side panel has Waiting / Backlog tabs; active tab persists via localStorage.
- [ ] At < 1024 px wide, the right panel collapses into a slide-in drawer with scrim.
- [ ] All existing functionality (sort, defer, respond, priority, drafts, cross-links, archive, etc.) is preserved — nothing is dropped.
- [ ] `web/responsive.selftest.js` updated and `node web/responsive.selftest.js` exits 0.
- [ ] Manual smoke-test at 1920 / 1440 / 1280 / 1024 / 800 / 480 px confirms each width is usable.

Findings UX-VR-01 through UX-VR-07 are addressed by the redesign in this PR.
UX-VR-08 through UX-VR-12 are captured in `docs/backlog.md` for follow-up.
